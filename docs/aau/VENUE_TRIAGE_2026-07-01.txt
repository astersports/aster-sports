2. ROOT CAUSE — COURT NULL (state discovered by triage; not in the original 3-state model)
================================================================================
- division_games.source_facility_ref is an OPAQUE TM facility id (e.g.
  h202606041542226344c6972f899ba43), NOT human text. Westchester Summer League has TWO
  such ids (96 + 73 games) both resolving to the single "House of Sports" building venue.
  Those two ids ARE Court 1 and Court 2 — the venue-dedup ("dedupe_venues_to_buildings")
  correctly collapsed them to the building for directions, but that dropped the court label.
- The facility-id -> "Court N" map is NOT recoverable in-DB (the venue's tm_venue_key does
  not equal either facility id; the court names lived only in the TM Complexes panel).
- BUT the human location cell (g.location = "4 - House of Sports - Court 1") IS available at
  ingest time: it's held transiently in __location, used to resolve the building venue via
  cleanPlaceName (which strips the ordinal + "- Court X"), then discarded. The COURT is
  exactly the "- Court X" suffix cleanPlaceName strips. So court parses from g.location with
  NO facility panel and NO re-scrape beyond the ingest that already runs.

THE +103->+202 "CLIMB" WAS PARTIAL COMPLETION, NOT DUPLICATION
- The unique index `division_games_extkey_unique (tournament_division_id, external_game_id)`
  is FULL (non-partial) — per-game duplication is impossible.
- Offline parse of all 14 WPCYO division pages sums to EXACTLY 1308, matching the pre-revert
  live count. Every one of the 11 backup divisions matches the DB exactly (114/99/130/90/105/
  73/103/96/107/89/100 = 1106). The delta is 3 divisions the ORIGINAL ingest lacked:
  8th Grade Boys (99 REAL games), HS Boys (85 REAL games), ADMIN TEAMS (18, junk).
- So the count converged toward a FIXED 1308 across two partial runs (1106->1209->1308). It
  is idempotent AT CONVERGENCE — not runaway. The +184 (8th Grade Boys + HS Boys) are REAL
  games the original ingest missed; prod is genuinely STALE by 184 games after the revert.

CAVEAT — CONVERGENCE-BY-COUNT IS NOT A SAFETY INVARIANT (architect, ratified)
- "It can never exceed 1308" holds for WPCYO's CURRENT snapshot only. The unique index bounds
  DUPLICATE (division, extid) pairs; it does NOT bound growth if extid (the slot label cells[0])
  itself churns. `src/lib/aau/stableGameKey.js` documents exactly this "regeneration tail where
  data-gameid churns" (extid = cells[0]; the durable data-gameid sits in source_game_id awaiting
  the §2.D key swap). A churned slot label is a NEW pair = a NEW row = count past 1308.
- THEREFORE the completeness gate compares DIVISIONS DISCOVERED vs DIVISIONS INGESTED — never
  "count == expected." The count version would be the wrong gate.

ROOT CAUSE (fetch layer, code-confirmed)
- `fetchTM` (index.ts): SINGLE attempt, no retry, aborts on timeout; a large-page IncompleteRead
  throws out of `await res.text()` the same way. (Reproduced directly: urllib/curl IncompleteRead
  on the 1.6MB "3rd Grade Boys" page mid-trace.)
- The division loop fetches each page INSIDE a per-iteration try/catch whose header comment says
  "one bad division doesn't abort the run" — a failed division is SILENTLY SKIPPED. The only
  hard-fail is ZERO divisions discovered (422). So a partial run returns 200 looking COMPLETE.
- WPCYO = 14 divisions x ~1.2-1.7MB (~19MB). Each invocation skips a different random subset of
  large pages -> a different partial each run -> union converges over repeated polls.

THIS IS AN HONEST-CLAIMS DEFECT, NOT RELIABILITY POLISH (architect, ratified)
- A division whose page RELIABLY exceeds the fetch timeout fails EVERY poll and is NEVER
  ingested -> permanent SILENT undercount. And silent: the Hub presents a partially-ingested
  league as complete, with no signal that divisions are missing. That is a FALSE-COMPLETE claim
  (same family as "eliminated from a bracket they never entered"). Priority is raised accordingly.
- Correction to my own earlier overstatement: it does NOT "drift +100 rows/poll forever." The
  ingest only ADDS (never deletes) and the index bounds duplicates, so a live large tournament
  CONVERGES to completeness over polls. But convergence is not guaranteed for a page that never
  fetches — that division is a permanent undercount, not a self-healing lag.
- RELEASE GATE (architect correction — right axis): the completeness gate must land before any
  LARGE multi-division tournament goes live-polling, REGARDLESS OF CIRCUIT (not "non-ZG").
  Westchester passed on luck of page size, not a guarantee.

FIX (this session's reliability PR — edge function + pure helper + tests; held for architect review)
- fetchTM: 3 attempts, linear backoff. Necessary but NOT sufficient (a reliably-oversized page
  still fails all attempts) — the gate, not the retry, is load-bearing.
- Completeness gate in the ingest response: never drop good divisions (partial success upserts
  what succeeded); report divisionsDiscovered / divisionsExpected / divisionsIngested / complete /
  missingDivisions as run truth; hard-fail (502, retain prior) ONLY when NOTHING succeeded.
- ADMIN filter: isAdminDivision(name) by SIGNATURE (admin/staff/officials/internal/test/placeholder),
  applied in the ingest with a per-rejection LOG (never a silent drop; a new junk variant surfaces).
  Mirrored parseTournament.js <-> _parse.ts (AP #30); vitest safety test asserts the 14 real WPCYO
  divisions + ZG divisions never match.
