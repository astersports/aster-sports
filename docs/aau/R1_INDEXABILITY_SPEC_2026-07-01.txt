R1 INDEXABILITY — DESIGN SPEC (for architect review)
Date: 2026-07-01
Author: CC (Claude Code)
Status: DESIGN DOC — review artifact, not a build. The build waits on this spec landing.
Scope: aster-sports repo, the public AAU Hub. R1's one open gate (per #1178: R1 build-req + R2 exit-gate).

──────────────────────────────────────────────────────────────────────────────
0. PURPOSE
──────────────────────────────────────────────────────────────────────────────
The public AAU Hub must be crawlable. SEO is the entire reason R1 stays web
instead of native, and an indexable Hub is the exit gate for the irreversible
R2 front-door cutover (a cutover that de-indexes the wedge is self-defeating).
R1's foundation is DONE (frozen PUBLIC_RPC_CONTRACT, client truth-port, no
client-side record math). Indexability is the remaining gate — and it is a
BUILD, not a config flip.

──────────────────────────────────────────────────────────────────────────────
1. GROUNDED CURRENT STATE (verified 2026-07-01 against main)
──────────────────────────────────────────────────────────────────────────────
- vercel.json: `rewrites: /(.*) → /index.html` — pure SPA fallback; every URL
  serves the same shell.
- build: plain `vite build`. No SSR / SSG / prerender. No entry-server, no
  prerender script, no react-snap/puppeteer, no react-helmet/unhead in deps.
- index.html: ONE static site-level OG block (og:type/title/description/url/
  image) + generic `<title>Aster Sports</title>`. No per-route meta.
- No public/robots.txt, no public/sitemap.xml. HTML served no-cache,no-store.
- Consequence: a crawler hitting /hub/tournament/<id> receives an empty
  <div id="root"> and one generic title. All tournament/division/team content
  materializes only after JS runs the get_public_* chain.

Small head start (does NOT change the conclusion): the generic OG block already
exists in index.html — the SCAFFOLDING for per-route meta is present; it just
carries site-level constants today.

Indexable surface (App.jsx:90-93) — 4 public routes, no auth:
  /hub                                              → AauHubPage (directory/browse)
  /hub/team/:teamKey                                → AauTeamSchedulePage
  /hub/tournament/:tournamentId                     → AauTournamentDetailPage
  /hub/tournament/:tournamentId/division/:divisionId→ AauDivisionDetailPage
  (Adjacent public, non-hub: /schedule/:teamId, /privacy, /terms — decide
   inclusion in §6 robots; the AAU wedge is the 4 /hub* routes.)

Public data plane — 7 RPCs + 1 search, all already the sole read path:
  get_public_tournament_directory, get_public_tournament_teams,
  get_public_tournament_standings, get_public_tournament_games,
  get_public_aau_team_schedule, get_public_team_bracket_path,
  get_public_program, search_public_aau.

──────────────────────────────────────────────────────────────────────────────
2. ORGANIZING PRINCIPLE — FRESHNESS IS AN HONESTY DECISION, NOT A PERF ONE
──────────────────────────────────────────────────────────────────────────────
The Hub is LIVE — scores update as often as every 5 minutes on game days. A
stale prerendered score served to a crawler-then-user is exactly the sub-100%
surface that loses a parent. Therefore:

  PURE BUILD-TIME SSG IS WRONG. It bakes live ticks into cacheable HTML.

The design is a FRESHNESS BOUNDARY, not an infra pick:
  • SSR/ISR the STABLE STRUCTURAL TRUTH — everything a crawler indexes.
  • CLIENT-HYDRATE the LIVE TRUTH — everything still moving.
  • NEVER cache a live tick.
This is the honesty doctrine expressed in rendering: server-render what's
settled, hydrate what's moving. The frozen RPC contract makes it clean — the
server renders from the SAME get_public_* the client does, with the basis-gate
(no-fabrication) carried into SSR. Infra follows the boundary; it does not set it.

──────────────────────────────────────────────────────────────────────────────
3. THE FRESHNESS BOUNDARY (per surface AND per field/row-state)
──────────────────────────────────────────────────────────────────────────────
The boundary is NOT per-RPC. Several RPCs return both settled and live rows in
one response (e.g. get_public_tournament_games returns final AND in-progress
games). The server renders the SETTLED projection; the client hydrates and
reconciles the LIVE rows on mount.

SSR-STABLE (server-rendered, indexed, revalidated on ingest):
  - Tournament existence, name, date window, schedule_status label.
  - Division list; team list (roster of entered teams).
  - Bracket STRUCTURE (slots, seed sources, advancement edges).
  - COMPLETED/FINAL game results (status='final', settled score).
  - Directory (which tournaments exist) — the crawl entry point.
  - Per-route <title> + meta + canonical + OG, derived per entity.

HYDRATE-LIVE (client-only, never in cached HTML):
  - In-progress scores (status='live'/in-progress).
  - Live standings deltas that depend on in-progress games.
  - Court fills / time changes still arriving (schedule_status in flux).
  - "Up Next" hero (AauUpNextHero) — inherently now-relative.

Rule of thumb: if a value can change without a re-ingest, it is hydrate-live.
If it only changes ON ingest, it is SSR-stable and safe to cache-until-revalidate.

Per-route SSR payload (structural projection only):
  /hub                 ← get_public_tournament_directory (settled list)
  /hub/tournament/:id  ← get_public_tournament_teams + _standings(final rows) +
                         _games(final rows) + bracket structure; live rows blanked
                         server-side, filled on hydrate.
  /hub/.../division/:id ← division projection of the above.
  /hub/team/:teamKey   ← get_public_aau_team_schedule (settled games) +
                         get_public_team_bracket_path (structure).

──────────────────────────────────────────────────────────────────────────────
4. INFRA OPTIONS (Vite SPA → SSR, weighed against MINIMIZE-BLAST-RADIUS)
──────────────────────────────────────────────────────────────────────────────
Hard constraint: add SSR for the PUBLIC INDEXABLE ROUTES ONLY. Leave the
authenticated-tier SPA (all `Protected`/`RequireAuth` routes) exactly as-is —
they don't need indexing and adding SSR there would touch auth/session. Do NOT
rewrite the app to index a wedge.

  A. Vercel-native on-demand SSR/ISR for /hub* only (LEAN).
     - A serverless/edge render function for the 4 routes, rendering React from
       the same get_public_* (service or anon-key read, basis-gate applied).
     - `revalidate`-on-ingest: the aau-ingest function already runs on every
       poll — it fires a revalidation (tag/path purge) after a successful,
       COMPLETE ingest, so cached structural HTML refreshes only when the
       settled truth actually changed. Live rows never cached (hydrate-only).
     - Blast radius: smallest. One render path, public routes only, SPA
       untouched. Vercel is already the host.
  B. Vite SSR (custom entry-server.jsx + server handler).
     - Most control, one app, but more hand-wiring (hydration, streaming,
       route-split) and we own the server runtime.
  C. Prerender plugin (Vike/@prerenderer).
     - Viable ONLY paired with client-hydration reconciliation + on-ingest
       revalidation (Vike does SSR+ISR). Plain prerender = SSG = §2 violation.
  D. Meta-framework island for /hub* (Astro/Next), reverse-proxied.
     - Cleanest SEO story but two apps / duplicated shell / duplicated design
       tokens — highest maintenance + drift surface. Rejected unless A–C prove
       infeasible.

LEAN: A (Vercel-native ISR, public routes only, revalidate-on-ingest). It
matches the host, honors the freshness boundary structurally, and holds blast
radius to the 4 public routes. FORK for architect: A vs B (own-the-runtime
control vs Vercel-managed) — see §7.

──────────────────────────────────────────────────────────────────────────────
5. PER-ROUTE META
──────────────────────────────────────────────────────────────────────────────
Each SSR route emits entity-derived <title>/description/canonical/OG from its
structural payload (e.g. "8th Grade Boys — WPCYO 2025-26 | Aster Sports"). The
existing index.html OG block becomes the fallback for the SPA (authenticated)
routes; public routes override per entity at render time. No react-helmet needed
if meta is emitted directly in the SSR document head.

──────────────────────────────────────────────────────────────────────────────
6. SITEMAP + ROBOTS (the same freshness question, different hat)
──────────────────────────────────────────────────────────────────────────────
Sitemap = the STRUCTURAL truth "which URLs exist," which changes only on ingest
— identical freshness class to §3 SSR-stable. So:
  - Generate sitemap.xml from get_public_tournament_directory (→ tournaments) +
    their divisions/teams (→ child URLs). NOT a static file, NOT a build-time
    snapshot (a new tournament wouldn't appear till redeploy).
  - Serve it from the same on-demand/ISR layer, revalidated on the SAME
    ingest trigger as §4.A. New tournament ingested → sitemap reflects it on
    next revalidate.
  - <lastmod> per URL from the entity's last-ingest timestamp (honest freshness
    signal to crawlers).
robots.txt: Allow the 4 /hub* routes + /schedule/:teamId (public) + /privacy,
/terms. Disallow the authenticated app shell (/, /admin/*, /account, etc.) —
they're behind auth and shouldn't be crawl targets. Point Sitemap: at the
generated sitemap URL.

──────────────────────────────────────────────────────────────────────────────
7. OPEN FORKS FOR ARCHITECT
──────────────────────────────────────────────────────────────────────────────
F1. Infra: §4.A (Vercel-native ISR) vs §4.B (own Vite SSR runtime). Lean A.
F2. Revalidation trigger granularity: purge-by-tag per tournament (precise, more
    wiring) vs purge-the-public-namespace on any complete ingest (simple,
    coarser). Lean per-tournament tag if A supports it cleanly, else coarse.
F3. Live-row hydration contract: does the SERVER emit the settled projection
    with live rows OMITTED (client fetches+fills), or emit them flagged
    `pending:true` (client replaces)? Omit-and-fill is more honest (no stale
    value ever in HTML); flag-and-replace avoids layout shift. Lean omit-and-fill
    for scores (never ship a stale score), flag-and-replace for court/time (show
    "syncing", replace on hydrate) — mirrors the push monotonic-vs-mutable split
    from the R4 forward-flag.
F4. Does the condition-3 ingest own the revalidation ping, or a separate hook?
    (Couples SEO freshness to the ingest that already knows "structure changed.")

──────────────────────────────────────────────────────────────────────────────
8. OUT OF SCOPE (this spec does NOT touch)
──────────────────────────────────────────────────────────────────────────────

- R2 front-door cutover / vercel.json rewrites for the domain move (separate

- Any money/child/schema surface.
- The condition-3 WPCYO completion (shadow-table) — separate thread; it only

──────────────────────────────────────────────────────────────────────────────
9. SEQUENCING
──────────────────────────────────────────────────────────────────────────────
This spec gates R2 regardless of whether R3 slots before or after — so it is not
wasted under any routing. Build waits on this spec landing (architect review of
the freshness boundary + F1–F4). Prerequisite for the eventual R2 step: a Vercel
preview-deploy dry-run before touching production rewrites (local build ≠ prod
rewrite behavior).
