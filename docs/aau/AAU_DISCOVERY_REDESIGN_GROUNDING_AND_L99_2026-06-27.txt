1 ¬∑ GROUNDING ‚Äî THE LIVE CONTRACT vs THE SPEC

  1.1 SEARCH  [GROUNDED: pg_get_functiondef]
    LIVE: public.search_public_teams(p_query text) -> jsonb
      Returns a FLAT array, TEAMS ONLY:
        [{ teamKey, name, tournamentId, tournamentName, startDate, endDate,
           divisionId, divisionName, gradeLabel, gender }]
      ‚Ä¢ NOT program-grouped (no program_id, no nested variants ‚Äî ¬ß2.E identity absent)
      ‚Ä¢ NO tournaments[] / divisions[] typed sections (the render's ¬ßR1 three-section result)
      ‚Ä¢ NO record / rating / is_live on the row (the ¬ßR4 value-in-row is absent)
      ‚Ä¢ NO tier / day (only gradeLabel + gender)
    SPEC WANTS (¬ßR1): typed sections { teams(program-grouped w/ variants), tournaments,
      divisions }, each variant carrying division{gender,grade,tier,day}, record, rating, is_live.
    => DELTA. Per spec ¬ß10 this is the named L99 item: search returns flat team rows,
       so program-grouped + typed + valued search is a CONTRACT ADDITION.

  1.2 DIVISION TAXONOMY (¬ß2.A)  [GROUNDED: information_schema]
    tournament_divisions HAS: gender, grade_label, grade_min, grade_max, tier, day,
      circuit, advance_count, bracket_size. The structured columns the chips need EXIST.
    The gap is only that search/directory don't SELECT tier+day yet ‚Äî a field addition,
    NOT a backfill. No name-parsing needed anywhere (DO-NOT #2 satisfied by the columns).

  1.3 PROGRAM IDENTITY (¬ß2.E)  [GROUNDED: information_schema]
    tournament_division_teams has NO program_id column. There is no programs table.
    "Program" today is DERIVABLE only from the team name / resolved_key + shared location
    (e.g. "High Rise ‚Äî Brie", "High Rise ‚Äî Will" share the "High Rise" stem). Grouping by a
    normalized program stem is an honest derive (NOT fabrication); a true program entity is
    a deeper backbone item (flagged, not built here).

  1.4 DIRECTORY / BROWSE  [GROUNDED: pg_get_functiondef]
    LIVE: public.get_public_tournament_directory() -> jsonb
      Returns tournaments (de-duped to the richest copy per name+date), each with circuit,
      start_date, end_date, divisions[]{id,name,grade_label,gender,advance_count,team_count}.
      ‚Ä¢ HAS circuit + dates => circuit -> season+year -> tournament grouping is CLIENT-SIDE
        derivable (season from start_date month, year from start_date). RENDER-GATED, no L99.
      ‚Ä¢ Division rows here also lack tier/day + is_live (same field-addition as 1.2).
      ‚Ä¢ No filter args, and no DIVISION-level browse (R3 division_browse) ‚Äî see A4.

  1.5 PASTE / INGEST (¬ßR2)  [GROUNDED: frontend aster.ts + pg_proc + aau_ingest_submissions]
    LIVE: edge fn `aau-submit-tournament` (POST {url}) -> { ok, status, submissionId,
      tournamentId } ; then public.get_aau_ingest_status(p_id uuid) -> { status,
      tournamentId, tournamentName, divisionCount, error }. aau_ingest_submissions carries
      status in {ingesting, ready/ok, duplicate, error, missing}.
    SPEC WANTS: resolve_source(url) {known? } + phased % status + source labeling.
      ‚Ä¢ known-vs-new: ALREADY covered ‚Äî submit returns `duplicate` + tournamentId for a
        known link. A dedicated resolve_source(url) is OPTIONAL sugar, not required. RENDER-GATED.
      ‚Ä¢ phased % + per-phase step copy: the status RPC returns a coarse status, not a %/phase.
        The render's 4 phase lines + % bar can ride on status + divisionCount today (honest,
        coarse) and tighten later. RENDER-GATED (no fabricated %; show real phase, not theater).
      ‚Ä¢ source label (recognized/external/unrecognized): client-derivable from the URL host +
        the submit result. RENDER-GATED.

  1.6 LIVE STRIP (¬ßR5)  [GROUNDED: pg_proc ‚Äî absent]
    No public.get_public_live_now(...) exists. The front-door "Live right now" strip + the
    browse/result live dots have NO public read RPC today. The realtime TICK is TECH-1
    (deferred, not a blocker) ‚Äî but even the ON-LOAD READ needs a public RPC. => A new RPC.

  1.7 FRONTEND STATE  [GROUNDED: file:line]

    exact font stack (Space Grotesk/Inter/Space Mono) + brand gradient
    (#E0631C‚Üí#E8902A‚Üí#F6CC55‚Üí#FBD56B) already live in index.css. All RPC calls go through
    named wrappers in lib/aster.ts (no RPC name in a component) ‚Äî so a new RPC is a one-file

2 ¬∑ THE L99 SURFACE (contract additions ‚Äî additive + versioned; never rename/remove)
  Consumer-impact [GROUNDED]: the only reader of these public RPCs today is the AAU hub
  itself (astersports-web). New RPCs / new nullable fields break no existing read ‚Äî Legacy
  Hoopers + future tenants read get_public_aau_team_schedule / standings, untouched here.

  A1 ‚Äî search_public_aau(p_query text) -> jsonb   [NEW RPC, the spec ¬ß10 item]
     Typed sections { teams, tournaments, divisions }. teams program-grouped:
       [{ program, location, variants:[{ teamKey, label, gender, grade_label, tier, day,
          record, rating, basis, is_live }] }]. Leaves search_public_teams INTACT.
     LEAN: NEW name (not a shape change to the old fn) so no consumer can break, and the old
     fn stays as the v1 fallback the screen renders against until A1 lands (spec ¬ß10 "held element").

  A2 ‚Äî value-in-row (record ¬∑ rating ¬∑ basis ¬∑ is_live)
     Fold INTO A1's variant objects AND a directory companion, so ONE round-trip per section
     carries its value (no N+1 per-row value fetch). record from division_games results;
     rating already computed by get_public_tournament_standings (reuse its basis gate); is_live
     from division_games.status='live'. basis flag drives the "projection" label (no #7 fabrication).
     LEAN: bundle with A1; do not ship a separate per-row value RPC (AP #40 ‚Äî dedicated shape,
     but same round-trip).

  A3 ‚Äî tier + day fields on search + directory division rows
     Pure field addition (columns already exist, 1.2). Additive nullable JSON keys.

  A4 ‚Äî get_public_division_browse(p_filters jsonb) -> jsonb   [NEW RPC, R3 filters]
     Division-grouped browse filtered by {gender, grade_label, tier, day, status}. Distinct
     SHAPE from the tournament-grouped directory (AP #40 ‚Äî dedicated contract, not a directory
     filter-arg bolt-on). LEAN: PHASE 2 ‚Äî the render's Browse screen (04) is TOURNAMENT-grouped
     (circuit‚Üíseason‚Üítournament), which is render-gated on the existing directory (1.4); the
     division-filter chips returning division lists are the second cut. Build screen 04 now;
     file A4 for L99 but don't block the redesign on it.

  A5 ‚Äî get_public_live_now(p_limit int) -> jsonb   [NEW RPC, R5 read]
     Currently-live public games for the front-door strip + live dots. One RPC feeding multiple
     surfaces (front door, results, browse) ‚Äî not per-row polling. The realtime TICK stays TECH-1.
     LEAN: small + high-leverage; ship with A1.

  NOT L99 (RENDER-GATED ‚Äî build now, no contract change):
     ‚Ä¢ The full 5-state visual redesign onto existing data.
     ‚Ä¢ Browse circuit‚Üíseason‚Üíyear grouping over existing get_public_tournament_directory (1.4).
     ‚Ä¢ Paste/ingest flow visual on existing aau-submit-tournament + get_aau_ingest_status (1.5).
     ‚Ä¢ Known-vs-new via the existing `duplicate` status; URL-host source label.
     ‚Ä¢ No-results escape hatch (paste + request-ingest + REAL near-match only, DO-NOT #4).
