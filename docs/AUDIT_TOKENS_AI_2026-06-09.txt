FINDING 1 — AI-draft advertised for weekly_digest + tournament_prelim but
            client has NO surface to invoke it (draft hidden, no hint)
SEVERITY: P2
==============================================================================
FILES:
  src/lib/briefings/anchoredAiFacts.js:10-14   (AI_DRAFT_FIELD: only 3 kinds)
  src/lib/briefings/anchoredAiFacts.js:36-76   (EXTRACT: only 3 kinds)
  src/lib/briefings/aiDraftPrompt.js:13         (ANCHORED_KINDS: 5 kinds)
  supabase/functions/briefing-ai-draft/_helpers.ts:7  (mirror: 5 kinds)
  src/components/briefings/AiDraftAnchored.jsx:28,33   (field lookup -> null)
  src/components/briefings/StepBodySignoff.jsx:78-79   (mount point)

WHAT'S WRONG:
  ANCHORED_KINDS (and AI_DRAFT_KINDS) advertise 5 AI-draftable anchored kinds:
  game_recap, games_recap, weekly_digest, tournament_recap, tournament_prelim.
  The edge fn (briefing-ai-draft) accepts all 5 and the AIDRAFT build spec
  (docs/AIDRAFT_COMPOSE_FN_BUILD_SPEC §8/§4d) lists weekly_digest +
  tournament_prelim as supported.

  But the CLIENT only wires 3 of them. AI_DRAFT_FIELD and the EXTRACT map in
  anchoredAiFacts.js cover ONLY game_recap, games_recap, tournament_recap.

  AiDraftAnchored.jsx:33 does `if (!entry || !field) return null;`. For
  weekly_digest and tournament_prelim, `field = AI_DRAFT_FIELD[kind]` is
  undefined, so the component renders NOTHING — no draft button, and critically
  no "AI draft not available for this kind" hint. AiDraftFreeForm also returns
  null (not a free-form kind). Net result: for these two reachable composer
  kinds the AI-draft capability is silently absent. This is exactly the
  "draft button hidden/disabled with no hint" UX class the audit targets.

  (tournament_prelim does get the SEPARATE suggest-briefing-closer button via
  SUGGEST_CLOSER_KINDS in StepBodySignoff:48 — that is a closer-line feature,
  not the full-narrative AI draft. weekly_digest gets no AI assist at all.)

  Note: weekly_digest's anchorFromState (registry.js:79) returns
  {orgId, period, pilotOnly} with no eventId/tournamentId/eventIds, so even if
  AI_DRAFT_FIELD had an entry, AiDraftAnchored's anchorReady gate (line 32)
  would also need widening. Two-layer gap.

PROPOSED FIX (pick one, operator/architect call):
  (a) If weekly_digest + tournament_prelim AI draft is in-scope: add their
      AI_DRAFT_FIELD entries + EXTRACT fact builders in anchoredAiFacts.js, and
      widen AiDraftAnchored anchorReady to accept the weekly_digest period
      anchor. (Matches the file's own "Add an anchored kind by..." header.)
  (b) If out-of-scope for v1: remove weekly_digest + tournament_prelim from
      ANCHORED_KINDS in BOTH aiDraftPrompt.js AND _helpers.ts (AP#30 mirror,
      same commit) so the advertised set matches the wired set, AND/OR render a
      one-line "AI draft isn't available for this briefing type" hint in
      AiDraftAnchored instead of returning null for a supported-but-unwired kind.
  Lowest-risk immediate fix: (b)'s hint, so the surface never silently vanishes.

FINDING 2 — callup-token-handler confirmation timestamp not timezone-pinned
            (parity drift vs rsvp-token-handler)
SEVERITY: P3
==============================================================================
FILE:
  supabase/functions/callup-token-handler/index.ts:131

WHAT'S WRONG:
  The callup confirmation page formats the event time with
  `new Date(ctx.start_at).toLocaleString("en-US", {... no timeZone ...})`.
  The rsvp-token-handler equivalent (index.ts:117) pins
  `timeZone: "America/New_York"`. Deno edge runtime defaults to UTC, so the
  callup confirmation page shows the event start in UTC (e.g. a 6pm ET game
  reads as 10pm/11pm). The unsubscribe-handler:92 correctly pins ET.
  This is an AP#43 cross-surface invariant drift (same "format an event time"
  concept rendered with divergent timezone handling across the 3 handlers).

PROPOSED FIX:
  Add `timeZone: "America/New_York"` to the toLocaleString options at
  callup-token-handler/index.ts:131 to match rsvp-token-handler:117.

FINDING 3 — useAiDraft still ships proposal_id path that the edge fn hard-
            rejects (dead defensive code, no live bug)
SEVERITY: P3 (informational / cleanup)
==============================================================================
FILES:
  src/hooks/useAiDraft.js:28,36          (accepts + forwards proposalId)
  supabase/functions/briefing-ai-draft/index.ts:79-81  (400-rejects proposal_id)

WHAT'S WRONG:
  useAiDraft.draft() accepts a proposalId and forwards it as body.proposal_id.
  The edge fn explicitly returns 400 "Pass resolved facts instead of
  proposal_id..." for any proposal_id. No live caller passes proposalId
  (AiDraftAnchored + AiDraftFreeForm both pass facts/gist only), so this is
  dead-but-harmless. Flagged only because if a future caller wired proposalId
  expecting server-side resolution, it would silently 400 with a confusing
  message. Not a current bug.

PROPOSED FIX (optional):
  Drop the proposalId param from useAiDraft, or leave a code comment that the
  server-side proposal path is intentionally unwired (design seam 4b). No action
  required for correctness today.
