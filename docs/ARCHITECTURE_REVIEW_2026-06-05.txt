PART B — FAMILY HUB (consolidated: pre-flight + claim verify + S1 proposal)
==============================================================================
B0. WHAT "THE VERIFY PASS FOUND" (plain language)
  The account-claim feature DOES NOT WORK YET for real parents. The only code that
  links a parent's login to their guardian record (autoLinkGuardian, client-side)
  is BLOCKED by RLS for the rows it targets. The 2 working accounts (Frank +
  Stephanie) were wired by hand in SQL during setup. Since the Hub is the first
  account-REQUIRED surface, a real claim mechanism is a prerequisite to the Hub
  having any users beyond the 2.

B1. THE NATIVE RLS CHAIN (pre-flight, query-grounded)
  auth.uid() -> guardians.user_id -> player_guardians -> players ->
  roster_members.team_id -> events / event_rsvps. Fully RLS-gated; already
  encapsulated in current_user_player_ids() + helpers. The Hub must compose ON TOP
  (as the parent / INVOKER), intersecting child_links with current_user_player_ids()
  — never bypass org isolation (no SECDEF over these tables for the read path).

B2. THE LOAD-BEARING GAP (claim verify, code-grounded)
  - invite-parent does NOT claim — it only mints an auth.users row + sends an invite
    email; never touches guardians.
  - autoLinkGuardian (the only guardians.user_id writer) issues a client UPDATE that
    RLS blocks: guardians_update_own is USING user_id = auth.uid(), which is FALSE
    when user_id IS NULL. A parent's own session cannot flip their unclaimed row.
  - The 2 existing claims were hand-linked admin SQL (NULL invite metadata, provider
    'email', user_roles written seconds after signup). No repeatable flow exists.

B3. PROPOSAL (GO GATE — nothing applied until approved)
  - claim_guardian(p_org_id) SECURITY DEFINER RPC = the real claim: require caller's
    email CONFIRMED + matches an unclaimed guardian (ilike) -> flip guardians.user_id
    NULL->self -> insert user_roles(parent) -> populate household. Atomic + idempotent.
    REVOKE FROM PUBLIC + anon (AP #23/#57). Replaces the RLS-blocked client UPDATE.
  - populate_household(user) SECDEF, idempotent, called on EVERY login (de-gated from
    the "first login only" branch) so new players backfill + a half-claim self-heals.
    Maps DISTINCT players via player_guardians -> family_children + child_links
    (ember_org_id, ember_player_id). De-dupe not needed today (1 org, 0 dup players,
    0 multi-team); fan-out matters only at the 2nd org (S6).
  - S1 TABLES (user-scoped; RLS owner_user_id = (SELECT auth.uid()); NO org_id on
    household rows per plan §2): family_children, child_links, calendar_subscriptions,
    synced_events (sync-shaped per the addendum: status active/cancelled + content_hash
    + last_sync_status). RLS proof (like #710) before S1 is "done."
  - NOT in S1: Hub UI (S2), ICS sync edge fn (S3), conflict detection (S4).

B4. ADOPTION FINDING (rollout, not architecture)
  173/175 guardians are unclaimed (user_id NULL). The native Hub is empty for ~99%
  of families until the claim flow runs. The briefing email channel (already reaches
  all 175) is the claim funnel: a magic-link "Claim your account" CTA. Confirm all
  173 have emails on file before a claim campaign. Claim ships in the SAME wave as
  the Hub foundation (a Hub with no accounts is a demo).
