==============================================================================
§0. EXECUTIVE SUMMARY — THE VERDICT
==============================================================================
Programs today = an admin-only CRUD surface + a one-way anonymous registration
funnel. It works, the tokens are clean, files are well-decomposed (every file
≤150 ln), and the DB/RLS substrate is sound. But it is NOT a 10/10 experience and
it is NOT "100% for all three roles." The three structural problems, in priority:

  P-1  THE PARENT HAS NO SURFACE. A logged-in parent sees their kid's name + team
       and NOTHING about the program they registered for, what they paid, what
       they owe, or how to register again. The registration confirmation
       dead-ends ("Spot reserved!" → Done → back to the public page) with no
       bridge to an account — even though the DB already links guardian→account
       by email. This is the single biggest gap and it is pure surface (zero
       schema change needed).

  P-2  PATTERN A #63 IS EVERYWHERE ON THIS SURFACE. The same concept resolves
       from divergent sources/scopes across at least SIX axes: money (3 price
       surfaces, server-authoritative number fetched-then-discarded), player
       count (rows vs distinct-active), team count (3 callsites), "active" (4
       definitions), "registration open" (3 window truths), and the word
       "program" itself (3-way overloaded: programs-entity / seasons-view /
       teams-mislabel). Collapsing these to one-source-per-concept (with AP#43
       invariant tests) IS the structural half of the redesign.

  P-3  TWO PARALLEL ADMIN DOORS + A BROKEN LIFECYCLE. /admin/programs and the
       ORPHANED /admin/seasons (no inbound link anywhere) both write the same
       season rows with DIFFERENT status models; AdminSeasonsPage runs retired
       "active|archived-on-create" logic the registry already replaced; draft
       seasons render as "Archived" on the seasons door; there is NO Archive
       action anywhere even though the delete guard tells admins to "archive
       instead"; and in-season there is no nav path to rollover at all.

The redesign is three coordinated tracks: (A) collapse the admin IA + vocabulary
to one door / one lifecycle / one noun, (B) build the parent surface + bridge the
funnel, (C) collapse the PATTERN A #63 sources and finish the small a11y/polish
tail. The coach needs essentially nothing (a read-only program label; a dedicated
coach programs page would be a redundant parallel system, AP#42).

  importing "the programs hook" can grab teams (usePrograms) or program-seasons
  (useAllPrograms) and not know. This is the highest-leverage SIMPLIFICATION and
  it is invisible in any per-file pass — it only appears across files.

§2. PER-ROLE CAPABILITY MATRIX + THE GAPS (file:line grounded)
==============================================================================
ADMIN — essentially complete CRUD owner. CAN: list/filter/group, create
  (basics+window+publish+divisions/fees), detail + read-only registrations,
  edit, delete (dependency-guarded), activate draft, manage teams in-place,
  program-health KPIs, link out to Financials. CANNOT: mark a registration
  confirmed in-app (ProgramDetailPage:14-15 "fast-follow"; the list is read-only,
  forcing the most common per-reg action out to Financials); SEE or EDIT
  divisions/fees after create; ARCHIVE a program; switch active season from the
  programs door (only the orphaned seasons page does that).

COACH — ZERO programs surface. CoachHomePage consumes useActivePrograms ONLY for
  the off-season gate (:46-47). Everything a coach does is team-scoped (roster,
  schedule, comp/pay). A program is an admin lifecycle/billing entity; the teams
  UNDER it are already covered. RECOMMENDATION (labeled lean): do NOT build a
  coach programs page (AP#42 redundant parallel system). The one defensible
  addition is a read-only "program/season" label on the coach's team context
  ("10U Black · Spring 2026") for multi-program clarity — not a surface.

PARENT — ZERO programs/registration surface (the headline gap). After creating an
  account a parent sees kid name + team (AccountPage:70-82, team name not program),
  schedule, records, messages — and NOTHING about: their registration, the
  program/season, what they owe, or how to re-register. family_balances is read
  only by admin/coach surfaces. The registration confirmation dead-ends
  (RegisterConfirm:23-24 → back to public page) with no account-claim CTA, despite
  guardian email being captured explicitly as the magic-link claim target
  (StepGuardian:7-8) and claim_guardian_by_email already wired (claimInvite.js:39).
