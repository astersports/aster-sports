## PART 2 — BUGS / DATA-CORRECTNESS ISSUES

**B1. "Team Pulse · 2% Going" misreads when check-ins are absent (CRITICAL).**
`src/components/gameday/TeamHeatmap.jsx:30-41` computes
`teamPct = totalGoing / totalPast`. `useAttendanceData.js:111`
increments `goingCount` only when
`arrival === 'arrived' || checkedIn || rsvp === 'going'` — so when
check-ins haven't been run and parents haven't actively pressed
"Going" on past events, the metric collapses to "% of past-event
slots where the family pre-RSVP'd Going" which floors at near-zero.
This is the team-scope shape of the same bug §4 ledger Cluster 2
closed at the player-row scope (PR #240 added the "Going" suffix).
The grid in the screenshot is mostly empty with red dashes — those
are the `no_response_past` cells. The header reads "2% Going" but
the grid says "you don't have data here." Two conflicting signals
on the same surface.

Fix shape (no code yet): either gate the pulse on
`totalPast > 0 && responseRate > 50%` (don't show a number under
sparse signal — show "Not enough data yet"), OR re-anchor the metric
on `responseRate` (computed at `useAttendanceData.js:131` already)
so the headline says "Team Pulse · 65% responded · 2% attended" with
both numbers honest. Frank picks via Q3.

**B2. Diagnostic "No RSVPs yet" pill bleeds into parent view from peer rows.**
`PlayerRow.jsx:24` defines `sparseRsvp = responsesReceived <= 1` and
shows the "No RSVPs yet" pill regardless of role. Parents looking
at the roster see this pill on every peer kid — same anti-pattern
class as the "No guardians linked" diagnostic PR #387 caught (kept
in CLAUDE.md §16.14 as a pinned example). The signal is
admin-actionable, not parent-actionable.

**B3. Roster `defaultOpen={isStaff(role)}` is the inverse of §16.14.**
`RosterSection.jsx:14` keeps the 13-player roster always-expanded
for coaches and admins. §16.14 says "closed by default unless
context demands otherwise." On a coach hitting their own team
mid-week, the 13 rows aren't the primary task — schedule +
briefing-send + pulse are. Auto-expand here is the same anti-pattern
as the L99 ArrivalBoard auto-expand that #392 reverted.

**B4. MessageTeamFAB overlaps content on small viewports.**
FAB is fixed at `bottom: calc(80px + safe-area + 16px)` which puts
it on top of TeamAchievements / TeamPlayerStats expanded content.
Same surface but the Send-briefing chip (line 80-86 of
TeamDetailPage) does the same job — two entry points for "message
families about this team" on the same page (briefing portal vs
`/messages?team=`). One canonical action per concern per §16.14
anti-patterns.

**B5. UpcomingEvents weather lat/lng is hard-coded.**
`UpcomingEvents.jsx:16` calls `useWeather(41.03, -73.76)` —
Westchester coords. Multi-tenant code shouldn't hard-code one org's
geography. Out-of-scope for this redesign but worth flagging in the
ledger. Already implicitly tracked under §4 multi-tenant work;
cross-reference at PR time.

**B6. Empty UPCOMING text "Time to work on those crossovers" reads
as admin-flavored.**
Charming for coaches; lands as confusing for parents who weren't
expecting a hoops in-joke. Either keep and ship a parent-flavored
variant, or move to a kindness-microcopy-bucket admin-only string.
§16.3 mandate.

## PART 3 — PER-ROLE INFORMATION ARCHITECTURE

What each persona actually needs on `/teams/:teamId`:

**Parent (parent of a roster player on this team):**
1. Their child's spot + status (jersey #, streak, next-event RSVP).
   Already shipped via MyChildSpotlight — keep but promote into the
   hero.
2. The team's next game/practice: when, where, what to bring.
   UpcomingEvents covers it.
3. Who their kid's teammates are (roster as a directory, not a
   management surface). Roster contact details (phone/email) are
   useful for carpools, but per-kid attendance % pills and the
   "No RSVPs yet" diagnostic are noise.
4. Coach contact + practice location/day. Currently NOT surfaced
   anywhere on this page — team data has `practice_day`,
   `practice_location`, `circuit_name` (per `usePrograms.js:27`
   `.select('*')`) but TeamHeaderCard doesn't render them. Gap.
5. Nothing related to "Send briefing", "Player Stats", "Team Pulse"
   (or only the parent-filtered RSVP-Pulse variant per
   `TeamHeatmap.jsx:39`).

**Coach (coach of this team):**
1. RSVP / attendance snapshot for the next event — "9 of 13 going,
   2 not replied" — actionable before practice.
2. Briefing entry point + Message entry point (these exist as 3
   separate surfaces today: Send briefing chip + CoachQuickActions
   tiles + MessageTeamFAB).
3. Roster as a phone-book + RSVP-trend view. Worth keeping
   defaultOpen=true ONLY when there's an event in the next 24-48h.
4. Pulse / attendance heatmap — close by default, summary text
   "12/13 responded · last 4 weeks" in the header.
5. Player stats — only meaningful once live-scoring runs (per the
   empty state). Default closed, header says "Live-scored games: 0"
   so the coach knows why.

**Admin (org admin):**
1. Everything the coach sees, plus:
2. Cross-team comparison context. The Admin Home already has
   `TeamPerformanceStrip` (`src/components/admin/TeamPerformanceStrip.jsx`)
   — not on this page. Suggest adding a "Switch team" affordance
   richer than the current FilterSelect (PerformanceStrip's W-L tile
   pattern could replace TeamSwitcher when admins land here from
   `/teams`).
3. Roster admin actions: invite missing guardians, copy roster,
   manage jersey numbers. CopyRosterButton + InviteButton exist;
   they're already gated correctly inside PlayerRow.
4. Schedule-import entry. Currently lives under the
   CoachQuickActions "Schedule" tile that routes to
   `/schedule?team=`. Adequate; leave alone.

**Role detection: use `useHomeRole().activeRole`, not `useAuth().role`
directly.**
`TeamDetailPage.jsx:31` reads `role` from `useAuth()` directly.
`useHomeRole` (`src/hooks/useHomeRole.js`) wraps `useAuth().role`
with the admin "view as" preview (anti-pattern #42 —
parallel-systems hazard). On this page admins testing "view as
coach" or "view as parent" will see the real-role variant, not the
previewed variant. The L99 detail pages have the same gap but it
was deferred. The Teams redesign should adopt
`useHomeRole().activeRole` since admin QA of parent/coach experience
is exactly what Frank uses to verify the redesign lands. Q1 confirms.

## PART 7 — OPEN QUESTIONS FOR FRANK

**Q1. Adopt `useHomeRole().activeRole` on TeamDetailPage in PR A?**
Anti-pattern #42 says yes — parallel role-resolution paths on detail
pages are exactly the buildup the rule guards against. Cost: one
import change + threading through to
CoachQuickActions/PlayerRow/etc. (currently those import `useAuth`
directly; some can stay since they're permissions checks per the
useHomeRole docstring scope warning at line 14 — "Other pages use
useAuth().role for permissions checks"). Recommend: page-level
rendering uses `activeRole`, in-row permission checks (InviteButton,
CopyRosterButton) stay on `useAuth().role` since those are real
permissions, not preview. Confirm boundary.

**Q2. What's the canonical Team Pulse metric?**
Today: `goingCount / totalPast` which collapses to RSVP-going-rate
when check-ins aren't running. Three candidate replacements:
- (a) Hide the % entirely when
  `totalPast < 2 || responseRate < 50%`. Header reads "Team Pulse ·
  Not enough data yet."
- (b) Show response-rate instead of attendance-rate: "Team Pulse ·
  65% responded · last 4 weeks." Honest about what we actually
  measure.
- (c) Two numbers: "Team Pulse · 65% responded · 58% attended."
  Requires check-ins to be populated to read sensibly.
Recommend (b) as the headline + (a) as the sparse-data fallback.
(c) waits on the check-in workflow promotion already noted in §9
ledger.

**Q3. Does Roster default closed for staff too, or stay open for
staff and closed for parents?**
§16.14 says closed by default. The L99 ArrivalBoard reversal was the
locked precedent. But the team detail page has more "stable" content
shape than an event — a coach landing here mid-week to do a quick
"who's missing from practice tonight" may genuinely want the roster
open as the dominant block. Recommend: closed for both roles, but
the section header carries the actionable count + responded-last-event
subtitle so the coach reads the answer without expanding. If the
coach taps, expansion is one motion.

**Q4. Parent "Teammates" collapsible — do we strip the per-kid RSVP
pill stack or keep it?**
Today PlayerRow renders the same goingCount/maybeCount/declinedCount
pills for all roles. Per B2 fix, the "sparse" diagnostic role-gates.
Question is whether the FULL pill stack (when data is rich) should
show to parents too — it's not strictly diagnostic but it is per-kid
attendance data shown across the roster. Recommend: hide all per-kid
RSVP pills from parent view EXCEPT on their own kid's row (where
`isMyChild === true`). Keeps the spotlight + matches the
parent-filtered Pulse pattern.

**Q5. Coach contact line in the parent hero — where does the data
come from?**
The hero wireframe in Part 4 [B] shows
"Coach Mike · 914-555-0140." Today's `teams` row doesn't carry a
head-coach FK on its face; `team_staff` table exists (per the
AdminHomePage `team_staff` query at `CoachHomePage.jsx:47-51`).
Need: a `useTeamHeadCoach(teamId)` helper or a join inside
`usePrograms` to surface the head coach's name + contact. If not yet
feasible, skip the coach contact line in PR B and add in PR C /
defer.

**Q6. PR A first as the bug-closure ship, or one combined PR?**
Same shape as L99 Q4. Recommend split — PR A is the trust-recovery
ship (B1's "2% Going" reaches families via screenshots Frank takes);
PR B is structural and benefits from a clean revert path.

**Q7. Hard-coded weather lat/lng (B5) — fix in this arc or queue
separately?**
`UpcomingEvents.jsx:16` calls `useWeather(41.03, -73.76)`.
Multi-tenant gap. Recommend: log to §4 ledger as a separate cluster,
leave alone in this arc. Don't expand scope.

**Q8. Hero parent-RSVP picker — render every kid on this team, or
only the kid associated with this team?**
`TeamDetailPage.jsx:61` already finds `myChild` as the first child
whose `teamIds.includes(teamId)`. A parent with multiple kids on the
same team (siblings on 10U Blue) gets only one rendered. Recommend:
iterate over `myChildren.filter(c => c.teamIds?.includes(teamId))`
and render ChildRsvp per kid, same as the L99 event-detail hero [D]
wireframe. Confirm.

## PART 8 — RISKS / DEPENDENCIES

**Risk R1: Admin power-features regressing.**
Today's page is admin-skewed: roster controls (search + 5 sort chips
including Att%), CopyRosterButton, InviteButton, TeamPlayerStats.
PR B's hero collapses the Send-briefing chip and CoachQuickActions
into the hero action row — admin shouldn't lose access to "Schedule"
deep-link. Either preserve it in the overflow menu (PR C) or as a
tertiary link inside UpcomingEvents collapsible header.

**Risk R2: Role-detection drift.**
Per Q1: adopting `useHomeRole().activeRole` on the page is the right
call but introduces a divergence between page-level rendering and
child-component permissions checks. The pattern needs to be
documented in CLAUDE.md as an extension to §16.14 — "detail pages
render against `activeRole`, in-row permissions check `realRole`."
If this isn't locked it'll drift back within weeks.

**Risk R3: Cross-surface invariant tests not yet load-bearing for
this page.**
PR A's `TeamPulseHeaderInvariant.test.jsx` and PR B's
`TeamDetailHeroPerRoleInvariant.test.jsx` are new shapes. The
Cluster 2 `playerGoingLabelInvariant.test.jsx` is the only similar
test today. Mention in the PR descriptions that this is the
test-class extension per anti-pattern #43.

**Risk R4: TeamHeaderCard + MyChildSpotlight + CoachQuickActions
deprecation hazard.**
PR B replaces these three with TeamDetailHero. Per anti-pattern #34
(registry/dispatch-table removals), the PR must include a pre-flight
grep proving zero off-page callers of those three components. Quick
scan today: TeamHeaderCard is only imported by TeamDetailPage;
MyChildSpotlight likewise; CoachQuickActions likewise. Safe but
verify in PR-pre-flight.

**Risk R5: Mobile-screenshot drift between this redesign and L99
#393.**
Hero pattern needs to read structurally identical to the event hero
