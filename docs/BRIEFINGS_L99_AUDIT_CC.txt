6. FORKS FOR THE ARCHITECT (the design decisions — none pre-picked, §11.8.2)
------------------------------------------------------------------------------
FORK A — Signature / contacts model (F1+F4+F5). Operator stated preference:
  recap sends CLEAN by default; signatures/contacts OPT-IN. Options to rule:
   (A1) Per-send toggle in the composer ("Include coach contacts" off by default)
        + pick which contacts (coach+#, PD+#, or other contact+#).
   (A2) Org-level default (organizations setting) + per-send override.
   (A3) Keep the voice signature ("— Frank & Coach Kenny", no phone) as default;
        make only the PHONE CONTACT BLOCKS opt-in. (Signature is warmth, no PII;
        phones are the over-share.)
  CC lean: A3 as the default behavior + A1 as the control (signature stays, phone
  blocks opt-in per send, choose-contacts when on). Resolves F1+F4+F5 together:
  one signoff renderer, signature default, contacts gated + team-scoped + dedup'd.
  Architect rules the model; this is parent-facing (hard 10/10), worth precision.

FORK B — Coach compose access (F2+F8). Should a COACH compose/send briefings, and
  how to make it coherent? Options:
   (B1) YES coaches compose recaps: allow coach on a scoped compose route (or route
        all staff CTAs through the modal that already works), fix the dead-end.
   (B2) NO, admin-only: REMOVE the coach-facing hero CTAs that dead-end (coaches
        use read-only /team-briefings; admin composes). Simplest, least code.
   (B3) Coach REQUESTS, admin sends: the hero "Request recap" creates a draft/
        proposal for admin (matches /team-briefings "your admin reviews & sends").
  CC lean: B3 — it matches the existing coach mental model (read-only Radar, admin
  sends) and turns the dead-end button into a real action (request → admin queue).
  But B1 is defensible if you want coaches to self-serve recaps. Architect rules.

FORK C — Forfeit handling (F3). Add an is_forfeit concept at score entry + render
  "W — Forfeit" (no fabricated scoreline)? This is UPSTREAM (score entry +
  game_results schema + recap render), likely its own small phase, NOT compose-UX.
  CC lean: ticket as a small dedicated item (schema + score-entry toggle + render
  branch); out of THIS audit's compose scope but flag it as the honest-bar fix.
