AP #60 — Fact-grounding discipline (CANDIDATE — promotes to permanent on third observed instance with stable findings rate)
--------------------------------------------------------------------------------

Every claim about tool behavior, system state, schema, deployed code, or user experience must be grounded in a verifiable source. Plausible-sounding inference from docstrings, defaults, memory, or pattern-matching is NOT a substitute for verification.

SOURCE HIERARCHY (REGISTERED AS PERMANENT — generalizes existing AP #16.7.1 pattern):

  1. Live system state — Supabase MCP execute_sql, GitHub MCP pull_request_read, file Read, deployed runtime. Highest authority for current behavior.
  2. User's empirical report — when the user reports observed behavior that contradicts a claim, the user's signal is authoritative. Reframe the model immediately; do not defend the inference.
  3. Installed source / type defs / migration files — actual code paths, not just docstrings.
  4. Vendor / tool documentation — only when (1)-(3) are unreachable.
  5. Inference / memory — only when no verification is possible AND explicitly labeled as such.

Tool-maximization sub-rule: The MCP infrastructure IS the verification layer.
Question-asking sub-rule: When verification reveals genuine uncertainty, ask the user the specific question that would resolve it.

Origin case (2026-05-28): unfounded claims about GitHub MCP webhook delivery during the L99 cleanup arc.

AP #61 — Pre-phase-cutover audit gate (CANDIDATE)
--------------------------------------------------------------------------------

Before any phase cutover, perform narrow-scope line-by-line bug + enhancement + redesign-potential surface review of every component touched in the current phase, dispatched as parallel narrow-scope agents per AP #50.

Required outputs: bug surface, enhancement surface, redesign-potential surface, routing decision.
Cross-role coverage mandatory per AP #43.
Stop conditions per AP #56 + #59.

(Detailed dispatch specifics live in the planning layer because their calibration is unproven.)

AP #62 — Comprehensiveness as inference signature (CANDIDATE)
--------------------------------------------------------------------------------

When producing a list (categories, redesign items, audit batches), uniform weight across items is itself a warning signal. Real lists derived from observation are uneven — some categories have many sub-items, others are single-point. Categories of equal weight = enumerated rather than observed.

Trigger: if the list "feels comprehensive" (round counts, similar-sized items, no outliers), STOP and verify each entry against an observed source.

Origin case (2026-05-28): the 27-category audit taxonomy in Round 1 of this very doctrine review.

AP #63 — Interim workflow cannot be claimed as permanent contract (CANDIDATE)
--------------------------------------------------------------------------------

When a regression forces an alternate workflow (e.g., "verify-before-stack" while the merge-webhook regression is unresolved), that workflow is INTERIM. It carries an explicit fix-condition or expiration. It does not calcify into doctrine unless and until the regression is permanently accepted as not-fixable.

Origin case (2026-05-28): verify-before-stack proposed as workflow during the merge-webhook regression. Must remain framed as interim until the webhook regression is closed.

AP #64 — Regression-vs-limitation framing (CANDIDATE)
--------------------------------------------------------------------------------

When user reports X doesn't work, and a docstring says X should work, treat the contradiction as regression-evidence, NOT as docstring-overrides-user. Per AP #60's source hierarchy, the user's empirical signal outranks vendor documentation.

Trigger: any time the response to "X doesn't work" begins with "the docs say…" — STOP. Reframe as regression investigation. Propose diagnostic instrumentation.

Origin case (2026-05-28): GitHub MCP webhook merge events — defended a docstring read against the user's 2-month empirical history.

AP #65 — Doctrine artifacts produced without grounding (CANDIDATE — meta-AP)
--------------------------------------------------------------------------------

The higher an artifact's authority, the more grounding it requires — not less because it's "obvious" or "comprehensive." Doctrine artifacts (CLAUDE.md edits, AP registrations, build-target rules) cannot be drafted primarily from inference. Each clause requires either:
  a. A grounded source (live system, user empirical report, source code, prior session evidence)
  b. An explicit "candidate" or "hypothesis" label

The discipline applies recursively: a doctrine artifact that adds new rules cannot itself violate AP #60.

Origin case (2026-05-28): this very doctrine review — drafted with comprehensive-feeling content that wasn't grounded against AP #60's source hierarchy. The Round 1 reviewer caught it.
