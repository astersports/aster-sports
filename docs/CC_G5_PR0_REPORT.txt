0. WHY THIS IS PR 0 (gates G5, not hygiene)
------------------------------------------------------------------------------
Four send helpers wrote comms_messages.status='sent' after invoke, checking ONLY
the transport error, never dispatch.data.ok:
  digestSend.js:125 · rsvpNudgeSend.js:93 · academyCallupSend.js:91 ·
  scheduleChangeSend.js:99.
A partial send (edge fn returns ok:false) force-finalized the message. Then the
alreadySent 409 guard (send-tournament-message:101, keyed on sent_at) fires on
ANY re-invoke — 409-LOCKING the message's own recovery: G5's `failed` re-drive
AND a human "Resend" could never process the stuck rows. A falsely-finalized
message lies to its own recovery. G5 must sit on a floor that tells the truth.

1. THE FIX (mirror the main path, composerSubmit.js:132-142)
------------------------------------------------------------------------------
In each of the four helpers:
  - REMOVED the redundant client `update({status:'sent'})` — the edge fn owns the
    terminal finalize (service role, authoritative). It already does this on the
    main composerSubmit path.
  - ADDED the `if (dispatch && dispatch.ok === false) throw ...` surface so a
    partial send raises instead of reporting success (the SEND-3 fix).
  - Captured `data: dispatch` from invoke where the helper only captured `error`.
No edge-fn change; no schema change; behavior change is the intended one (a
partial send now throws instead of silently reporting sent).
