GOAL
------------------------------------------------------------------------------
Make camps/clinics (and the rest of the program_type enum) self-serve in-app:
create a non-season program, create teams of a chosen type under it, manage
those teams, and (already works) assign events to them. Remove the need to SQL-
seed what the camp arc just seeded by hand.

(b) DEEP-READ ADDENDUM  (second-pass findings the gap doc didn't surface)
------------------------------------------------------------------------------
D1. PROGRAM_NOUN drift — src/lib/home/coachHomeData.js:43 maps
    { season, camp, group_training, clinic } to display nouns. 'group_training'
    is NOT in the enum; tryout / evaluation / interest_list have NO noun. When
    program create opens to all six types, this map must cover all six or the
    coach-home grouping renders blank/undefined nouns. (Mechanical fix; flagged
    so it isn't missed — same orphan-map class as AP #34/#38.)
D2. Divisions/fees coupling — ProgramSetupPage couples program create to a
    Divisions section (divisions + division_fees). Camps/clinics often have a
    flat fee and no divisions. Forcing a division on a camp is wrong shape.
    -> §7 OPEN DECISION 2.
D3. circuit vs team_type axis — the form's "Competition type" chip
    (aau/league_play/tournament) is orthogonal to team_type_id and does NOT
    drive classification. Adding a team_type control means the form now has TWO
    type-ish controls; spec keeps both (circuit = AAU sanctioning label for
    competitive teams; team_type = competitive-vs-not). Do NOT merge them.
D4. AdminTeamsPage empty-state — "No active season … create a season before
    adding teams" (AdminTeamsPage.jsx:46-56) assumes season context. A program-
    switcher rework must not strand camp-only orgs behind a "no active season"
    wall.
D5. ProgramDetailPage already exists as the per-program home (/admin/programs/:id)
    — the natural host for "this program's teams," which argues for adding team
    management THERE rather than only on the season-scoped /admin/teams. -> §7
    OPEN DECISION 4.
D6. SeasonContext drives usePrograms' season scoping. A program-aware team list
    needs a program selector decoupled from the global active-season context, or
    it will keep snapping back to the active season. Non-trivial; see §5 STEP 3.

7. OPEN DECISIONS — RATIFY BEFORE PR LOCK  (recommendations included; AP #39 —
   where I lean, I commit; the hedge is not the answer)
------------------------------------------------------------------------------
D1. Status default on create, per program_type.
    Season already defaults 'archived' (guard against a 2nd active season
    hijacking the home cards). Recommendation: non-season types
    (camp/clinic/group-less) default 'active' (they don't touch the seasons view
    so they can't hijack season cards, and self-serve wants them live
    immediately); tryout/evaluation/interest_list default 'archived' (pre-launch
    states). RECOMMEND: camp/clinic active; tryout/eval/interest archived.

D2. Do non-season programs get Divisions/fees?
    Recommendation: HIDE the Divisions section for non-season types in PR-1
    (camps bill a flat fee, modeled later in the Phase-2 enrollment archetype,
    not as season divisions). RECOMMEND: divisions = season-only for now.

D3. team_type default from program_type, override allowed?
    Recommendation: default the team_type chip from the parent program_type
    (camp->clinic_camp, season->game_team, etc.) but allow override (a season
    can have an academy team; a camp could have a hybrid). RECOMMEND:
    smart-default + override.

D4. Where does team management live (W3 α vs β)?
    Recommendation: β-leaning — put team management on ProgramDetailPage
    (/admin/programs/:id) since it already exists as the per-program home, AND
    add a lightweight program switcher to /admin/teams for the common "jump
    between programs" case, sharing the Phase-3 portfolio switcher component.
    RECOMMEND: teams on ProgramDetailPage + switcher on /admin/teams (shared
    component with Phase 3).

D5. 'Mixed' age_group — just add to the chip set? RECOMMEND: yes, add 'Mixed';
    keep 8U-12U. Trivial, low-risk.

8. SEQUENCING WITH HOME-REDESIGN PHASE 3
------------------------------------------------------------------------------
PR-3's program switcher (W3) and the Phase-3 portfolio rollup + program switcher

switcher; if this spec lands first, PR-3 builds the switcher Phase 3 then reuses.
