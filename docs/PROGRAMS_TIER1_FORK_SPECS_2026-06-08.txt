THE FIVE REQUIREMENTS (architect-ratified set; one txn, NOT separate PRs):
 (1) status 'draft' not 'planning'
 (2) team-copy carries team_type_id := source.team_type_id   [makes S2 safe]
 (3) roster insert WITHOUT roster_type (correct cols only)
 (4) atomic all-or-nothing + audit row written in the same txn
 (5) activate-on-rollover decision inside the txn (see OPEN Q-A)

PROPOSED RPC (signature + body sketch — architect ratifies shape):
  CREATE OR REPLACE FUNCTION public.rollover_season(
    p_from_season_id uuid,
    p_plan           jsonb     -- { newSeasonName, startDate, endDate,
                               --   carryLocations, activate,
                               --   teams:[{ src_team_id, name, team_color,
                               --     sort_order, age_group, division, circuit,
                               --     players:[{id, action}], coaches:[{user_id,
                               --     role, keep}] }] }
  ) RETURNS season_rollovers
  LANGUAGE plpgsql SECURITY DEFINER SET search_path = public AS $$
  DECLARE
    v_org_id uuid; v_new_season uuid; v_new_team uuid;
    v_carried int:=0; v_advanced int:=0; v_dropped int:=0; v_coaches int:=0;
    v_teams int:=0; v_row season_rollovers; t jsonb; p jsonb; c jsonb;
  BEGIN
    -- ORG-OWNERSHIP ASSERT (caller's org must own the source season).
    SELECT org_id INTO v_org_id FROM seasons WHERE id = p_from_season_id;
    IF v_org_id IS NULL OR v_org_id <> ANY (current_user_org_ids()) THEN
      RAISE EXCEPTION 'not_authorized'; END IF;     -- see OPEN Q-C (helper choice)

    -- (1) new season as DRAFT, lineage set
    INSERT INTO seasons (org_id, name, start_date, end_date, status, parent_season_id)
      VALUES (v_org_id, p_plan->>'newSeasonName',
              (p_plan->>'startDate')::date, (p_plan->>'endDate')::date,
              'draft', p_from_season_id)
      RETURNING id INTO v_new_season;

    -- carry season_locations (optional)
    IF COALESCE((p_plan->>'carryLocations')::boolean, true) THEN
      INSERT INTO season_locations (season_id, location_id)
      SELECT v_new_season, location_id FROM season_locations WHERE season_id = p_from_season_id;
    END IF;

    -- recreate teams (2: carry team_type_id) + roster (3: no roster_type) + staff
    FOR t IN SELECT * FROM jsonb_array_elements(p_plan->'teams') LOOP
      INSERT INTO teams (org_id, season_id, name, team_color, sort_order,
                         age_group, division, circuit, team_type_id)
        VALUES (v_org_id, v_new_season, t->>'name', t->>'team_color',
                NULLIF(t->>'sort_order','')::int, t->>'age_group', t->>'division',
                COALESCE(t->>'circuit','aau'),
                (SELECT team_type_id FROM teams WHERE id = (t->>'src_team_id')::uuid))
        RETURNING id INTO v_new_team;
      v_teams := v_teams + 1;
      FOR p IN SELECT * FROM jsonb_array_elements(t->'players') LOOP
        IF p->>'action' = 'drop' THEN v_dropped := v_dropped + 1; CONTINUE; END IF;
        INSERT INTO roster_members (team_id, player_id, registered_at)
          VALUES (v_new_team, (p->>'id')::uuid, now());   -- (3) no roster_type
        v_carried := v_carried + 1;
        IF p->>'action' = 'advance' THEN v_advanced := v_advanced + 1; END IF;
      END LOOP;
      FOR c IN SELECT * FROM jsonb_array_elements(t->'coaches') LOOP
        IF COALESCE((c->>'keep')::boolean, false) THEN
          INSERT INTO team_staff (team_id, user_id, role)
            VALUES (v_new_team, (c->>'user_id')::uuid, c->>'role');
          v_coaches := v_coaches + 1;
        END IF;
      END LOOP;
    END LOOP;

    -- archive the source season
    UPDATE seasons SET status='archived', rolled_over_at=now() WHERE id = p_from_season_id;

    -- (5) activate-on-rollover (OPEN Q-A). If activating, do it HERE in-txn,
    -- archiving the org's other active season of the same scope first (CASE
    -- update so the single-active partial index never sees 0 or 2).

    -- (4) audit row, same txn
    INSERT INTO season_rollovers (from_season_id, to_season_id, org_id, initiated_by,
        players_carried, players_advanced_age, players_dropped, coaches_carried,
        teams_recreated, financial_balances_carried_cents, status, completed_at)
      VALUES (p_from_season_id, v_new_season, v_org_id, auth.uid(),
        v_carried, v_advanced, v_dropped, v_coaches, v_teams,
        0 /* OPEN Q-B financial-carry */, 'complete', now())
      RETURNING * INTO v_row;
    RETURN v_row;
  END $$;
  REVOKE EXECUTE ON FUNCTION public.rollover_season(uuid,jsonb) FROM PUBLIC;
  REVOKE EXECUTE ON FUNCTION public.rollover_season(uuid,jsonb) FROM anon;   -- AP#23/#57
  GRANT  EXECUTE ON FUNCTION public.rollover_season(uuid,jsonb) TO authenticated;

S3 — FORK-REG-UNIQUE  [P0-2]  close the submit_registration TOCTOU
==============================================================================
PROBLEM: registrations has ONLY registrations_pkey(id) (live-verified). The RPC
DOES soft-check "already registered? skip" (SELECT-LIMIT-1 then CONTINUE), but
with NO row lock + NO unique constraint, two concurrent anon submits both pass the
SELECT and both INSERT → double registration + double registration_fees → double
bill. Latent today (0 registrations) but anon-reachable.

RATIFIED AS PARTIAL (architect): a FULL unique would block a player from ever
re-registering after a withdrawal. registration_status enum (live) =
{pending, confirmed, waitlist, cancelled, payment_overdue}; the only TERMINAL
(re-registerable) state is 'cancelled' (payment_overdue is still a live reg that
must block a dup; waitlist/pending/confirmed are live). So the predicate excludes
'cancelled' only.

  -- 1. PARTIAL unique index (CONCURRENTLY not needed at 0 rows; plain is fine).
  --    A partial unique index can serve as an ON CONFLICT arbiter via its predicate.
  CREATE UNIQUE INDEX registrations_program_player_live_uniq
    ON registrations (program_id, player_id)
    WHERE status <> 'cancelled';
  -- 2. make the INSERT race-safe (edit submit_registration body): the existing
  --    INSERT INTO registrations (...) RETURNING id INTO v_reg_id  becomes
  --    INSERT ... ON CONFLICT (program_id, player_id) WHERE status <> 'cancelled'
  --      DO NOTHING RETURNING id INTO v_reg_id;
  --    IF v_reg_id IS NULL THEN  -- lost the race (a live reg already exists)
  --      v_already := v_already || (v_child->'player'->>'first_name'); CONTINUE;
  --    END IF;
  -- (keeps the soft pre-check as a fast path; the partial index is the real guard.)

VERIFY: pg_index shows the partial unique on (program_id,player_id) WHERE
  status<>'cancelled'; replay two rapid submit_registration calls with the same
  (program,player) via execute_sql → exactly one live registrations row, one fee
  set; a third call after cancelling the first SUCCEEDS (re-registration allowed).
NOTE (B5): guardians_org_email_partial_idx(org_id,email) is NON-unique — do not
  reuse it to back any ON CONFLICT.

S4 — FORK-DELETE  [P0-1]  stop program-delete from cascade-nuking financials
==============================================================================
PROBLEM: financial_accounts.season_id → programs is ON DELETE CASCADE (live
confdeltype='c'), AND financial_transactions.account_id → financial_accounts is
ON DELETE CASCADE (confdeltype='c') — so a program delete cascades
program → accounts → transactions, unguarded at the DB. deleteProgram's only
guard counts registrations (0 platform-wide → NO-OP); the confirm names only
teams/players/events. coach_payouts.season_id → programs is ON DELETE NO ACTION
(would already block, but with a raw error).
BLAST RADIUS (architect correction — ALL programs with accounts, not just Fall 2025):
    Fall 2025      archived  64 accts  124 txns   $61,370.00
    Spring 2026    ACTIVE    60 accts   63 txns   $70,242.97   <- live season
    Winter 2025-26 archived  40 accts   57 txns   $34,022.50
    TOTAL                   164 accts  244 txns  $165,635.47  one click from loss.

TWO-LAYER FIX:
  (a) DB defense — flip the cascade to RESTRICT so the ledger can never be
      silently destroyed by a program delete:
        ALTER TABLE financial_accounts DROP CONSTRAINT financial_accounts_season_id_fkey;
        ALTER TABLE financial_accounts ADD  CONSTRAINT financial_accounts_season_id_fkey
          FOREIGN KEY (season_id) REFERENCES programs(id) ON DELETE RESTRICT;
  (b) App message (useProgramAdmin.js deleteProgram, separate build PR): extend
      the pre-check to also count financial_accounts (and coach_payouts) and
      return the kind "archive instead" message BEFORE attempting the delete
      (count-and-block, not catch-and-translate — same pattern already used for
      registrations). Also fix programDelete.js's stale "cascade clean" comment
      that caused the R-2 false closeout.

VERIFY: pg_constraint financial_accounts_season_id_fkey confdeltype='r'; attempt
  DELETE of a program with a financial_accounts row → 23503 (blocked), ledger intact.
LEAN: block + archive (collab §6). Archived programs are never deleted, so RESTRICT
  costs nothing operationally and removes the footgun.
ARCHITECT NOTES: FK-RESTRICT (form (a)) is the ratified shape over a BEFORE DELETE
  trigger — declarative, subtraction-over-addition. The trigger is the alternative
  ONLY if product wants a soft/force path. LEAVE financial_accounts.org_id →
  organizations ON DELETE CASCADE as-is: an org-level delete legitimately cascades
  the whole financial graph (rarer, intended, bigger operation).
