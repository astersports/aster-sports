2. IMMUTABLE I/O CONTRACT  [CONTRACT]
==============================================================================
Invoked as: supabase.functions.invoke('briefing-ai-draft', { body })

REQUEST (JSON body):
  org_id      : uuid     (always)
  kind        : string   (always)
  mode        : string   (always; 'draft' | 'redraft' — semantics §13)
  audience    : object    (always; {} or { team_id } — semantics §13)
  Exactly ONE of:
    proposal_id : string  (AUTO-PROPOSED lane; no gist)
    facts+gist  : facts:object AND gist:string  (FREE-FORM lane; no proposal_id)

RESPONSE (JSON body) — success (HTTP 200):
  body         : string          (the drafted PROSE)
  card_summary : string          (one-line ~10-second summary)
  facts_used   : array           (facts the draft used — shape §10)
  warnings     : array<string>   (missing-fact / caveat messages)
  -> All four keys ALWAYS present. UI defaults missing keys to '' / [], but emit
     all four.

RESPONSE — failure:
  { error: "<human-readable message>" }   returned IN THE JSON BODY.
  Rationale: supabase.functions.invoke collapses non-2xx into an opaque
  FunctionsHttpError; the hook reads data.error FIRST. You MAY return HTTP 200
  with {error} or a 4xx/5xx WITH the {error} body — the hook handles both, but
  the {error} key in the body is what surfaces the message to the admin. Match
  useImportSchedule's convention.

DELEGATED TO SERVER (the hook does none of this): JSON parse, markdown-fence
stripping, retry-on-bad-output. The client consumes clean JSON only.

==============================================================================
4. REQUEST VALIDATION
==============================================================================
  - OPTIONS -> 204 with CORS (preflight).
  - Require org_id, kind, mode, audience. Missing -> { error: "..." }.
  - Require exactly one of {proposal_id} XOR {facts & gist}. Both/neither ->
    { error: "Provide either proposal_id or facts+gist, not both." }.
  - kind not in the v1 supported set (§13) -> { error: "kind not supported" }
    (the test exercises this exact error path — test case e).
  - Validate org_id matches the caller (defense-in-depth; the JWT already scopes,
    but assert org membership before reading voice_config / facts).

==============================================================================
5. MODE / LANE HANDLING
==============================================================================
5.1 FREE-FORM (AI-1 lane: announcement, custom_message)
  - Input: facts (admin-supplied object, may be sparse) + gist (admin's intent).
  - No DB fact walk required; the admin gave you the facts. Optionally enrich
    from audience (e.g., team name from team_id) but never INVENT.
  - Compose prose from gist + facts in the org voice (§7).

5.2 AUTO-PROPOSED (AI-2 lane: calendar-anchored kinds)
  - Input: proposal_id.
  - [PROPOSED] proposal_id = a comms_messages row id in 'draft' status (the same
    rows briefing-auto-draft-tick already writes per anchor). Load it, read its
    kind + anchor_kind + anchor_id, then call the matching resolveX(anchor,
    {supabase}) to assemble the GIVEN facts. Compose prose for that kind.
    - If you instead store proposals in a dedicated table, report the table +
      columns so AI-2 can render the facts panel from the same source.
  - The proposal supplies the anchor; the resolver supplies the facts; the LLM
    supplies the prose. Three distinct layers, one source of facts (§6).

==============================================================================
6. FACTS ASSEMBLY  [PROPOSED — reuse resolvers; architect may override]
==============================================================================
Recommendation (de-risks PATTERN A "same concept, two sources"): assemble facts
by calling the existing pure resolvers, NOT a new walker.
  - resolveGameRecap({eventId}, {supabase})        -> score, opponent, POG, venue
  - resolveGamesRecap({eventIds}, {supabase})      -> per-game lines + W-L record
  - resolveTournamentRecap({tournamentId}, ...)    -> placement, game log
  - resolveTournamentPrelim({tournamentId}, ...)   -> schedule, venues, logistics
  - resolveScheduleChange({eventId}, ...)          -> before/after diff
  - resolveWeeklyDigest / composeWeeklyDigest      -> week's events
  - (rsvp_nudge / academy_callup_notice have resolvers too if those kinds are v1)
Each returns a `context` object already shaped as the GIVEN facts. Flatten that
context into:
  (a) the structured fact list passed to the prompt, and
  (b) the facts_used array returned to the UI (§10).
Resolvers are pure with injected IO (AP #27): pass options.supabase = the
service-role client. Do NOT add a static supabase import to a resolver.

Missing-fact rule (voice profile §7): when a required fact for the kind's
template is absent (null location.name, no score, etc.), DO NOT fabricate. Add a
`warnings` entry ("venue missing") and instruct the model to leave a clear blank.

==============================================================================
7. PROMPT ASSEMBLY  [PROPOSED]
==============================================================================
SYSTEM message = the voice profile verbatim:
  SELECT voice_config->>'ai_draft_profile' FROM organizations WHERE id = org_id.
  If null/short -> { error: "Org voice profile not configured." } (fail loud).
  [PROPOSED] Mark the system block with cache_control: {type:"ephemeral"} —
  it is large + stable per org, so prompt caching cuts cost/latency materially
  across the admin's draft/redraft iterations.

USER message = a STRICT, structured brief:
  - The kind + its structure template (the voice profile §5 has per-kind
    templates; pass the relevant one explicitly so the model follows it).
  - The FACTS as a labeled list (from §6), each "label: value". State plainly
    that these are verbatim and must not be altered or invented.
  - FREE-FORM only: the admin's gist, framed as "the point to make."
  - The OUTPUT INSTRUCTION: "Return ONLY minified JSON with keys body,
    card_summary, facts_used, warnings. No markdown, no code fence, no prose
    outside the JSON." (Belt: still fence-strip server-side per §9.)
  - Re-state the HARD DON'TS inline (no em dashes; no corporate jargon; signoff
    by name) so they bind even if the system prompt is cached/truncated.

redraft mode (§13): include the prior body + (optionally) the admin's tweak note
so the model revises rather than regenerates from scratch. [PROPOSED]

==============================================================================
8. LLM CALL  [GROUNDED + PROPOSED model]
==============================================================================
  - Endpoint/pattern: copy callClaude() from suggest-briefing-closer
    (api.anthropic.com/v1/messages, anthropic-version 2023-06-01, x-api-key from
    app_secrets 'anthropic_api_key').
  - [PROPOSED] model: latest Claude per CLAUDE.md (Sonnet 4.6 =
    'claude-sonnet-4-6' for cost/quality; bump to Opus for max voice fidelity
    during Frank's tuning phase). The existing fn pins 'claude-sonnet-4-5' —
    match-or-bump is your call.
  - max_tokens sized for a full briefing body (~1500-2500). System+facts via the
    messages API; put the voice profile in a cached system block (§7).
  - Temperature: moderate (voice has personality but facts are fixed). [PROPOSED]

==============================================================================
9. OUTPUT PARSING / RETRY  [PROPOSED]
==============================================================================
  - Strip ```json / ``` fences and leading/trailing prose before JSON.parse.
  - Validate the 4 keys exist + types (body:string, card_summary:string,
    facts_used:array, warnings:array).
  - On parse/validation failure: ONE retry with a terse "return valid JSON only"
    correction appended. Second failure -> { error: "Draft generation failed,
    try again." } (kindness microcopy, CLAUDE.md §16.3).
  - Defensive: scan the produced body for an em dash (U+2014); if present, either
    auto-replace with a colon or add a warning — the voice rule is hard and the
    deterministic engine already enforces it (PR #712). [PROPOSED]

==============================================================================
10. facts_used + warnings SHAPE  [PROPOSED — confirm for the AI-2 panel]
==============================================================================
  facts_used: array of { k: string, v: string } pairs (the test uses {k,v}).
    Example: [{k:"record", v:"5-2"}, {k:"opponent", v:"NY Gauchos"},
              {k:"final", v:"42-38 W"}, {k:"venue", v:"WCC"}].
    AI-2 renders these in a facts panel so the admin can verify the model used
    the real data. Keep keys human-labelable.
  warnings: array of short strings ("venue missing", "no score on file yet").
    AI-1/AI-2 render these inline near the draft.
  Confirm the {k,v} shape (or propose a richer one) so the panel renders right.

==============================================================================
11. ERROR CONTRACT  (see §2)
==============================================================================
  Always surface a human message in { error }. Categories: bad request
  (validation), kind unsupported, voice profile missing, proposal not found,
  resolver/DB error, LLM/transport error, parse failure. Keep messages
  admin-readable (CLAUDE.md §16.3 kindness microcopy), not raw codes.
