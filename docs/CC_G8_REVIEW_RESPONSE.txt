4. DURABLE WINDOW-CLOSURE — A vs B, with the Resend finding
------------------------------------------------------------------------------
The review leaned Option A (Resend idempotency keys). CC verification result:
  - Resend docs are 403 to WebFetch from this environment; I could NOT first-party
    confirm batch-key behavior. Labeling the next as INFERENCE (§11.7 source 5),
    to be verified before relying: Resend's idempotency key is applied PER API
    CALL (one Idempotency-Key header per request), NOT per-recipient within a
    batch. If so, Option A on the batch API does NOT give per-recipient dedup on
    a re-drive — and worse, a re-drive's batch composition differs (already-sent
    rows excluded), so a per-call key wouldn't even match. Per-recipient keys
    would require switching batch.send -> per-recipient emails.send (a throughput
    change that collides with B1's throttle work).
  - Option B (a `sending` claim state) has an IN-CODEBASE PRECEDENT: the change-
    alert dispatcher already does an atomic `queued->sending` claim before
    dispatch (`_changeAlertDispatch.ts:78`) and reverts to `failed` on throw. The
    same pattern on comms_message_recipients (add `sending` to the delivery_status
    CHECK — a small migration) disambiguates: `queued`=never-attempted (re-drive
    safe), `sending`=attempted-unknown (do NOT auto-re-drive; surface). It's
    proven here, survives a state-machine bug less gracefully than A's provider-
    level dedup but doesn't depend on an unverified provider feature.
  CC LEAN (labeled, architect decides): Option B for the durable fix — it's
  in-codebase-proven and provider-independent — UNLESS CC confirms Resend batch
  supports per-recipient idempotency keys, in which case A is the cleaner
  defense-in-depth. Either way, Option C ships first as the pilot floor; the
  durable choice (A or B) is its own post-pilot item, not a go-live blocker
  (pilot mode + send-now-only + C's no-auto-re-drive-of-queued already make the
  pilot safe).
