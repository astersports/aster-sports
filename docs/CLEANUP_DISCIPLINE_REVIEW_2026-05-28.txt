SECTION 2 — PROPOSED DOCTRINE (CLAUDE.md additions)
================================================================================

AP #60 — Fact-grounding discipline (no inference where verification is possible)
--------------------------------------------------------------------------------

Every claim about tool behavior, system state, schema, deployed code, or user experience must be grounded in a verifiable source. Plausible-sounding inference from docstrings, defaults, memory, or pattern-matching is NOT a substitute for verification and compounds confidently-stated wrong answers across sessions.

Source hierarchy (verify against the closest to ground truth available):
  1. Live system state — Supabase MCP execute_sql, GitHub MCP pull_request_read, file Read, deployed runtime. Highest authority for current behavior.
  2. User's empirical report — when the user reports observed behavior that contradicts a claim, the user's signal is authoritative. Reframe the model immediately; do not defend the inference.
  3. Installed source / type defs / migration files — actual code paths, not just docstrings.
  4. Vendor / tool documentation — only when (1)-(3) are unreachable.
  5. Inference / memory — only when no verification is possible AND explicitly labeled as such ("I'm inferring because I can't query X").

Tool-maximization sub-rule: The MCP infrastructure IS the verification layer. Inference becomes acceptable only after the source hierarchy is exhausted. Using the tools IS the discipline, not a step we skip when busy.

Question-asking sub-rule: When verification reveals genuine uncertainty, ask the user the specific question that would resolve it. Cost of one clarifying question is bounded; cost of a wrong inference compounds across sessions.

Trigger patterns that mean STOP-and-verify:
  - About to write "the tool only does X" / "the system always Y" / "this was designed to Z" — what's the verifiable source?
  - User pushes back with empirical contradiction — that pushback is data, not an obstacle. Reframe immediately.
  - I genuinely don't know — say so + name the verification mechanism.

Origin case (2026-05-28): unfounded claims about GitHub MCP webhook delivery during the L99 cleanup arc.

Cross-reference: AP #16.7.1, AP #26, AP #39, AP #40, AP #44.

AP #61 — Pre-phase-cutover audit gate (line-by-line surface before transition)
--------------------------------------------------------------------------------

Before any phase cutover (cleanup arc → multi-program build, current-season → next-season rollover, beta → GA), perform a line-by-line bug + enhancement surface review of every component touched in the current phase. NO surface skim. NO "we'll catch it later." The 40% cascade rate of broad-scope line-by-line per AP #50 is unacceptable; the discipline runs narrow-scope line-by-line in parallel to defeat the cascade.

Required outputs before the phase is "done":
  1. Bug surface — every defect on a touched surface, across parent/coach/admin roles (AP #43 cross-role mandate).
  2. Enhancement surface — every concrete improvement with measurable user-experience lift.
  3. Redesign-potential surface — every architectural item that's defensible but would benefit from restructure; added to §17.4 Known Redesign Backlog.
  4. Routing decision — every item classified ship-this-phase / next-phase / defer, with rationale.

Methodology: L99 per §16.15 + AP #50 — narrow-scope line-by-line audits dispatched in parallel. Each audit covers ONE bounded surface. Cross-batch pattern check per AP #58 surfaces cross-cutting findings.

Cross-role coverage requirement (per AP #43): every surface reviewed against parent + coach + admin variants. A finding "fixed" for one role but creating friction for another is not fixed.

Stop conditions (per AP #56 + #59): the user sets a max-PR ceiling + hard-stop time before dispatch. Cycle does not self-extend.

Discipline registration (2026-05-28): the cleanup arc (PRs #543-#550) → multi-program build cutover is the active phase boundary.

Cross-reference: AP #43, AP #50, AP #56, AP #58, AP #59.

SECTION 3 — PROPOSED §17 BUILD TARGETS (new CLAUDE.md section)
================================================================================

§17.1 — User experience commitment ("first 3 seconds, never wants to leave")
--------------------------------------------------------------------------------

The North Star: any user (parent, coach, admin) opening the app or transitioning between tabs MUST see meaningful content within 3 seconds AND the content must give them a reason to stay. Both halves matter. A blazing-fast empty state still loses them. A rich page that takes 6 seconds loses them differently.

Operational implications:
  - Every page's first-paint critical path is treated as load-bearing. Audit category #1 + #2 is recurring, not one-time.
  - Blocking skeleton gates are presumptively wrong; per-zone skeletons are the default.
  - Every new feature shipped to a user-facing surface includes a render-time check: does this give them a reason to stay, or is it noise they'll scroll past?
  - The 3-second target is measured against prod, not dev (StrictMode double-mount inflates dev numbers).

Verification: browser Performance flame chart in prod; not inference.

§17.2 — Multi-tenant + multi-sport architectural scope
--------------------------------------------------------------------------------

The platform target is multi-tenant (multiple youth-sports orgs) + multi-sport (basketball today; soccer, baseball, lacrosse, etc. as the platform matures). Every architectural decision is verified against the full target before it lands.

Multi-tenant invariants:
  - Every org-scoped table has org_id + RLS scoping by current_user_org_id() per §4 + §11.5 + AP #37.
  - Every query touching an org-scoped table filters by org_id explicitly (AP #37 defense-in-depth).
  - Every UI brand-token application resets to org defaults on login.
  - No cross-org data leak path exists from any role on any surface.

Multi-sport invariants:
  - Sport-specific data lives in sport-specific tables/columns; sport-agnostic data lives in the foundation (events, teams, players, registrations, financials, comms).
  - game_plays and the live-scoring schema are basketball-specific today and will need a sport discriminator when soccer/etc. lands. Don't refactor preemptively (per AP #45 ship-not-gate); flag the seam in §17.4.
  - Terminology surface is parameterized per org (per LeagueApps walkthrough: "Players" vs "Athletes" etc.) — out of scope today.

Multi-program invariants (per LeagueApps walkthrough 2026-05-28):
  - Members are org-level + persistent across programs.
  - Programs are polymorphic (Season / Camp / Class / Club Team / Event / League / Tournament / Booking) — needs a program_kind discriminator added when multi-program registration lands.
  - Registrations are per-program-per-member, not the identity layer.

§17.3 — Phase rigor & methodology
--------------------------------------------------------------------------------

L99 is the audit methodology framework (§16.15). The methodology by scope per AP #50:
  - Broad codebase audit → breadth-via-parallel-agents (low cascade).
  - Specific surface audit → line-by-line at narrow scope (high signal-to-noise).
  - Broad surface line-by-line → ~40% cascade. Don't do this.

Pre-phase-cutover audits run as parallel narrow-scope agents. Cross-batch pattern check per AP #58 — every batch's findings cross-reference prior batches' patterns by letter name (ALPHA, BETA, GAMMA) so cross-cutting findings surface as single concerns.

§17.4 — Known redesign backlog (architectural items deferred but tracked)
--------------------------------------------------------------------------------

  - Realtime channel dedupe context (replaces Math.random() suffix workaround across 7 hooks).
  - SWR/TanStack Query layer for the home-feeder hook fleet (16+ hooks fire fresh on every mount).
  - Blocking-gate decomposition with per-zone skeletons + AP #43 invariant tests.
  - program_kind discriminator on programs table (multi-program polymorphism).
  - Sport discriminator on game_plays + sport-specific play tables.
  - Onboarding pipeline (bulk-invite + QR + status column + PWA install prompt + push opt-in promo on Parent Home).
  - Per-kid spotlight strip on Parent Home.
  - Tokenized RSVP from weekly_digest + Stream A reminder emails.
  - Payment-due card action surface (tel: / mailto: / external link).
  - Reports Hub navigation surface (top-level, mirrors LeagueApps Reporting).
  - Permissive policy consolidation on event_comments/rsvps/duties (60 advisor findings; defer to multi-program scoping).
