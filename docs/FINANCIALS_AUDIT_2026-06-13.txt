CROSS-BATCH PATTERNS (AP#58 synthesis — promoted at ≥3 surfaces)

PATTERN A — SAME CONCEPT, DIVERGENT SOURCE *OR* SCOPE (AP#63). The dominant pattern; 9
instances across 4 agents. The family VALUE is single-sourced, but it is read at different
SCOPES on different surfaces, and the coach "owed" is read from different SOURCES. Instances:
  A1. Coach "owed" — 3 sources: rate×sessions (list) / Σpay_cents (detail) / Σpending-payouts
      (own card). [P0/architectural]
  A2. Coach "paid" — 4th scope split: own card all-seasons ($2,780) vs admin season ($1,980).
      useCoachComp has NO season_id filter at all. [P1]
  A3. Admin "Payments overdue" lane (2 fams/$1,345, no age filter) vs payment_overdue alert
      (1 fam/$1,275, 30-day + >$1 gate) — same concept, two predicates, same screen. [P1]
  A4. Parent home balance ALL-seasons vs admin dashboard ACTIVE-season default — Zinman shows
      $1,275 (Fall 2025) to the parent, $0 (Spring 2026) to the admin landing. [P2]
  A5. "Outstanding" on admin home: shortDollars "$1.3k" (KPI) vs formatCurrency "$1,345.00"
      (lane) — same hook value, two renders. [P2]
  A6. Coach own-card formatter rounds to whole dollars; all admin coach surfaces show cents.
      Latent (all payouts whole-dollar today). [P3]
  A7. Family detail "Balance" Stat shows a raw signed credit ("-$50.00") while RecordPaymentForm
      labels it "$50.00 credit". [P3]
  A8. Coach detail balance shows bare signed "-$680.00" while the list card uses
      Math.abs + "still owed"/"settled" label. [P3]
  A9. (latent vectors) LIST vs DETAIL owed will split the instant a same-team multi-scope
      assignment, an orphan session, an unhonored practices_only scope, or a non-flat pay_cents
      appears — all 4 verified absent today, so the two agree only by data coincidence. [P1 latent]
  FIX CLASS: one source + one scope per concept; where scope legitimately differs, label it;
  lock each with an AP#43 cross-surface invariant test.

PATTERN B — SILENT SUCCESS / SWALLOWED ERROR ON WRITE + SECONDARY-READ PATHS. 4 instances.
  B1. doVoid: console.error then close+refetch as success. [P0]
  B2. CoachPayoutEditSheet.handleDelete: non-atomic two-write, first error discarded. [P1]
  B3. useFamilyLedger: logs error but has no `error` return → a failed fetch renders a family
      as "Paid in full". [P2]
  B4. useFamilyTeams / useCoachPayouts / useCoachDetail: secondary queries log-but-continue →
      a dollar figure computed from a partially-failed fetch, no user signal. [P2]
  NOTE: the AP#36 *hard* form (`{data=[]}` default, no error var) exists NOWHERE — every query
  destructures and at least logs error. This is the SOFT form. Family forms (RecordPayment,
  AddCharge, RecordCoachPayout) all do it correctly (surface warm microcopy, refetch only on
  success); the gaps are the void path + the secondary reads.

PATTERN C — DORMANT-BUT-UNWIRED SCHEMA SUBSTRATE (§16.16 trap). Shaped config/columns with
zero consuming writer, none carrying a DEPRECATED comment or a DEPRECATIONS_REGISTRY row:
  - Settlement substrate: event_coach_assignments.{pay_status='paid', settled_by_payout_id} +
    coach_payouts.source_assignments — 0 writers, 0/137 live. [tied to D1]
  - coaching_assignments.rates jsonb + get_coach_rate_cents() fn + GIN index — 0 readers/writers,
    every row '{}'. [tied to D2]
  - financial_transactions.{receipt_url, applied_to_player_id, registration_id} — 0 readers. [§16.16]
  - financial_accounts.{season_fee_cents, discount_cents} — legacy, fetched-not-used. [§16.16]
  FIX CLASS: per ruling, either wire the writer or COMMENT-deprecate + registry row + drop the
  attached unused indexes. Do not leave shaped-but-unwired.

PATTERN D — THE SETTLEMENT WALL (the architectural core; needs D1). owed is a live recompute
that ignores payments; paid is a free sum that ignores sessions; nothing flips a session to
'paid' or writes settled_by_payout_id. 0/137 sessions settled. CoachPayoutEditSheet even has
dead un-settle-on-delete code (matches 0 rows) — proof the schema was built for D1-option-(b).

PATTERN E — MONEY-SAFETY RESTS ON APP CODE ONLY (no schema guard). 2 instances, both money-
integrity:
  E1. Append-only is enforced by RLS-policy-ABSENCE only — there is NO trigger. Any BYPASSRLS
      path (service_role, a future SECDEF RPC, the SQL editor) can silently UPDATE/DELETE a
      posted transaction.
  E2. Void-pair integrity: reverses_transaction_id has an FK but nothing guarantees the reversal
      row's account_id/type/amount match the row it points at. The view nets pairs by the
      pointer alone, so a mismatched reversal corrupts a balance. The only guard is doVoid
      copying fields in JS.
