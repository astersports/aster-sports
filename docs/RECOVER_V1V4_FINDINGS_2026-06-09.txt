THE EXISTING RECOVERY DESIGN (the load-bearing finding)
================================================================================
The crash-after-dispatch window is ALREADY handled, conservatively, and the limit is
documented in-code:
  send-tournament-message/_dispatch.ts:13-21 (architect G8 review, 2026-06-07):
    "...does NOT cover the crash-after-dispatch window — if the fn dies AFTER Resend
     accepts a batch but BEFORE rows are marked 'sent'... a blind re-drive of 'queued'
     would double-send them. So G5 must NOT auto-re-drive ambiguous 'queued' rows..."
Two-pronged design:
  (a) AUTO re-drive ONLY delivery_status='failed' rows (certain NO send occurred), bounded
      by redrive_count<3 then escalate: briefing-auto-draft-tick/_redrive.ts:118-150,129.
  (b) Ambiguous 'queued' rows (the crash window) -> HUMAN REVIEW, not auto-replay:
      src/components/radar/StuckSendsRegion.jsx + src/hooks/useStuckSends.js (resend or
      mark-delivered, operator decides).
Locked by a static invariant test: src/lib/__tests__/sendIdempotencyInvariant.test.js
  asserts the send only processes delivery_status='queued' rows.

Why this is safe WITHOUT provider idempotency: failed rows never reached Resend, so
re-driving them cannot double-send (idempotency is irrelevant); queued rows are
ambiguous, so they are NOT auto-replayed at all. The design trades a small amount of
automation (human taps "resend"/"mark delivered" on the rare stuck batch) for a hard
no-double-send guarantee.

DECISION FOR THE ARCHITECT (CC lean labeled per AP#39; CC does NOT settle)
================================================================================
Option A (CC LEAN): treat RECOVER as already SATISFIED by the existing G5 design. The
  auto-sweep the ratify doc specified would auto-replay the ambiguous queued window,
  which is the one operation that can double-send real family email and which the
  current design intentionally routes to human review. Marginal automation gain, real
  double-send risk. Net-new build to enable it (idempotency keys + token persistence +
  a batch-unit record + a 24h gate) is large and buys a riskier behavior than what
  ships today. Recommend: declare RECOVER green on the human-review design; optionally
  add a tiny hardening (a dispatched_at on the batch + surfacing age in StuckSendsRegion)
  if you want the 24h signal for the human reviewer — but NOT auto-replay.

Option B: build the full idempotency stack to enable safe auto-replay — persist a
  deterministic batch idempotency key (R1), persist the minted tokens so the replay
  payload is byte-identical (R2), add a batch-unit record so writeback is all-or-nothing
  (R3), and a 24h dispatched_at gate (R4). Only after ALL FOUR can the auto-sweep ship
  per your spec. This is a multi-PR arc, schema-touching (architect lane).

The four R-constraints in your ratify doc are correct; the finding is that meeting them
is a large build, and the existing design already closes the gate a safer way.
