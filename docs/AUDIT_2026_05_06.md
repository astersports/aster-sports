## Headline: The 3-second test FAILS

A first-time parent will not understand within 3 seconds why this app helps them. Five specific reasons:

1. Three sequential "Loading…" gates blank the screen for 1.0–1.5s before any content appears: RequireAuth.jsx:12-18 shows three grey card skeletons with no header, no bottom nav, no branding (parent can't tell what app they're in), then HomePage.jsx:18-21 shows plain-text "Loading...", then ParentHomePage.jsx:80 shows it again. Three different loaders, none branded, none shape-matched.
2. The hero is informational, not motivational. Greeting = "Good morning, / Frank" and nothing else. No org identity, no value-prop line, no "what this app is for." HOME_DESIGN_SPEC §1.1 specifies emergency banners, action zones, live-now cards, tournament-weekend banners — none of the 13 spec sections beyond the basic four are implemented.
3. First-time / empty-tank state is broken — appears as a broken app, not as "we're getting you set up." A parent who hasn't been linked to a kid yet sees: greeting, "No teams yet" empty card, hidden teams strip, broken records button, "Nothing in the next 48 hours." No CTA, no contact-coach link, no orientation.
4. The bell badge is bait-and-switch. Header.jsx:25 — bell tap shows "Notifications coming soon" toast, but useNotificationBadge runs a real query and shows a real number. New parent sees "3" within the first second, taps, gets nothing. Top trust-killer.
5. Nothing tells the parent what the app is FOR. No tagline, no onboarding tooltip, no first-run tour.

The one thing that does work: when a parent already has an upcoming event, NextUpCardMed.jsx:51-138 is genuinely well-designed — comparable to GameChanger / PlayMetrics. The test fails on the empty tank, the chromeless cold-load, and the bell-bait — not on the core card design.

## P0 — Critical bugs (crash, corrupt data, or break a major feature)

### 1. useEventRideCounts queries dead table
**File:line:** `src/hooks/useEventRideCounts.js:19`
**Bug:** Queries dead `event_rides` table — dropped by migration `025_rides_redesign.sql:8`. Every "X seats / X rides needed" pill, conflict hint, and "ride needed" warning silently shows nothing.
**Affects:** Parent + Coach + Admin home, schedule, every event card
**Fix:** Rewrite against `event_ride_offers` (seats) + `event_ride_claims` (`status='pending'`) matching `useEventRidesView.js` shape.

### 2. ClaimSeatForm filters by non-existent column
**File:line:** `src/components/ride/ClaimSeatForm.jsx:15-19`
**Bug:** Filters `eligibleKids` by `offer.team_id` — but `event_ride_offers` has no `team_id` column. Filter never narrows. Multi-kid parent can claim seat for the wrong child.
**Affects:** Parent ride flow
**Fix:** Pass `event.team_id` from surrounding `EventRidesTab` and filter on that.

### 3. Toast never auto-dismisses
**File:line:** `src/components/shared/Toast.jsx:13-18` + `AdminTeamsPage.jsx:131-135`, `AdminSeasonsPage.jsx:140-144`
**Bug:** `onDismiss` passed as inline arrow → new identity each render → effect cleanup cancels timer → toast never auto-dismisses. No manual close button either.
**Affects:** Admin
**Fix:** `useCallback` the `onDismiss` callbacks, OR memo Toast and read latest via ref.

### 4. Rules-of-hooks violation
**File:line:** `src/pages/EventDetailPage.jsx:68`
**Bug:** `useEventDelete(event)` called after early returns at lines 59-60. Real rules-of-hooks violation.
**Affects:** All edit-event paths
**Fix:** Hoist hook above early returns. (Lines 31, 35 — drop unused navigate/showToast and their imports.)
**NOTE:** Verified in this session — lint reports clean. The hook IS above early returns in the deployed version. Audit may have read stale file.

### 5. TeamPerformanceStrip hardcoded "0-0"
**File:line:** `src/components/admin/TeamPerformanceStrip.jsx:30`
**Bug:** Hardcoded "0-0" record on every team card on AdminHome. Data exists (`useOrgTeamRecords`).
**Affects:** Admin
**Fix:** Pass `byTeamId` down, render `summary.record`.

### 6. NextEventCard reads non-existent column
**File:line:** `src/components/admin/NextEventCard.jsx:27-28`
**Bug:** Reads non-existent `event.opponent_name`; schema column is `opponent`. Plus only renders title + teamName + countdown — no date, no time, no location.
**Affects:** Admin
**Fix:** Use `event.opponent`; add date/time/location lines like ParentHomePage's NextUpCard does.

### 7. useEventDutyCounts over-counts claimed slots
**File:line:** `src/hooks/useEventDutyCounts.js:29`
**Bug:** Counts a slot as "claimed" if `claimed_by_name` OR `guardian_id` is set. Wizard pre-populates `claimed_by_name` for display, so admin event cards over-report "filled."
**Affects:** Admin event UI
**Fix:** Use only `guardian_id IS NOT NULL`.

### 8. useEventDelete non-atomic series delete
**File:line:** `src/hooks/useEventDelete.js:13-22`
**Bug:** Series delete = two un-transactional supabase calls. Failure between them leaves orphaned head event with deleted children.
**Affects:** Admin/coach
**Fix:** Wrap in RPC transaction, or single `.delete().eq('parent_event_id', X).gte('start_at', Y).or('id.eq.<head>')`.

### 9. useNotificationBadge shape mismatch
**File:line:** `src/hooks/useNotificationBadge.js:29, 60`
**Bug:** Derefs `c.id`, but `parentContext.js:30` builds myChildren with key `playerId`. `kids` array is `[undefined, undefined, ...]` → `.in('player_id', kids)` matches nothing. Bell badge always shows 0 for parents.
**Affects:** Parent + admin "view as parent"
**Fix:** Change `c.id` → `c.playerId` (both call sites).

### 10. Missing org_id defense-in-depth
**File:line:** `src/pages/EventDetailPage.jsx:46-49` + `src/components/admin/GettingStarted.jsx:18-25`
**Bug:** Queries `event_duties` / `team_players` / `events` without explicit `org_id` filter. Defense-in-depth violation per CLAUDE.md Rule 8. RLS-only safety.
**Affects:** All roles
**Fix:** Add `.eq('org_id', orgId)` to every cross-org-sensitive query.

## P1 — High bugs (visible UX degradation, intermittent breakage)

### Async / data layer

- **useRsvps.fetch** no race-cancellation (`src/hooks/useRsvps.js:12-34`) — rapid RSVP taps can stomp; no cancelled flag, no AbortController. Same shape in `useDuties.js:17-27`, `useComments.js:16-27`, `useCheckIns.js:15-24`.
- **useActivities.refetch** flips `loading=true` even when cache exists (`src/hooks/useActivities.js:32`) — visibility change = full screen wipe + scroll-position loss. `useEventDetail.refetch` deliberately doesn't do this; should match.
- **Module-level cache cross-account leak** — `ChildRsvp.responseCache` (ChildRsvp.jsx:7), `useEventRsvpNotes:6`, `useActivities:10`, `useLocations:5`, `useTournaments:7`. Sign-out at AuthContext.jsx:103-108 clears auth state but not these caches. Next user on the same browser sees previous user's data until refetch completes.
- **useCreateActivity** non-atomic series + duty creation (`useCreateActivity.js:51-89`) — duty-insert failure leaves orphan events. Same in `seriesReconcile.js:36-50`. No transaction.
- **useTournaments.create** non-atomic (`useTournaments.js:89-104`) — three writes, no transaction.
- **useEventDetail.js:18** uses `.single()` not `.maybeSingle()` — errors when row missing, console noise plus wrong UX path.
- **parentContext.fetchParentContext** keeps only first team per child (`lib/parentContext.js:32-33`). **FIXED in Wave 4** — now reads `parent_context_v` which returns all active teams.

### React correctness

- **useScoreDraft.performSave** retry timer never cancels on unmount (`useScoreDraft.js:92`) — only `debounceRef` and `flashRef` cleaned up. `flashRef.current = setTimeout(...)` at line 88 also overwrites without clearing previous.
- **SchedulePage** lazy-init reads URL once (`SchedulePage.jsx:24`) — `selectedTeam` is sticky if URL changes via tap. Use `useSearchParams`.
- **RsvpPlayerRow.noteText** desyncs from server (`RsvpPlayerRow.jsx:20`) — initialized from `existingNote` once; after save → refetch → prop updates but local state stale.
- **NextUpCardMed + NextUpCardMax** double-timer (`NextUpCardMed.jsx:30-36`, `NextUpCardMax.jsx:24-27`) — `useNow()` already ticks every 60s; cards run a second `setInterval` for countdown. Two clocks tick out of phase; countdown can lag urgency class by a full minute.

- **ToastProvider** value re-creates each render (`ToastProvider.jsx:14`) and `PreferencesProvider.updatePreference` deps include `preferences` (`PreferencesProvider.jsx:84`) — both cause cascading re-renders across all consumers.

### Schema / canonical-table drift

- **useEventRsvpCounts.js:28** reads `roster_members` for "team size" — CLAUDE.md §11.5 names `team_players` as canonical for current membership. Plus no `.is('left_at', null)` filter → inflates `noResponse` counts with departed players.

## Cross-cutting patterns

### Strengths

- Hook return shape `{ data, loading, error, refetch }` is consistent.
- Microtask `Promise.resolve().then(...)` wrap to satisfy set-state-in-effect is uniform across most hooks.
- `useEventDetail.refetch` deliberately does not flip `loading=true` — preserves scroll. Pattern should propagate.
- Phase 1.5 rides rebuild (`useRideOffers`, `useRideClaims`, `useEventRidesView`) is well-isolated: realtime channels with proper `removeChannel` cleanup, optimistic insert + rollback, atomic RPC for waitlist promotion.

### Weaknesses

- Org_id scoping is RLS-only in many surfaces — Rule 8 demands defense-in-depth.
- `roster_members` vs `team_players` inconsistency — five+ surfaces read the wrong canonical table per §11.5.
- `window.confirm` pervasive on destructive actions — wider than DEFERRED B4 names (8+ surfaces).
- Module-level caches never invalidated on sign-out — cross-account stale-data bleed.
- `<div onClick>` shape used everywhere — no `<TappableRow>` primitive; same a11y bug copy-pasted.
- Color-only state signaling repeated across RSVP, payment, urgency.
- TZ inconsistency — browser TZ on EventDetailTab vs NY-anchored elsewhere.
- Optimistic-with-rollback done correctly in rides; missing in `useRsvps.setRsvp`, `useDuties.claim`, `useComments.post`.
