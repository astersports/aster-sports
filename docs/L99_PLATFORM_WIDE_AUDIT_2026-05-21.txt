**P2. Two fragility points worth registering:**

**E1. Synthetic-row carve-out is fragile (`digestSend.js:121-122`).**
```
allSynthetic = renderedFamilies.every((f) => f.family.guardian_id == null);
effectiveTestOnly = testOnly && !allSynthetic;
```
If synthetic rows ever carry non-null guardian_id in the future, test-mode carve-out silently breaks. The comment explains intent but the guard is implicit. **Recommend:** explicit `is_synthetic` boolean on the slice, replacing the guardian_id-null heuristic. Same fix shape applies to any other heuristic-typed guard across the engine.

**E2. Magic HTML wrapper duplicated.**
`digestSend.js:53` and `rsvpNudgeSend.js:25-26` both hardcode:
```
<div style="max-width:600px;margin:0 auto;background-color:#ffffff;font-family:Inter,system-ui,sans-serif;padding:0 0 24px 0;">
```
If scheduleChangeSend or academyCallupSend already does the same (likely), 3-4 callsites of identical hardcoded HTML. Extract to `EMAIL_WRAPPER_OPEN` constant in a shared lib module.

- academyCallupSend.js token minting flow

2.4 EDGE FUNCTIONS + SECURITY (STRONGEST POSITIVE)
────────────────────────────────────────────────────────

**Verdict: PASS. No P0 violations across 11 edge functions, app_secrets, token handlers, JWT minting, or recent migrations.**

The late-cycle anti-pattern additions are holding:

**#31 (verify_jwt config):** All 11 functions correctly declared. `verifyJwtConfigAudit.test.js` enforces this in CI.
- verify_jwt=false (token-based): callup-token-handler, rsvp-token-handler, unsubscribe-handler, resend-webhook-receiver, team-feed
- verify_jwt=true (JWT-based): invite-parent, send-tournament-message, parse-tournament-schedule, suggest-briefing-closer
- Bearer/CRON_SECRET: briefing-cron-dispatch, briefing-auto-draft-tick

**#33 (app_secrets vs Deno.env):** All shared secrets sourced from app_secrets via getAppSecret() helper or SECURITY DEFINER verify_*_token RPCs. The only Deno.env shared secret is `RESEND_WEBHOOK_SECRET` — platform-managed by Resend, documented exemption.

`getAppSecret()` helper fails LOUD on NULL (no silent fallback), returning admin-diagnosable error message. Good failure mode.

**#23 (REVOKE FROM PUBLIC):** Recent migrations (2026-05-20) correctly REVOKE EXECUTE ON FUNCTION ... FROM PUBLIC before GRANT to authenticated + service_role. Pattern locked.

**Token handlers (rsvp, callup, unsubscribe):**
- HMAC verification via SECURITY DEFINER RPC ✅
- Expiry validated at RPC layer ✅
- Replay protection: nonce single-use via unique-constraint race handling. Re-tap of same token returns prior response (idempotent) ✅
- CORS appropriate to function shape (token handlers no CORS — stateless GET → redirect; JSON functions allow OPTIONS+POST)

**briefing-cron-dispatch JWT minting:** HMAC-SHA256 with 5-minute expiry, secret from app_secrets, claims = {sub, exp, iat}. Architectural seam flagged in CLAUDE.md anti-pattern #33 — cleaner refactor is to bypass send-tournament-message JWT verification entirely, but the current pattern is locked and safe.

**Recent 5 migrations** (sync_opponent_record, sync_tournament_team_record, etc.): all SECURITY DEFINER functions REVOKE/GRANT correctly; no cmd='ALL' policies without with_check; ghost migrations present at repo paths per CLAUDE.md §5.

**P1. One documentation gap:**

**S1. Edge function mirror byte-compare not automated.** Per anti-pattern #30, six edge functions have helper mirrors (_helpers.ts, _handlers.ts) at src/lib/* for vitest coverage. The mirrors are documented but there's no automated byte-comparison test detecting drift between the two trees. **Recommend:** add a CI assertion that diffs the .ts files against their .js mirror counterparts (ignoring TS-annotation-only differences). Same shape as `verifyJwtConfigAudit.test.js`. Closes the drift class structurally rather than relying on review discipline.

PART 8 — METHODOLOGY NOTES FOR FUTURE PLATFORM AUDITS
═══════════════════════════════════════════════════════════════════════════

What worked:
- **Parallel Explore agents** for non-overlapping subsystems. 4 agents covered ~700 files in one batched dispatch. Sequential would have been 4x the wall-clock.

- **Hot-spot deep reads** (Teams audit) before platform pass. The Teams audit's catalog of anti-patterns (V/C/D/A prefixes) became the platform audit's pattern-scan inputs.

What to refine next time:
- **Agent tool-budget allocation.** Each agent had ~50 tool uses and 3 of 4 ran out before reaching "deferred" zones. Next time: allocate more budget OR scope each agent to a narrower zone (1 subsystem each rather than 1 cluster each).
- **Explicit "verified vs deferred" reporting.** Engine audit agent's "compliant but not verified" language was honest but could surface earlier in the agent output. Lock as part of the agent prompt template.
- **Cross-reference Teams audit findings as platform pattern-scan inputs.** Worked once; should be locked as methodology.
