VERDICT
==============================================================================
THE STAGE GOAL IS DONE. The headline build — the admin notification-policy surface
(S9 Channels + Alerts + Automatic messages) plus S1/S2 self-settings — is shipped and
its HIGH-VALUE controls are WIRED. The NotificationPrefs duplicate is removed; Channels
is the single source.

BUT a close-out audit surfaces a WRITE-ONLY TAIL — the platform's known "data plumbing
terminates in admin UI blindness" pattern (pre-existing; NOT introduced this stage). On
the settings surfaces specifically, 5 controls:

  PARENT-FACING (silent broken promises — higher stakes):
    1. Quiet Hours — a parent sets quiet hours; the send path ignores them (fixed ET).
    2. Weekly-digest opt-out — a parent turns off the weekly digest; it still sends.
       (Game-recap / org-announcement opt-outs: same — digest path doesn't filter them.
        Tournament-briefing opt-out DOES work.)
  ADMIN-ONLY (cosmetic — lower stakes, no parent harm):
    3. Registration (org registration_open) — drives nothing; registration is
       per-program (see REGISTRATION note). MISLEADING — implies an org-wide on/off
       that does not exist.
    4. Features toggles (Futures Academy / Carpool) — drive nothing.
    5. Custom domain — drives nothing.

REGISTRATION — the model (answers Frank's question, ties to the programs redesign)
  Registration open/close is PER-PROGRAM and already correct in the schema:
  programs.reg_opens_at / reg_closes_at (the window) + status ('active' gates public
  visibility) + is_published + public_slug. get_public_program enforces all of it per
  program. There is NO working org-wide registration shutdown — and there shouldn't be
  one; each program activates/closes on its own window. So the org Settings → Registration
  row is not just dead, it contradicts the right model. In the programs redesign,
  registration belongs on the PROGRAM (open/close window + status), and the org-level
  Settings row should be removed. (If a true "freeze ALL registration org-wide" kill-switch
  is ever wanted, that's a deliberate new feature — get_public_program would AND it in —
  not the current vestigial toggle.)
