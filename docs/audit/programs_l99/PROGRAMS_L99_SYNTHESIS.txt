1. THE TWO PRODUCTION P0s (act first; both are migration-lane fixes → architect)

P0-1 — PROGRAM DELETE BLAST RADIUS  [A3 (a)P0-1; A6 TEST-3]
  src/hooks/useProgramAdmin.js:44-56 · src/lib/programDelete.js:4-7
  deleteProgram() pre-checks ONLY registrations (FK RESTRICT). financial_accounts
  .season_id → programs is ON DELETE CASCADE, and financial_transactions cascade
  through financial_accounts. The confirm dialog (dependencySummary) lists only
  teams/players/events — never financials, divisions, tryout_sessions,
  season_locations, player_equipment (all CASCADE) or coach_payouts (NO ACTION →
  raw 23503). The lone guard (registrations RESTRICT) is a NO-OP for every
  production program: 0 registrations platform-wide (the LeagueApps financial
  import predates the registrations table).
  LIVE EVIDENCE: Fall 2025 (archived) = 0 regs, 64 financial_accounts, 181
    financial_transactions (~$61,370). Winter 2025-26 = 0 regs, 40 accounts.
    Spring 2026 = 0 regs, 60 accounts, 2 coach_payouts.
  COMPOUNDED BY (A3 (b)): useProgramAdmin count-error is console-only and the
    confirm opens synchronously before the async refetchCounts resolves — the
    dialog can render "0 teams / 0 events" (stale/initial), reinforcing the
    false-safety. (AP#36 + a stale-counts race.)
  LEAN (A3, committed): BLOCK delete when financial_accounts>0 (mirror the
    registrations guard) — "This season has N family financial records — archive
    it instead." Add coach_payouts to the count-and-block. Archived seasons ARE
    the financial record; they should not be deletable. → FORK-DELETE.

P0-2 — REGISTRATION TOCTOU DOUBLE-BILLING  [A1 (b)P0; A5 confirms write path]
  supabase/migrations/20260601044918_registration_public_rpcs.sql:150-167
  The anon-executable submit_registration RPC's "already registered?" guard is a
  non-atomic SELECT-then-INSERT with NO UNIQUE(program_id,player_id) backing it
  (verified absent in pg_constraint). Two concurrent submits (double-tap, two
  devices) both read zero rows → both INSERT a registration + a full set of
  registration_fees → family billed twice. READ COMMITTED + SECURITY DEFINER does
  not serialize them. Current zero-dupe count is low-traffic luck.
  Otherwise the RPC is well-built: org_id server-derived (no spoof), is_published
  + reg-window enforced server-side, fee total summed server-side, division→
  program coupling checked. The TOCTOU is the lone defect.
  LEAN (A1, committed): add UNIQUE(program_id,player_id) + change the guard to
    INSERT ... ON CONFLICT DO NOTHING RETURNING id. The constraint is the real
    fix; the guard cannot be made safe without it. → FORK-REG-UNIQUE.

PATTERN B — multi-org singular/plural divergence (single-org-masked, latent)  [A1,A2,A5]
----------------------------------------------------------------------
A cluster of "correct today because LH is single-org, breaks on the 2nd org."
The platform is multi-tenant; these are first-cross-org-customer bugs.
  - sports_select uses current_user_org_id() (SINGULAR) vs the plural cohort. [A1 P1 / A5 P2]
  - guardians.email is GLOBALLY UNIQUE; submit_registration ON CONFLICT(email)
    lets an anon registrant in org B overwrite/relink org A's guardian PII. [A5 P1 F-1]
    → the one that MUST land before a 2nd org publishes a reg page.
  - public_slug is GLOBALLY UNIQUE but checkSlugAvailable pre-checks per-org →
    org B passes the pre-check for a slug org A owns, hits a raw global unique
    violation on INSERT. The code comment claiming "no DB constraint" is stale. [A2 P1]
→ FORK-SPORTS-PLURAL, FORK-GUARDIAN-UNIQUE, FORK-SLUG-SCOPE. All single-org-benign
  today; all should be resolved as a batch before multi-org onboarding.

PATTERN D — AP#36 destructured-default / error-swallow on count paths  [A3,A4,A6]
----------------------------------------------------------------------
  - useAllPrograms / useActivePrograms: PRIMARY query error → setPrograms([]) +
    console.error only; UI shows "No programs yet" empty state on a transient DB
    error, indistinguishable from a real empty org. Secondary teams/players

→ ship-this-phase app-layer fix (no migration): add error returns + distinct
  error states; gate the delete confirm on fresh counts.

----------------------------------------------------------------------
PATTERN E — single-active-season enforcement incompleteness  [A1,A3,A5 converge]
----------------------------------------------------------------------
Three independent agents flagged the same seam. The partial unique index
programs_one_active_season_per_org closes the >1-active race but NOT the 0-active
window: useSeasons.setActive / useProgramAdmin.activate do archive-then-activate
as TWO non-atomic statements. If stmt2 fails, the org is left with ZERO active
seasons; and the index can now make stmt2 throw 23505 on a race. Stale comments in
useSeasons.js (7-9, 55-58) still claim "Postgres doesn't enforce single-active."
→ FORK-ACTIVATE-ATOMIC: single activate() RPC / CASE update. Defer-able for LH
  (admin rarely toggles) but it's a real "no active season" failure mode.
