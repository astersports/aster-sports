--- PERF / DATA ---

[P1][AP#36] PERF/DATA-1 — useAllPrograms.js:21 & useActivePrograms.js:27 —
primary-query error is swallowed into an empty list with only console.error; UI
renders the "No programs yet" empty state on a transient DB error.
EVIDENCE: `if (error) { console.error(...); setPrograms([]); setLoading(false);
return; }` — no error state returned, so AdminProgramsPage (line 45) shows
EmptyState, indistinguishable from a real zero-programs org. Contrast
usePrograms.js (line 32-38) and useProgramRegistrations.js which DO expose error.
LEAN: add an `error` return to useAllPrograms (matches the canonical shape) and
have AdminProgramsPage render a retry/error state distinct from EmptyState.
Defensible as P2 if Frank-only surface, but the anti-pattern is real.

[P2] PERF/DATA-2 — useAllPrograms.js:24-44 — index does 3 round-trips
(programs → teams → team_players) sequentially (each awaits the prior because
ids feed the next .in()). Correct shape (can't .in() before you have ids), but
at growth (many programs) the team_players fetch pulls EVERY roster row for the
org to count in JS. At 95 rows today this is nothing; flag for the
multi-program-scale conversation — a count-per-program via a grouped view/RPC
would avoid shipping the full roster to the client.
LEAN: leave as-is now (YAGNI, AP#39 — the dedicated-RPC alternative is the truer
position only once data grows); note in the growth backlog.

[P1][AP#63] PERF/DATA-3 — useAllPrograms.js:38-44 vs useProgramHealthMetrics.js:
71-89 — "players in a program/season" computed two ways. useAllPrograms counts
EVERY team_players row (no status filter, no distinct) → a kid on two teams in
one program counts TWICE. useProgramHealthMetrics filters status='active' AND
dedups via Set(player_id). Same real-world concept, divergent math.
EVIDENCE: useAllPrograms line 43-44 `playerCount[pid] = (playerCount[pid]||0)+1`
per row; health line 89 `new Set(...).size`.
LEAN: align the index row to distinct-active to match the health card (the
health definition is the truer "players" count). PATTERN A detail deferred to A4
per instructions; A6 flags the invariant gap (XSURF-1).

[P3] PERF/VIRT — AdminProgramsPage.jsx:51 — the index renders one
<ProgramIndexRow> per program with no virtualization. Live count = 6 programs;
§16.10 virtualization threshold is >30 items. Not needed now. Reason about
growth: a multi-org / many-season future could exceed 30 grouped rows — revisit
when a single org's program count crosses ~30. No action this phase.

[P2] A11Y-3 — AdminProgramsPage.jsx:36-40 — filter chips are <button>s with
