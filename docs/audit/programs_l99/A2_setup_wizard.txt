--- [P1][AP#36 / kindness §16.3] useProgramSetup.js:38 — single-active-season
    unique violation surfaces a RAW Postgres error string to the admin.
    The create writes status from statusForProgramType(); for program_type='season'
    that's 'draft' (registry season.defaultStatus='draft'), so a normal season
    create does NOT hit programs_one_active_season_per_org (index is WHERE
    status='active'). GOOD — draft-on-create deliberately dodges the collision at
    create time (confirmed: registry comment lines 34-37 + programSetup.js:11-16).
    BUT: the create path still does `if (e1) { setError(e1.message); ... }` — it
    forwards the raw DB `error.message` verbatim. If ANY insert error fires
    (the slug global-unique index below, a future activate-time path that routes
    through here, an RLS denial, a NOT NULL miss), the admin sees a Postgres
    string like "duplicate key value violates unique constraint
    programs_public_slug_key" or "new row violates row-level security policy".
    That is the §16.3 anti-pattern ("Error 403" / "Operation failed" raw).
    Evidence: useProgramSetup.js:38 (e1.message), :51 (e2.message), :60 (e3.message).
    LEAN: map known constraint names (programs_public_slug_key, RLS '42501',
    programs_one_active_season_per_org) to kindness copy before setError; raw
    message only as last-resort fallback. The slug path is partially guarded by
    checkSlugAvailable (pre-insert) but that guard is per-org while the DB index
    is GLOBAL — see next finding — so a raw collision IS reachable.

--- [P1][AP#60 / data-integrity] programSetup.js:31-38 + DB —
    SLUG UNIQUENESS SCOPE MISMATCH. checkSlugAvailable() filters
    `.eq('org_id', orgId).ilike('public_slug', slug)` → per-org check. But the
    enforcing DB index programs_public_slug_key is UNIQUE on (public_slug) GLOBALLY
    (cross-org), WHERE public_slug IS NOT NULL. So: Org A already has slug "spring-2026";
    Org B's admin types/derives "spring-2026" → checkSlugAvailable queries only Org B's
    rows, returns null (free), the INSERT then RAISES a global unique violation, and
    the admin gets the raw Postgres string (finding above). The pre-check's own comment
    (programSetup.js:26 "App-level public_slug uniqueness on (org_id, slug) — the DB
    has no unique constraint") is STALE/WRONG: the DB DOES have a unique constraint,
    and it's cross-org, not (org_id, slug). This is a live correctness + first-cross-org-
    customer bug (the platform is multi-tenant; today single-org Legacy Hoopers masks it).
    Evidence: live pg_indexes programs_public_slug_key def vs checkSlugAvailable .eq org filter.
    LEAN: either (1) make the pre-check global (drop the org_id filter) to match the DB,
    OR (2) change the DB index to UNIQUE(org_id, public_slug) so slugs are org-scoped
    (the /r/<slug> public route resolution must then also be org-aware). (1) is the
    smaller, ship-now fix and matches the DB's current contract; flag (2) to architect
    as the truer design if public links are meant to be per-org. Commit-to-lean: (1) now.
