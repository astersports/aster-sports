[P2][multi-org divergence] live RLS: sports_select vs programs_select
  sports_select USING `org_id = current_user_org_id()` (SINGULAR — LIMIT 1, picks
  ONE arbitrary org row for the user). programs_select / divisions_select /
  division_fees_select / registrations_select all USING `org_id = ANY
  (current_user_org_ids())` (PLURAL — ARRAY_AGG of all the user's orgs). Verified
  both helper defs live: current_user_org_id = `select organization_id ... limit 1`;
  current_user_org_ids = `ARRAY_AGG(DISTINCT organization_id)`.

  payload), so the ANON surface is unaffected — this is an authenticated multi-org
  consistency issue, not an anon leak.

  siblings. Cheap, no behavior change for single-org. (See fork FK-2.)

[P2][atomicity] src/hooks/useSeasons.js:59-69  (F-2)
  setActive() is NOT atomic and NOT transactional: stmt1 archives all-other
  seasons in the org (`update status=archived ... neq id`), stmt2 activates the
  target. If stmt2 fails (network, or the partial-unique index rejects because of
  a race with a concurrent admin), the org is left with ZERO active seasons — and
  SeasonContext.jsx:53-54 + useActiveSeasonEnd.js:11 silently fall back
  (rows[0] / null). Now that the partial-unique index exists (P3 above), stmt2
  can also throw 23505 if any other 'active' season slips in between the two
  statements. The whole "two statements because Postgres can't enforce one active"
  rationale is obsolete; the index would let a single `UPDATE ... SET status=CASE
  WHEN id=$1 THEN 'active' ELSE 'archived' END` run atomically (or an RPC).
  EVIDENCE: useSeasons.js:61-69 (no txn wrapper, sequential awaits with early
  return between them).
  LEAN: wrap in a SECURITY-INVOKER RPC or single CASE update so the activate is
  atomic with the archive. Multi-org safe (index is per-org). Defer-able for LH
  (admin rarely toggles), but it's a real "leaves org with no active season"
  failure mode.
