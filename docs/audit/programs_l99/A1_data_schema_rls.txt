[P0][AP#26 / AP#25-adjacent] submit_registration TOCTOU race — registration_public_rpcs.sql
  lines 150-167. The dedupe guard `SELECT id INTO v_reg_id FROM registrations WHERE
  program_id=v_program_id AND player_id=v_player_id` then `IF v_reg_id IS NOT NULL ... CONTINUE`
  else INSERT — is a read-modify-write with NO row lock and NO unique constraint backing it
  (see (a) P1). Anon-executable (anon_exec=true verified). A double-submit (parent
  double-taps "Register", or two devices) races: both transactions see no existing row, both
  INSERT a registration + a full set of registration_fees -> family billed twice for one
  child. SECURITY DEFINER runs each call in its own transaction; READ COMMITTED isolation
  does not serialize these. — LEAN(P0): add UNIQUE(program_id,player_id) on registrations AND
  change the guard to `INSERT ... ON CONFLICT (program_id,player_id) DO NOTHING RETURNING id`
  (then treat null-return as "already registered"). The constraint is the real fix; the guard
  alone cannot be made safe without it. This is the single highest-value finding in A1.

[P1][AP#42 / seam-coupling] assert_org_owns_season SECDEF helper reads the seasons VIEW,
  not programs. Evidence: pg_get_functiondef — `SELECT EXISTS(SELECT 1 FROM seasons WHERE
  id=p_season_id AND org_id=p_org_id)`. It is STABLE SECURITY DEFINER search_path=public,
  anon_exec=false (good). Functionally correct because the view = programs WHERE
  program_type='season', so the helper silently scopes to seasons only. Latent: if a caller
  ever passes a camp/tryout program id expecting "org owns this program", the helper returns
  false (it's season-only by construction). The name implies season-only so this is arguably
  intended, but the program-graph generalization (camps/clinics) makes a same-named
  assert_org_owns_program desirable. — REC: leave as-is; note the season-only scope when the
  registry generalizes ownership checks to non-season program types.

[P1][stale-comment / data-integrity drift] useSeasons.js createSeason + setActive comments
  are STALE post-migration 20260607162813. Lines 7-9 + 55-58 assert "Postgres doesn't enforce
  only one active per org" and "the check constraint already guarantees valid status." Both
  are now wrong: the partial unique index `programs_one_active_season_per_org`
  (ON programs(org_id) WHERE status='active' AND program_type='season', verified live) DOES
  enforce single-active at the DB. Two consequences:
    (i) createSeason (useSeasons.js:43) inserts through the seasons view with no status =>
        programs.status DEFAULT 'active' applies => if an active season already exists, the
        INSERT fails with a unique_violation (23505). The error surfaces raw via
        `return { error: e?.message }` with NO kindness microcopy (AP#16.3 violation —
        "duplicate key value violates unique constraint programs_one_active_season_per_org"
        shown to admin). — LEAN: createSeason should default new seasons to status='draft', OR
        catch 23505 and render a kindness message ("There's already an active season —
        archive it first.").
    (ii) setActive (useSeasons.js:61-66) archives-others-then-activates. Order is safe for the
        index (the conflicting active row is archived in statement 1 before statement 2
        promotes the target). But it is NON-ATOMIC (AP per migration comment): if statement 1
        succeeds and statement 2 fails/network-drops, the org is left with ZERO active seasons.
        The unique index prevents >1 active but does not prevent 0 active. — REC: wrap in a
        single RPC (the migration 20260607162813 comment anticipates "the unified activate()").
        The index closes the >1 race; it does not close the 0-active window.

--- WITH CHECK OPTION WRITE PATH: ESCAPE ANALYSIS (the highest-risk question) ---
seasons view def (verified pg_get_viewdef): `SELECT id,org_id,name,start_date,end_date,
status,created_at,updated_at,parent_program_id AS parent_season_id,rolled_over_at FROM
programs WHERE program_type='season' WITH CHECK OPTION` (check_option=cascaded, verified).
KEY RESULT — CANNOT ESCAPE THE VIEW:
  * program_type is NOT a projected column of the view. A write through the view physically
    cannot set program_type (PostgREST/SQL can only target the 10 projected columns). So
    useSeasonRollover.js:16 INSERT and useSeasons.js:43/50/62/66 UPDATE/INSERT cannot flip a
    row to program_type='camp' to "escape" — the column is unreachable.
  * On INSERT through the view, programs.program_type DEFAULT 'season' supplies the value;
    WITH CHECK OPTION (cascaded) then re-validates the inserted row satisfies the view qual
    (program_type='season') — it does, by default. Safe.
  * On UPDATE through the view, the row's program_type is untouched (not in the update target)
    and WITH CHECK OPTION re-validates it still satisfies the qual — it does. Safe.
  * sport_id defaults NULL on view INSERT (not projected) — seasons created via rollover have
    no sport. Acceptable for LH single-sport; note for multi-sport orgs (rollover-created
    seasons will have NULL sport_id).
  VERDICT: the compat-view seam is SOUND. No write can escape program_type='season'. This was
  the kickoff's flagged "highest-risk seam" — it is correctly fenced. (No P0/P1 here.)

GAPS / NOTES:
- registrations has NO parent SELECT policy. A parent CANNOT see their own child's
  registration row via PostgREST (only admin can). If the parent-facing registration-status
  surface needs to read registrations, it will silently return [] for parents (AP#36 class if
  a destructured default hides it). Flag for A2/A4 (hooks/UI): confirm whether any
  parent-facing screen queries `registrations` directly — it will be empty for them.
  (A1 verdict: this is a likely PRODUCT gap, not necessarily a bug — registration confirmation
  may be intended admin-only. Surface to architect.)
- tryout_sessions/tryout_attendees are parent/coach SELECT-able org-wide (org_id = ANY
  current_user_org_ids()) — a parent can see ALL tryout attendees in their org incl
  evaluation_note (tryout_attendees.evaluation_note is coach eval text). [P1][AP#16.7 privacy]
  evaluation_note is coach-private eval content but the SELECT policy exposes it to any

  current_user_org_ids()) with no role narrowing. — LEAN: narrow tryout_attendees SELECT (and
