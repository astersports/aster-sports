(c) ANTI-PATTERN CROSS-REFERENCE + PATTERN A #63
--------------------------------------------------------------------------------
AP#36 (destructured-default / bare-await swallow): P0-2, P1-4, DR-4 — the
   roster/archive/coach writes are bare `await supabase...` with no { error }
   check. PostgREST resolves (doesn't reject) on error, so failures are
   invisible; loop counters increment regardless. This is the load-bearing
   reason P0-2 is silent rather than loud.
AP#14 (Futures Academy as roster_type, NOT a separate concern): P0-2 — the
   code reaches for roster_type on the wrong table; roster_type is a
   team_players column, never roster_members.
AP#20 / atomicity doctrine: the whole execute() is the multi-write hazard the
   codebase elsewhere guards with SECURITY DEFINER RPCs (e.g. assert_org_owns_*
   AP#57). Rollover has no server-side transactional wrapper.
AP#16.1 (bulk ops pessimistic): the rollover is bulk + cascading-admin-write
   (the §16.1 "stay pessimistic" class) but ships neither a txn nor a
   validate-before-commit; the ConfirmDialog (page:80) is the only "confirm"
   and it gates intent, not data validity.

PATTERN A #63 (same concept, divergent source/scope across surfaces):
   "Does this family owe money / carry a balance across the season boundary?"
   - financial_accounts: per-season, NOT NULL season_id, no carry (P2-7).
   - roster_members.amount_due/paid: per-(player,team)=per-season, not carried
     (DR-6).
   - family_balances view (§11.5): all-seasons aggregate — would still show
     the old-season balance after rollover because old accounts persist.
   So post-rollover: the all-seasons family_balances view shows the old debt,
   but the NEW season's financial_accounts shows zero → a parent's "balance"
   reads differently depending on whether a surface is season-scoped or
   all-seasons-scoped. Classic #63 divergence, latent until a rollover ever
   succeeds. LABEL-the-scope discipline (per #63) applies the day this ships.

ARCHITECTURAL FORKS (AP#39 — committed lean, architect/Frank ratifies)
--------------------------------------------------------------------------------
F1  NEW-SEASON STATUS VALUE (P0-1 fix)
    Options: (1) 'draft' — matches the constraint, signals "being set up";
    admin later activates via setActive. (2) 'archived' — both seasons
    archived, no active until explicit activate. (3) widen constraint to add
    'planning' (schema change).
    LEAN: (1) 'draft'. It satisfies the existing constraint with no migration,
    matches the registration-arc 'draft' program semantics, and keeps the
    new season visible-but-inactive until the admin deliberately activates.
    'planning' was a value the constraint never knew about — the bug is the
    code, not the schema.

F2  ATOMICITY MODEL (P0-3 fix)
    Options: (A) wrap the entire rollover in a single SECURITY DEFINER
    plpgsql RPC (rollover_season(plan jsonb)) that runs in one transaction —
    all-or-nothing, audit row written in-txn, partial failure rolls back
    cleanly. (B) keep JS-sequential but add a compensating-delete on catch +
    write the audit row FIRST as in_progress and flip to complete.
    LEAN: (A) single RPC. This is the §16.1 cascading-admin-write class; the
    codebase already standardizes server-side transactional helpers (AP#57
    assert_org_owns_*). An RPC also fixes AP#36 swallow (errors raise in-txn),
    fixes the 0-active window if it also activates atomically, and gives a
    real idempotency guard (unique on from_season_id). (B) is the hedge — and
    per AP#39 the dedicated RPC is the truer position; do (A).

F3  FINANCIAL CONTINUITY INTENT (P2-7 / DR-6)
    Question for product: is each season billed fresh (clean slate, no carry)
    or should outstanding balances/credits carry into the new season's
    financial_accounts? The field financial_balances_carried_cents implies
    carry was designed.
    LEAN: clean-slate-per-season is the simpler, likely-correct youth-sports
    model (Fall/Winter/Spring each bill independently). If so, DELETE the
    financial_balances_carried_cents field (or keep at 0 and document
    "no carry by design") and LABEL family-balance surfaces by scope per
    PATTERN A #63. But this is a genuine product fork — surface it, do not
    settle it in code.

F4  NEW-SEASON ACTIVATION ON ROLLOVER (P1-5)
    Should rollover leave the new season inactive (admin activates later) or
    activate-on-commit (archive old + activate new atomically)?
    LEAN: activate-on-commit, inside the F2 RPC, so the partial-index single-
    active invariant is enforced transactionally (archive old + insert new
    active in one txn — the partial index permits it because the old flips to
    archived in the same statement-set). This also kills the post-rollover
    "0 active, SeasonContext picks rows[0] fallback" ambiguity.
