LEAK PATH FOUND (structural, P2, latent today) — PATTERN A #63:
  The standings TABLE excludes non-competitive teams, but the org-wide GAME
  COUNT aggregates do NOT:
    * RecordsPage.jsx:41-45 — totalGames query counts ALL published
      game_results for the org (events!inner->teams!inner->org_id), with NO
      team_type filter. Rendered as "{totalGames} Games" (line 68) and feeds
      no per-team gating.
    * GamesView.jsx:71-75 — totalGames sums gamesPlayed across recordsByTeamId,
      which comes from useOrgTeamRecords (un-filtered, see below). But note:
      GamesView passes `allTeams` (competitive-filtered) to StandingsTable while
      totalGames sums the UN-filtered recordsByTeamId — same split.
    * useOrgTeamRecords.js:29-33 — fetches every published game_result for the
      org's teams with NO competitive filter; returns byTeamId for ALL teams.
      RecordsPage/GamesView only LOOK UP competitive teams in that map, so the
      table is safe, but any AGGREGATE over the whole map leaks.
  CONSEQUENCE: the day a clinic_camp/training_only team gets a published
  game_result, "{N} Games" / "Active Teams" counter will include it while the
  team itself is absent from the standings list — a same-surface contradiction
  (team count says "excluded", game count says "included"). Classic PATTERN A:
  one concept ("how many games this org played competitively"), two scopes.
  FIX LEAN (AP#39): filter useOrgTeamRecords at the query/group layer by the
  same competitive slug set, OR have RecordsPage/GamesView restrict the
  totalGames aggregate to the competitive-team id set. Prefer filtering in
  useOrgTeamRecords so every consumer inherits the correct scope (single
  source) — the embed cost is one more join on team_types(slug). This is the
  truer position (the hedge "filter at the surface" duplicates the rule across
  2 surfaces and re-opens drift). [FORK B8-1, see below]

SCOPE 3 — INDEX COLUMN SET vs LeagueApps REFERENCE (REDESIGN INPUT)
--------------------------------------------------------------------------------
CURRENT INDEX ROW (ProgramIndexRow.jsx:13-18): name + badge, then meta line =
  range · "{N} teams" · "{N} players". No financial columns.

LeagueApps reference (architect-supplied): Activity Start · Invoiced · Paid ·
  Outstanding · Players · Staff, grouped Live/Upcoming/Unpublished/Completed.

GAP: we show teams+players; we DON'T show Invoiced/Paid/Outstanding or Staff.
For an admin running the org's money, per-program financial roll-up at the index
level is the single highest-value missing column set.

FEASIBILITY (live): financial_accounts is season_id-scoped, and season_id == the
program id for season-type programs (financial_accounts cols: season_id,
season_fee_cents, discount_cents...). So Invoiced/Paid/Outstanding per program
IS computable today by joining financial_accounts + financial_transactions on
season_id = program.id. The family_balances view already nets payment/refund/
adjustment/fee per the §11.5 ground-truth (use it as the single source, NOT a
re-derivation — PATTERN A #63 / HOME-4 precedent).

LIVE: financial_accounts by program type:
  season => 164 accounts across 3 seasons. camp/tryout/clinic => 0 accounts.
=> financial columns are MEANINGFUL ONLY for season-type programs today. A
   tryout/camp/clinic/evaluation/interest_list has no invoiced/paid concept
   (flat-fee camps could later, but no financial_accounts wired today).

LEAN (AP#39) — per-type-conditional column set, NOT a uniform financial row:
  * season: range · teams · players · Invoiced · Paid · Outstanding
            (Outstanding is the action signal — net owed from family_balances)
  * camp/clinic: range · players · (registrations) — flat-fee, no per-season
            invoice ledger today; show registration count not "teams"
  * tryout/evaluation/interest_list: range · registrations/signups — no teams,
            no money; "X players" is misleading (these have no roster yet)
  * other: range · whatever is populated
  The truer position (hedge-check): do NOT bolt a uniform Invoiced/Paid/
  Outstanding onto every row — for 4 of 7 types it renders $0/blank and reads
  as "this program collected nothing" rather than "this program has no fee
  ledger". The registry already declares per-type semantics (noun, hasDivisions,
  competitive); ADD a per-type `indexColumns` (or `showsFinancials` flag) to
  PROGRAM_TYPE_REGISTRY so the row renders the right columns per type from one
  source. [FORK B8-2 — architect decision: which columns per type]

  Scope label REQUIREMENT (PATTERN A #63): if both index "players" and detail
  "Registrations" survive, LABEL them distinctly (rostered vs registered) —
  they are different concepts and currently both read as "people in program".
