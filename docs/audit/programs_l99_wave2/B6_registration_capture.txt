(b) DEEP-READ ADDENDUM — what a single pass misses
================================================================================
DR-1  The audit doc's central security claim is FALSE in live state. AUDIT_
      REGISTRATION_CAPTURE_L99.md §2 G1 + the RPC migration header (lines 2-4)
      both assert "anon gets EXECUTE on exactly [these] functions, zero table
      grants." LIVE: anon holds SELECT,INSERT,UPDATE,DELETE,TRUNCATE,REFERENCES,
      TRIGGER on ALL 10 graph tables (registrations, registration_fees, players,
      guardians, player_guardians, player_equipment, divisions, division_fees,
      tryout_sessions, tryout_attendees). Source: migrations 6/7 + the divisions/
      tryouts migrations each end with `GRANT ... TO anon, authenticated,
      service_role`. The "zero table grants" story is aspirational, not real.
DR-2  RLS *currently* saves the day for INSERT/UPDATE/DELETE: every write policy's
      check_expr requires user_has_role_in_org(...,'admin') which anon fails. So
      anon cannot INSERT/UPDATE/DELETE through PostgREST despite the grant. BUT:
      (i) TRUNCATE is NOT subject to RLS at all — the anon TRUNCATE grant is a
      latent data-loss primitive the moment any path can issue it; (ii) the grant
      is the only thing standing if any future RLS policy ships permissive/
      fail-open (the platform's dominant historical bug class — see §0 grep #20
      WITH CHECK, AP #20). Defense-in-depth is inverted here.
DR-3  conduct_acknowledged_at is NEVER written. The column exists on registrations
      (line 36 of mig 6) but the RPC INSERT column list (lines 156-159) omits it,
      and the client payload (RegisterFlowPage handleSubmit, lines 67-72) has no
      conduct/waiver field. Every registration lands with conduct_acknowledged_at
      = NULL. No code path ever sets it.
DR-4  Player gender is never captured and never validated against division.gender.
      divisions carry an M/F CHECK; players has no gender in the payload; the RPC
      grade-band check (lines 121-124) checks grade only. A player can be written
      into a gender-mismatched division with zero server objection.
DR-5  early_bird / prorated / discount fee types are dead in the capture path. The
      RPC only ever copies fee_type IN ('base','add_on') (lines 169-171) and emits
      family_discount (line 192). A division with an early_bird template would have
      that fee SILENTLY DROPPED at registration — the wave-1 "unnamed-division fee
      silently dropped" flag generalizes: any non-base/add_on template is dropped.
DR-6  authoritative_total reconciliation contract is unbuilt. Audit §2 G2 promised
      "client estimate and server total must reconcile — if they differ, server
      wins and the UI re-renders." RegisterConfirm.jsx never reads
      authoritative_total_cents; StepReview shows the CLIENT estimate only. The
      server value is returned and discarded. (Dormant today because LH policy is
      null + fees come from the same division_fees the client fetched, so the two
      are equal — but the divergence guard PATTERN A #63 demanded does not exist.)

GOOD (verified correct):
+ org_id is SERVER-DERIVED from programs.org_id (lines 75-80), never read from
  client payload. No org spoof. (Closes a class wave-1 worried about.)
+ Program gate: published + inside reg window (lines 76-79); else
  'registration_closed'. Correct.
+ Division-belongs-to-program: WHERE program_id = v_program_id (line 117); else
  'invalid_division'. Correct — a foreign division id is rejected.
+ Player dedupe scoped to org (lines 127-132): org_id + lower(name) + dob. Org-
  scoped (good — not global like guardian email).
+ player_guardians ON CONFLICT(player_id,guardian_id) DO NOTHING (lines 142,146):

+ Fee total summed SERVER-SIDE from registration_fees after insert (line 172),
  returned as authoritative_total_cents. The server is the source of "what's owed."
+ Children array bounds: array-typed + length 1..10 (lines 83-85). Good input guard.

FINDINGS:
P0-1  TOCTOU double-register / no UNIQUE(program_id, player_id).  [EXTENDS wave-1]
      RPC lines 150-154 do SELECT ... FROM registrations WHERE program_id=? AND
      player_id=? then conditionally INSERT. There is NO unique constraint backing
      this (verified: pg_constraint has none). Two concurrent submits for the same
      child both pass the SELECT (each sees zero rows) and both INSERT → duplicate
      registrations + duplicate fee rows + double billing. The dedupe is advisory,
      not enforced. Live evidence: constraint query returned only the gender/
      member_type/email/tryout uniques — no registrations composite unique.
      file:line submit_registration:150-154.
      FIX LEAN (AP#39): add UNIQUE(program_id, player_id) on registrations + change
      the body to INSERT ... ON CONFLICT (program_id,player_id) DO NOTHING RETURNING,
      treating "no row returned" as already_registered. One migration, removes the
      race. (The truer fix; the read-then-write guard is the hedge — drop it.)

P0-2  anon TRUNCATE/DELETE/UPDATE grants on the entire registration graph.  [NEW]
      See DR-1/DR-2. anon should hold ZERO table grants; the only anon surface is
      the two SECDEF RPCs. file: migrations 6/7 trailer GRANT lines + divisions/
      division_fees/tryouts migration trailers.
      FIX LEAN (AP#39): REVOKE ALL ON <10 tables> FROM anon (+ PUBLIC per AP#23/#57),
      keep only GRANT EXECUTE on the two RPCs. The RPC is SECDEF and runs as owner,
      so revoking anon table grants does NOT break the flow. This is the truer
      contract the audit *claimed* shipped; the live grant is the drift.

P1-1  conduct_acknowledged_at never set + no waiver capture.  [NEW — liability]
      See §2 below. A registration with no signed waiver / no conduct acceptance is
      a youth-sports liability gap. file: submit_registration:156-159 (omitted col)
      + RegisterFlowPage:67-72 (payload omits it) + StepDetails (no checkbox).

P1-2  Non-base/add_on fee templates silently dropped.  [EXTENDS wave-1 fee flag]
      DR-5. early_bird/prorated/discount division_fees never copied to
      registration_fees. If an org configures an early-bird discount, the registrant
      is billed full freight with no error. file: submit_registration:169-171.
      FIX LEAN: either (a) widen the copy to all non-family_discount fee_types with
      explicit auto_apply_rule evaluation (division_fees.auto_apply_rule jsonb exists
      live but is unread), or (b) if early_bird/prorated are genuinely out of MVP
      scope, add a CHECK or a comment + reject divisions carrying them. Lean: (a) —
      the auto_apply_rule column was built for exactly this and is dead.

P1-3  Player gender not captured / not validated vs division.gender.  [NEW]
      DR-4. file: submit_registration:121-124 (grade-only band check),
      RegisterFlowPage emptyPlayer (no gender). FIX LEAN: capture player gender in
      Step 1, add server check `IF v_div.gender IS NOT NULL AND player.gender <>
      v_div.gender THEN RAISE 'gender_mismatch'`. (Hedge: skip if LH divisions are
      single-gender-by-grade-anyway — but the M/F CHECK exists for a reason; the
      validation is the truer position per AP#39.)

P2-1  guardian dedupe by GLOBAL email (no org scope).  [EXTENDS wave-1]
      Line 91 ON CONFLICT(email) — guardians_email_unique is global (verified).
      In single-org LH this is benign, but cross-org (St Pat's 2027) a guardian
      registering in two orgs collides → the second org's INSERT routes to UPDATE
      on the FIRST org's guardian row (org_id is set only on INSERT, not updated),
      so the guardian stays attached to the wrong org_id while player_guardians
      links across orgs. file: submit_registration:88-96 + guardians_email_unique.
      FIX LEAN: make the uniqueness (org_id, lower(email)) before multi-org; the
      RPC already has v_org_id. Defer is acceptable ONLY while org_count=1.

P2-2  registration_tier hardcoded 'full_roster'; waitlist_state hardcoded 'none';
      status hardcoded 'pending'.  [NEW — observation]
      Lines 156-160. The RPC cannot produce a practice_roster/practice_player
      registration nor a waitlist row — the tier/waitlist enums (built in mig 6)
      are unreachable from the only write path. Not a bug for the season-MVP, but
      the tier selection promised by registration_tier enum has no capture surface.
      file: submit_registration:160.

P2-3  family_discount attached to v_reg_ids[1] only (lines 191-192). The whole
      family discount lands as a single negative line on the FIRST child's
      registration_fees. If that child's registration is later cancelled/deleted
      (CASCADE), the discount row vanishes and the OTHER children's net owed
      silently increases. file: submit_registration:191-192. Dormant (policy null).

P3-1  jsonb_array_length>10 → 'too_many_children' (line 85) is a raw error code with
      no kindness-microcopy mapping client-side; same for grade_below_band etc. The
      client (StepReview:21) collapses ALL errors to "Looks like that didn't go
      through." So grade/gender/closed all read identically — no actionable message
      (§16.3 violation). file: StepReview:21.
