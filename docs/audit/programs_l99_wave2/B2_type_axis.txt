FINDING D-1 (P2) — UI render gate bypasses single-source divisions rule
  File: src/pages/admin/ProgramSetupPage.jsx:54
    {form.program_type === 'season' && ( <Divisions/> )}
  The Divisions UI section is gated on a HARDCODED `program_type === 'season'`
  rather than the existing helper divisionsApplyTo(type) (== programRule(type)
  .hasDivisions). divisionsApplyTo is defined in programSetup.js:21-23 and is
  ALREADY imported + re-exported by the page's own hook (useProgramSetup.js:4,8)
  but the page does not import or use it.
  Mitigation (downgrades from P1→P2): the WRITE path is correct —
  useProgramSetup.js:40 uses `divisionsApplyTo(programType)` to decide whether
  to persist divisions. So data integrity is fine TODAY. Only the UI render
  could diverge from the data rule if a future archetype is given
  hasDivisions:true — the section would silently not render while the write
  path still expects divisions. This is exactly the single-source-bypass class
  the registry was built to eliminate (programRegistry.js:1-6 names this page's
  switches as the thing it replaced).
  Fix (1 line): import divisionsApplyTo and replace the gate with
    {divisionsApplyTo(form.program_type) && ( ... )}

FINDING D-2 (P3) — embed-presence is load-bearing but only test-locked, not type-locked
  isCompetitiveTeam(team) reads team.team_types.slug; a team object WITHOUT the
  team_types(slug) embed yields slug=undefined → returns TRUE (competitive
  default) → phantom standings inclusion. All 3 competitive-filter surfaces DO
  embed the slug today (verified): useTeams.js:24, useProgramTeams.js:19,
  TeamMultiSelect.jsx:17. But the dependency is implicit — a new surface that
  filters by isCompetitiveTeam off a teams query that forgets the embed would
  silently include every team. The teamTypes.test.js cases (lines 27-29) lock
  the DEFAULT-true behavior intentionally (no-regression), so a missing embed
  would NOT fail any test. P3 because no live instance; it is a latent
  foot-gun in the "NULL slug → competitive" design choice.

No other conflation or parallel-rule sites found on the second pass. The
program_type === 'season' check at TeamDetailPage.jsx:66/138 and
pulseRange 'season' are unrelated (a UI date-range toggle literal, not the
program-type enum). PlayerProfilePage academy badge reads roster_type, not
program/team type (out of this slice's axis).

FORKS (AP#39 leans)
================================================================================
FORK F-1 — Should "NULL team_type → competitive default" be inverted to fail-closed?
  Context: isCompetitiveTeam returns TRUE for a team with no joined type
  (teamTypes.js:22-24). Today 0 teams are NULL (F1 NULL-guard in TeamFormSheet
  prevents new NULLs), so it never fires. The design choice was no-regression
  for legacy pre-team_types rows.
  Option A (keep TRUE default): no-regression preserved; matches current tests
    (teamTypes.test.js:27-29 lock TRUE). Phantom risk only if a future surface
    filters without embedding the slug (D-2).
  Option B (flip to FALSE / fail-closed): a team that loses its type embed gets
    EXCLUDED from standings rather than phantom-INCLUDED with a meaningless 0-0.
    Excluding a real game team from standings is more visible (coach notices a
    missing team) than silently including a camp team — arguably the safer
    failure for a youth-sports standings surface.
  LEAN: Option A for now (zero live NULLs, no-regression intent is sound, and
  the real fix for D-2 is an embed-presence lint/test, not flipping the default).
  Per AP#39 the hedge (Option B) is worth surfacing: if the architect wants
  fail-closed semantics platform-wide, B is defensible — but I commit to A
  pending a concrete NULL-team scenario. Decision belongs to architect/Frank.
