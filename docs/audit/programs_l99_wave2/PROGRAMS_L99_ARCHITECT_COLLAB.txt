==============================================================================

R-1 [architect grounding §1 — WRONG] "No WITH CHECK OPTION in the [seasons] def."
  LIVE TRUTH: the seasons view HAS check_option=cascaded. Evidence:
  information_schema.views.check_option='CASCADED' for seasons; pg_class.reloptions
  = {security_invoker=true, check_option=cascaded}. CAUSE OF THE ERROR: pg_get_viewdef()
  does NOT render the WITH CHECK OPTION clause (it is stored as a reloption, not in the
  SELECT text) — inspecting only the viewdef hides it. CONSEQUENCE: the architect's
  "sharpest edge" (a filter-escaping write) is DOUBLY blocked — (a) program_type is not
  a projected column so no write can target it (INSERT falls to DEFAULT 'season'), AND
  (b) cascaded re-validates every written row against program_type='season'. B1 proved
  no src/ write path defeats either guard. SEAM VERDICT: SOUND. (A1+A5+B1 concur.)

R-2 [prior closeout #788 "VERDICT: no P0/P1 bugs; cascades verified" — REFUTED]
  Both P0s reproduce on independent live replay (B4, B5). ROOT CAUSE of the original
  miss: the closeout was a code-reading + happy-path audit — it trusted in-code comments
  asserting the cascade was "clean" instead of querying pg_constraint (an AP#60 source-
  hierarchy inversion) and never modeled the anon submit_registration RPC under
  concurrency (AP#26 not applied). The two P0s ARE the delta between a code-read and a
  live+concurrency audit. Genuine misses, not new framing. → the "no P0/P1" verdict must
  be RETRACTED before programs can close.

R-3 [CLAUDE.md §8 "7-C / 7-B.2 rollover ✅ DONE — atomic execution" — FALSE]
  Season rollover is NON-ATOMIC and currently NON-FUNCTIONAL. useSeasonRollover.js is
  ~5+N sequential uncoordinated awaits, no RPC/txn/rollback, AND it throws on the first
  statement (see P0-3). LIVE CORROBORATION: season_rollovers table = 0 rows — no rollover
  has EVER completed in production. (B1+B7.)

R-4 [CLAUDE.md §5 migration-003 lists waivers/waiver_signatures/form_fields/
  form_responses — STALE] None of these tables exist (verified to_regclass = NULL for
  all four). The public registration flow captures NO waiver, NO conduct acknowledgment,

  (never written). A parent can fully register a child having signed nothing. (B6.)

R-5 [wave-1 A5 "anon table grants are Supabase-default, RLS gates them, not a leak" —
  PARTIALLY WRONG] anon holds GRANT ALL — including TRUNCATE, REFERENCES, TRIGGER — on
  every program-graph + financial + guardian/player table (verified:
  has_table_privilege('anon','financial_transactions','TRUNCATE')=true, and the full
  role_table_grants dump). This is NOT the standard Supabase default (SELECT/INSERT/
  UPDATE/DELETE). TRUNCATE is NOT RLS-gated. Practical API exploitability is bounded
  (PostgREST exposes no TRUNCATE endpoint; DELETE is RLS-gated to zero anon rows), so CC
  grades it P1 / latent-P0 defense-in-depth — but on the financial ledger it is a real
  hardening gap and the platform's dominant bug class is fail-open RLS. (B6 + CC live verify.)

R-6 [closeout deferral E3 "filter chips omit evaluation+interest_list" — STALE/FIXED]
