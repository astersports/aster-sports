THE ONE ARCHITECT-STOP — CO-GUARDIAN MONEY-READ RLS WIDENING (needs your GO)
------------------------------------------------------------------------------
WHY IT'S A STOP: B1's balance must resolve via the CHILD (your note), so a
  co-guardian sees their child's program balance even though the financial_account
  is keyed on the OTHER (registering) guardian. The live policy only lets the
  account-holding guardian read it:
     fin_accounts_parent_read  USING (guardian_id IN (my guardian))
     fin_tx_parent_read        USING (account_id IN (accounts of my guardian))
  So a co-guardian reads 0 balance for their own child. Closing that means ADDING a
  read path = WIDENING money visibility. Widening money RLS is security-sensitive
  (the opposite direction from PR-0b's narrowing + F-RLS-1's admin-only money
  posture), so CC will NOT apply it without your explicit GO. This is the only
  thing between B1 and full correctness for the 2-guardian families (KHOJASTEH,
  DeMasi, etc.).

CC CANDIDATE (per your given predicate "a parent sees an account whose program
  enrolls one of their children") — for your ratification before CC applies:
    -- financial_accounts: add a co-parent read path via the child's enrollment.
    CREATE POLICY fin_accounts_parent_read_via_child ON public.financial_accounts
      FOR SELECT TO authenticated
      USING (season_id IN (
        SELECT t.season_id FROM team_players tp JOIN teams t ON t.id = tp.team_id
        WHERE tp.player_id = ANY (public.current_user_player_ids())));
    -- financial_transactions: mirror, via the account.
    CREATE POLICY fin_tx_parent_read_via_child ON public.financial_transactions
      FOR SELECT TO authenticated
      USING (account_id IN (
        SELECT fa.id FROM financial_accounts fa
        WHERE fa.season_id IN (
          SELECT t.season_id FROM team_players tp JOIN teams t ON t.id = tp.team_id
          WHERE tp.player_id = ANY (public.current_user_player_ids()))));
  SCOPE CHECK CC FLAGS for your ruling (the reason this is a STOP, not a
  CC-decides): this predicate is "any account for a PROGRAM my child is enrolled
  in." If a program has MULTIPLE families, an account is one family's — but the
  predicate keys on season_id (program), not on the account's family. So as
  written it could let a parent read OTHER families' accounts for the SAME program
  (every family in Spring 2026 shares the season_id). THAT IS A LEAK. The correct
  predicate must tie the account to the CHILD'S family, e.g. the account's
  guardian_id is a co-guardian of one of my children:
    USING (guardian_id IN (
      SELECT pg.guardian_id FROM player_guardians pg
      WHERE pg.player_id = ANY (public.current_user_player_ids())))
  CC LEAN: the guardian_id-via-player_guardians form (my children's guardians'
  accounts), NOT the season_id form (which leaks the whole program). This keeps it
  family-scoped: a co-guardian reads accounts held by ANY guardian of their child.
  But this is exactly the kind of money-RLS predicate you should rule explicitly,
  hence the stop. CC will replay it under an impersonated co-guardian JWT (prove a
  co-guardian sees ONLY their child's family account, and a same-program stranger
  sees 0) before/after apply, same as PR-0b.
