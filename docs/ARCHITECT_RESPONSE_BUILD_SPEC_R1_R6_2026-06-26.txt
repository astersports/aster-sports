R6 — OQ6 (division games source) → RULE: OPTION A (division_games table).
--------------------------------------------------------------------------------
WHY A, not B/C. The fact you surfaced is decisive: opponent-vs-opponent games have no
events row and no game_results row, so "query events by tournament_division_id" yields
standings built only from games WE played in. That isn't a smaller version of the
feature — it's a WRONG one: every external team's record is understated (only their
games against us count), so 1st/2nd, the advance line, and the predictor's odds are
all computed on a false board. A predictor that publishes "78% to make the bracket"
off partial standings is confidently wrong — the exact failure D-FV5's deterministic-
truth principle exists to prevent. The operator's headline ask (true division
standings + a real predictor) is unbuildable without the full board. So:
  - B (events-only/partial) is rejected for any surface that shows division standings
    or odds — it would publish wrong numbers.
  - If a faster v1 is ever wanted, the HONEST smaller scope is to ship tracking +
    schedule + live scores WITHOUT division standings/predictor — never wrong
    standings. But A is cheap enough (the engine already eats a generic games array)
    that there's no reason to defer it. Build A.

AUTHORITY MODEL (the part that needs to be right). A scraper-fed division_games table
risks becoming a SECOND source of truth for OUR games (which also live in
game_results) → drift (scraped 50-48 vs our official 50-49). Pin it so each game has
exactly ONE source:
  PREFERRED (a): division_games holds EXTERNAL-vs-EXTERNAL games ONLY. Our games stay
     authoritative in game_results/events. The standings RPC UNIONs (our games scoped
     by tournament_division_id) + (external games from division_games), then feeds the
     engine. No game lives in two places → no mirror job, no conflict rule, no drift,
     and our official score can never be shadowed by the scraper. The read is a UNION
     instead of one SELECT — cheap, and worth the hard one-source guarantee.
  ACCEPTABLE FALLBACK (b) [your lean — whole board in division_games, our games
     mirrored in, game_results wins on publish]: workable, but it reintroduces a
     mirror step + a conflict-resolution rule + a drift surface (a lagging/failed
     mirror publishes a stale scraped score for our own game). Only take (b) if you
     have a concrete reason the single-table read outweighs that; default is (a).
  EITHER WAY: division_games rows reference tournament_division_teams (the standings
     units, by resolved_key) for both sides — not raw teams/opponents — so the engine
     groups consistently. And division_games inherits the standings posture: staff/
     service-role WRITE only (the scraper ingests via the authenticated path — that
     cross-program ingestion auth is still its own standing review), NO anon write,
     public READ only via get_public_tournament_standings.

→ STEP 1 can finalize: add division_games (model (a) preferred) to the Track-0 schema,
  owner applies, CC drafts get_public_tournament_standings UNION-ing the two sources
  into the engine.
