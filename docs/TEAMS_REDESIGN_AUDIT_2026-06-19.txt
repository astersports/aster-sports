1. EXECUTIVE SUMMARY (the five things that matter)
--------------------------------------------------------------------------------
1. ROSTER-EDIT UI IS ENTIRELY ABSENT. You can create/edit/delete a TEAM
   (TeamFormSheet) but there is NO surface anywhere to add a player to a team,
   remove a player from a team, set a jersey number, or move a player between
   teams mid-season. The only writes that touch roster membership today are the
   season-rollover RPC and the public registration RPC. (§2, §4)

2. REGISTRATION → ROSTER IS MANUAL AND DECOUPLED. submit_registration() creates
   guardians/players/registrations/fees but writes NOTHING to team_players /
   roster_members / financial_accounts. "Mark confirmed" flips a status enum and
   nothing else. So "ported but not all on it" = players exist with pending/no
   roster rows. "Never accepted" = registrations.status='pending'. (§4, §5)

3. THE FINANCIAL MODEL IS FAMILY-KEYED, NOT PLAYER-KEYED. The money lives on
   financial_accounts(guardian_id, season_id) and the family_balances VIEW; the
   roster is player-keyed. financial_transactions HAS an unused applied_to_player_id
   column — so per-player attribution is possible WITHOUT a billing rewrite, but
   it is a real architectural fork (annotate-family vs new-player-account). (§5, §7-FORK-A)

4. DEPOSIT / PAYMENT-PLAN SCHEMA IS COMPLETE BUT 100% DORMANT. payment_plans +
   payment_plan_installments (WITH per-seq due_date + amount_cents) + invoices
   (WITH due_date) + discounts all ship with RLS and correct columns — VERIFIED
   live — but have ZERO UI, ZERO RPC, ZERO reads, and ZERO rows in the entire DB.
   So "$100 deposit, $400 due June 22" is fully REPRESENTABLE with existing
   schema (a plan + 2 dated installments, or an invoice w/ due_date); the gap is
   purely UI + one RPC, NOT a migration. This is the single biggest build
   decision — but cheaper than first assumed. (§6, §7-FORK-C)

5. CREDIT IS A POSITIVE ADJUSTMENT, NOT A STORED POOL. "In-house credit" today =
   a financial_transactions row of type 'adjustment' with a credit sign. There is
   no credit-balance ledger, no "apply credit" affordance. A negative family
   balance (overpaid) is the only "credit" representation. (§5, §7-FORK-D)

6. SUMMER SELECT IS A CLEAN SLATE. 24 players are rostered across the two new
   teams; there are ZERO financial rows / registrations for the season. Frank's
   described payment states are states he needs to ENTER from scratch (tracked
   out-of-band today), not data to migrate. No legacy summer money is at risk.
   (§3.E)

Net: the redesign is ~30% "surface the data we already have" (payment dot →
real numbers, roster read view) and ~70% NET-NEW write plumbing (roster member
CRUD + per-player financial actions + optionally the deposit/plan layer), most
of which needs new SECURITY DEFINER RPCs (§9 lists the 7 missing primitives).
The substrate is friendlier than feared: the money model is immutable + well-
constrained, and the deposit/plan/invoice schema is complete-but-dormant, so
most of the build is UI + thin RPCs over an existing data shape.

  >>> SAFE REMOVAL SEQUENCE (load-bearing for ASK-4-iv "delete from roster"):

3.B FINANCIAL (family-keyed ledger + view)
  financial_accounts   — one row per (org_id, guardian_id, season_id). UNIQUE.
     season_fee_cents + discount_cents are LEGACY/zeroed (billing unified 2026-06-11).
  financial_transactions  — APPEND-ONLY ledger (trigger fin_tx_append_only blocks
     UPDATE/DELETE). Columns of interest:
       transaction_type CHECK IN ('payment','refund','adjustment','fee')
       amount_cents (signed for adjustment; positive otherwise)
       payment_method ('zelle'|'venmo'|'cash'|'check'|'stripe'|'other')
       reference, notes, processing_fee_cents, occurred_at, recorded_by
       reverses_transaction_id  (void = append a reversing row; pair nets out)
       applied_to_player_id  → FK to players, INDEXED, **NEVER WRITTEN (unused)**
       registration_id, invoice_id, discount_id  (FKs; mostly unused in UI)
  family_balances  (VIEW, security_invoker; migration 20260611105511)
       billed_cents   = SUM(amount) WHERE type='fee'
       net_paid_cents = SUM(payment) − SUM(refund) + SUM(adjustment)
       balance_cents  = billed − net_paid   (positive = owes; negative = credit)
       total_fees_cents, last_payment_at.  Keyed by account_id (+org+season+guardian).
  >>> This VIEW is the canonical money source for ALL surfaces (AP#63, financialMathAudit
      test). The redesign must NOT recompute billed/paid/balance inline. <<<

FORK-C  DEPOSIT / PAYMENT PLAN DEPTH
  6.1 thin (residual balance + note) vs 6.2 full (wired payment_plans + dated
  installments + reminders). NOTE the installment schema (payment_plan_installments

  sounds). Whichever way: the deposit-vs-full CHOICE needs a capture point.

FORK-D  CREDIT REPRESENTATION
  D1 (lean): "issue credit" = a positive 'adjustment' (reuse RecordPaymentForm

  D2: a real stored credit pool (new table) with explicit apply-credit flow.
  CC LEAN: D1 — matches the "partial + used in-house credit" Frank described
     (an adjustment that offset part of the fee). No pool needed for the pilot.

FORK-E  ACTIONS: DIRECT INSERT vs NEW RPCs
  Current financial writes are direct table inserts. The 4 roster actions touch
  money + roster across tables; doing them as SECDEF RPCs gives atomicity + audit
  + a clean RLS story. §9 lists 7 missing RPCs. CC LEAN: build the 4 roster-facing
  ones as RPCs (drop_roster_member, record_payment, issue_refund/adjust,
  void_transaction), defer invoice/plan RPCs to FORK-C.
