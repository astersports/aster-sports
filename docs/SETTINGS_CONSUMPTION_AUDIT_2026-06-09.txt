        organization_settings jsonb key the settings master spec touches, then

Premise (architect's pillar 1): consumption is the gating fact, not the schema
        inventory. A populated table is not a wired table.

LEGEND
  WIRED      = a shipped surface reads it today -> an editor is meaningful (build).
  SHIPPED    = already has an editor in /admin/settings -> nothing to build.
  ORPHAN     = nothing reads it (S3-class) -> do NOT build an editor; descope/park.
  MISMATCH   = the named substrate is orphaned; the app reads the SAME concept from
               a DIFFERENT relation -> re-spec against the consumed source, or park.

==============================================================================
FLAGS WORTH THE ARCHITECT'S EYE
==============================================================================
  1. tiebreakerExplainer (engine template T3) HARDCODES the tiebreaker copy that
     circuit_rules is meant to drive. That's a latent divergence: editing
     circuit_rules would never change the rendered explainer. If "operator edits
     standings rules and Records reflects it" is a real Phase-4 goal, the consumer
     work is "make tiebreakerExplainer read circuit_rules", not "build an editor".
  2. division_fees vs registration_fees: two tables for the same concept (fees).
     Worth a ground-truth-table entry (§11.5) naming the canonical one before any
     S5 work, so the next session doesn't re-spec against the orphan.
  3. roster_rules / S8 schedule keys don't exist as consumed keys at all — they are
     aspirational master-spec rows, not dormant-but-real config. Park, don't tag a
     table (there's no table — they'd be jsonb keys on organization_settings).
