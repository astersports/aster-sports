------------------------------------------------------------------------------
The SECDEF RPC is the correct fix for the RLS-blocked client UPDATE (a parent's
session cannot flip their own user_id IS NULL row). Approved as the approach.
A self-service claim is the one place a bug = account takeover, so the EXACT SQL
returns to architect before apply, and it MUST satisfy ALL of:

  1. SESSION-DERIVED EMAIL. The email matched against guardians is read
     server-side from the authenticated user (auth.uid() -> auth.users.email),
     NEVER a caller parameter. This is the entire claim-vs-takeover line.
  2. CONFIRMED ONLY. Require auth.users.email_confirmed_at IS NOT NULL before any
     claim. An unconfirmed email cannot claim.
  3. EXACT NORMALIZED MATCH. lower(trim(guardians.email)) = lower(trim(session_email)).
     NOT a wildcard ILIKE. (Defense in depth against pattern injection.)
  4. NULL -> SELF ONLY. Only flip guardians.user_id from NULL to auth.uid().
     NEVER touch a row already claimed by another user. No reassignment, ever.
  5. MULTI-MATCH: claim ALL unclaimed guardian rows matching the confirmed email
     (they are the same person); skip any already claimed by another. Fail safe.
  6. SECDEF HARDENING: SET search_path = public, auth; REVOKE EXECUTE FROM public,
     anon; GRANT EXECUTE TO authenticated (AP #23/#57).
  7. ATOMIC + IDEMPOTENT: flip + user_roles(parent) + populate in one tx; a repeat
     call is a no-op.
  8. FAIL CLOSED on email mismatch (guardian email != session email -> no claim;
     correcting a guardian's email is an admin path, not a claim bypass).

  ABUSE PROOF before "done" (like #710): prove a user (a) claims only their own
  email's row, (b) CANNOT claim another's row, (c) CANNOT claim with an
  unconfirmed email, (d) repeat call no-ops, (e) anon/public cannot execute.
