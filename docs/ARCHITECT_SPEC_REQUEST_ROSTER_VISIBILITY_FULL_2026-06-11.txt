0. PURPOSE
   Frank's directive: "Parents should only see their kids and only other kids on
   their team rosters. Plus on the program and team setup we should have a setting to
   mask or not show all roster players to other players. Tryouts no, camp maybe, team
   yes." The RLS-enforced default is shipped; this asks the architect to spec + render
   the ADMIN CONTROLS (program toggle + team override) and the PARENT-FACING
   hidden-roster treatment.

------------------------------------------------------------------------------
  - programs.roster_visibility (nullable boolean). NULL = default by program_type:
    season VISIBLE, tryout HIDDEN, camp HIDDEN. Existing rows backfilled by type.
    (Frank ratified camp default = hidden.)
  - current_user_teammate_player_ids() now excludes teams whose program's effective

------------------------------------------------------------------------------
2. DATA MODEL (current + proposed)
------------------------------------------------------------------------------
  CURRENT: programs.roster_visibility (boolean, nullable; NULL => by-type default).
  PROPOSED ADD (this build): teams.roster_visibility_override (boolean, nullable;
    NULL = inherit the program). EFFECTIVE visibility for a team =
      COALESCE(team.roster_visibility_override,
               program.roster_visibility,
               program.program_type = 'season',
               true)
  The teammate fn switches to read the team override first, then the program, then
  the type default. (§16.16 two-bucket: the override column is READ by the fn AND
  WRITTEN by the team form in the SAME build — not dead config.)
  program_type domain (live): season | tryout | camp.

3. OPEN DECISIONS FOR THE ARCHITECT (CC lean per AP#39)
------------------------------------------------------------------------------
  RV-2  TEAM OVERRIDE shape. CC LEAN: a 3-state control on the team —
        Inherit (default, = program) / Show / Hide — stored as
        teams.roster_visibility_override (NULL / true / false). Inherit keeps the
        common case zero-touch; Show/Hide is the per-team exception (e.g. one camp
        team you DO want visible). RULE: confirm 3-state vs a 2-state "override on/off".

  RV-5a PROGRAM CONTROL placement + copy. CC LEAN: a single toggle in the program
        EDIT form (src/components/admin/program-detail/ProgramEditSheet.jsx — a
        FullScreenForm; current fields: name, dates, reg window, public link,
        published). Label lean: "Show roster to families", helper line naming the
        type default ("Tryouts and camps hide the roster by default"). New programs
        inherit the type default automatically (NULL), so the CREATE flow does not
        strictly need the control — RULE: surface it on create too, or edit-only?

  RV-5b TEAM CONTROL placement + copy. CC LEAN: the 3-state control in
        src/components/admin/TeamFormSheet.jsx (a FullScreenForm; already receives
        the programType prop, so it can show "Inherit (Hidden for camps)" inline).
        Label lean: "Roster visibility" with Inherit / Show / Hide.

  RV-5c PARENT-FACING hidden-roster treatment (THE render that doesn't exist yet).
        When a team's roster is hidden, a parent on that team currently sees only
        their own child in any roster surface (RLS returns own kid). QUESTION: what
        should that surface SAY? Options: (a) show only the own child silently;
        (b) show the own child + a quiet line "Rosters aren't shown for this
        program"; (c) hide the roster section entirely for hidden programs.
        CC LEAN: (b) — a quiet explanatory line so the parent isn't confused by a
        one-kid roster. This needs an architect render (the Teams/roster surface for
        a parent on a hidden program).

  RV-6  WITHIN-TEAM granularity. CC LEAN: team-level all-or-nothing for v1; no
        per-player opt-out (carry a trigger for later). RULE: confirm.
