B. SEASON ACTIVATION SHAPE (§3 — for the unified activate())
------------------------------------------------------------------------------
src/hooks/useSeasons.js:59-69, setActive(id):
  step 1: supabase.from('seasons').update({status:'archived'}).eq('org_id',orgId).neq('id',id)
  step 2: supabase.from('seasons').update({status:'active'}).eq('id', id)
  then refetch.
Confirmed facts the unified activate() must honor:
  - TWO separate awaits, NOT a transaction. If step 2 fails after step 1, the
    org is left with ZERO active seasons. (The unified version should wrap both
    in a single transaction/RPC — strict improvement.)
  - It operates on the `seasons` VIEW (= programs WHERE program_type='season'),
    so "archive all others" is implicitly scoped to seasons.
  - SCOPING NUANCE: a generalized activate() over `programs` (all types) MUST add
    `.eq('program_type', target.type)` to the archive-others step, or activating
    a season would archive nothing / activating any type could over-reach. The
    current code gets type-scoping "for free" from the view; the generalized one
    must scope explicitly by type (+ org). This is the load-bearing detail.
  - single-active is APP-ONLY (no DB trigger/constraint/unique index — re-
    confirmed). The §3 optional partial-unique index would need to be partial on
    (org_id, program_type) WHERE status='active' AND program_type IN
    (the singleActive types), since only some types are single-active.
