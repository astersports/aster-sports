6. PROPOSAL B — BRANCH / GOVERNANCE DISCIPLINE (prevent wrong-branch writes)
-----------------------------------------------------------------------------------------
- Single rule, all repos: feature branch off `main` → PR → green CI → merge. Never commit
  to `main`. Routine auto-merges on green; money/auth/schema hold for Architect (§2.4).
- Every repo declares the rule in README §5.5 (and CLAUDE.md where present).
- Observed risk THIS session: the task framing assigned ONE branch name
  (`claude/railway-whoami-vh67cj`) across all five repos, but real work correctly went to
  descriptive per-change branches. Recommendation: drop the shared pre-assigned branch
  name; each change names its own branch. A pre-assigned cross-repo branch invites
  exactly the "wrote to the wrong branch" error.
- CI guards stay the gate (each repo's CI blocks merge on red).

8. PROPOSAL D — CROSS-SURFACE "ONE SOURCE OF TRUTH" SYNC (vision → architecture)
-----------------------------------------------------------------------------------------
OPERATOR VISION: one tournament/schedule source that, when updated, live-propagates across
every surface — the app (aster-sports), the marketing site (astersports.io), and the
tenant/agency websites (legacy-hoopers, st-pat). "One tournament website live-pauses and
updates the structure + scheduling on multiple things."
EVIDENCE / PRECEDENT: `aster-weather` already proves the shared-code model (one package,
many consumers). aster-sports already owns the canonical schedule/tournament DATA (Supabase
`vrwwpsbfbnveawqwbdmj`, the tournaments/events tables + the comms engine).
ARCHITECTURE OPTIONS (for the Architect to choose — design-first, not built yet):
  (i)  Shared DATA source: the websites read the app's Supabase (RLS-scoped read views or a
       public read API) so schedule/tournament state has ONE home (the app's DB). Cleanest
       single-source; needs a safe public read surface.
  (ii) Shared PACKAGE: extract a tournament/schedule render+fetch package (like
       @aster/weather) consumed by app + sites. Good for render parity; still needs a data
       source under it.
  (iii)Publish/event propagation: app emits schedule changes (webhook/edge function) that
       rebuild/refresh the static sites. Good for static hosts; more moving parts.
CC LEAN: (i) as the spine (one DB = one truth) + (ii) for render parity, with (iii) only if
  a site is statically hosted and can't read live. But this is an Architect design call —
  flagged, not decided.
