§1 THE TIME-STATE SPINE  (PR-A' foundation — build this FIRST)
==============================================================================
The redesign rests on ONE shared time-state primitive. Today there are 4
divergent "is it done?" definitions and 3 duration constants; in-progress games
fall into "Past" mid-game; 5 production events with NULL end_at never gray out.
The spine collapses all of that.

1.1 HELPER  —  src/lib/eventWindows.js  (NEW)
  export const EVENT_DEFAULT_DURATION_MS = 2 * 60 * 60 * 1000; // 2h, the ONE constant
  export function eventEnd(event) {
    // end_at when present; else start_at + default duration. Never returns null
    // for an event with a start_at.
    return event.end_at ?? (new Date(new Date(event.start_at).getTime()
           + EVENT_DEFAULT_DURATION_MS)).toISOString();
  }
  export function eventTimeState(event, now = Date.now()) {
    const start = new Date(event.start_at).getTime();
    const end   = new Date(eventEnd(event)).getTime();
    if (now < start) return 'upcoming';
    if (now <= end)  return 'happening_now';
    return 'completed';
  }
  CONTRACT: three states only — 'upcoming' | 'happening_now' | 'completed'.
  Every consumer reads this. The 4 legacy is-past predicates and the 3 scattered
  duration constants are DELETED and re-pointed here (grep gate in §8).

1.2 STATE TREATMENTS (the visual contract the renders encode)
  upcoming      -> full-color card; team-color left stripe at full opacity;
                   the NEXT upcoming event in the active filter is the NOW SLOT
                   (pinned to top of Upcoming, accent hairline, countdown chip).
  happening_now -> "live" treatment: green status dot + "Happening now" label
                   where the countdown would be; card full color; sits above
                   Upcoming in its own implicit band (no separate header needed
                   when only the NOW slot is live; if multiple, a "Happening now"
                   subhead).
  completed     -> retired into the COMPLETED section (collapsed by default,
                   header carries summary "Past 7 days · N events"); card at
                   0.55 opacity; W/L/T chip when a published game_result exists
                   (SD-5: League/Practice never carry a result; Games may).

1.3 COUNTDOWN (CP-5 fix)
  Countdown is the UPCOMING treatment at EVERY density, not a detailed-only chip.
  The minimal-density branch that suppressed it is deleted. Show countdown on the
  NOW SLOT always; on other upcoming events show it when < 24h (existing
  threshold, retained). Format: ">2d" / "Tomorrow 4:30p" / "in 3h" / "in 25m".
  At start (countdown hits 0) the card flips to happening_now via eventTimeState,
  not via a separate timer.

1.4 COMPLETED SECTION
  Window: last 7 days (existing). Collapsed by default. Header summary text.
  Inside, newest-first. W/L chip pulls from the published game_result composed
  through the events gate (P0 §D3 shape). No result -> no chip, just the dimmed
  card. NO per-player stats (§16.12).

1.5 DB-8 (import writes end_at NULL)
  Forward fix lands in the wizard/import path (PR-C'). The spine TOLERATES NULL
  end_at via eventEnd() so the 5 legacy NULL-end events gray out correctly NOW;
  the import fix stops the NULL class from growing. Both required; neither alone.

§10 CARD CONTENT MODEL AMENDMENT  (operator directive, 2026-06-12)
==============================================================================
SUPERSEDES the §5 PARENT line "parent sees THEIR commitments, not the org
rollup." Operator call: parents are high-engagement and want the basics
GLANCEABLE without opening detail. Most parents never open the detail page.
The card therefore carries the basics at EVERY density, COMPACT INCLUDED.

10.1 THE ON-CARD GLANCEABLE ROW (all roles, all densities)
  A chip row sits below the location line, carrying in priority order:
  (1) WEATHER chip (icon + temp, e.g. "Sun 63 deg"). Shown when the event is
      within the forecast window AND the venue has coords. Per-venue weather is
      gated on the DB-10 coord backfill (SD-9): the NOW slot gets weather
      immediately; full per-card weather rolls out as venues are backfilled. No
      coords / outside window -> OMIT (no placeholder, no "weather unavailable").
  (2) ATTENDANCE COUNT "N going" — the going RSVP count.
      PRIVACY: a count is aggregate (no names) and is allowed where names are
      not. BUT suppress the count when the team's RESOLVED roster_visibility is
      Hidden (tryout/camp), where even a count is sensitive — show only the RSVP
      control there. Everywhere else the count shows.
  (3) RIDES-NEEDED chip "N rides needed" (amber) — unfilled ride count, shown
      only when > 0 (SD-15 non-zero always-visible at every density). When 0,
      a quiet "Rides covered" check or omit.
  (4) VOLUNTEER/DUTY chip "N needed" / "Snacks open" (amber) — unfilled duty
      count, shown only when > 0.

10.2 COMPACT CARD (the density most parents live in)
  Compact tightens padding and type but KEEPS, non-negotiably: time, weather,
  countdown, title, team dot, location, the 10.1 chip row, and (parent) a 44px
  RSVP control. The chip row is REQUIRED in compact — it is the whole point of
  the directive. Display chips are not tap targets and MAY be < 44px; the RSVP
  control stays >= 44px. Countdown + weather right-align on the time row.

10.3 DATA (no new N+1 — extends the §4 batch hook)
  useScheduleData returns per event: going_count, rides_unfilled,
  duties_unfilled, weather. Weather is batched per DISTINCT venue+date (not per
  event) and cached, so the chip row adds ZERO per-card requests (VF-11 holds).
  going_count / rides_unfilled / duties_unfilled come from the same batched
  RSVP + coverage fetch already specced in §4.

10.4 ROLE NOTES
  - PARENT: shows the full 10.1 row PLUS their own commitment line ("You're
    bringing snacks") when applicable. The count is the org count; the
    commitment line is personal. Both can appear.
  - COACH/ADMIN: already carry the ops chips (§5); 10.1 unifies the chip styling
    and adds weather. Coach/admin counts are never suppressed (staff see roster).
