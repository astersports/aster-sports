CONFIRMED P0s (CC-verified against source — fix-ready)
------------------------------------------------------------------------------
P0-1  FINANCIALS — coach payouts shown 100× too small.
      `CoachPayoutsSection.jsx:56` calls `formatCurrency(p.amount_cents / 100)`.
      `formatCurrency` (formatters.js:67) ALREADY divides by 100. A $500 payout
      (amount_cents=50000) renders "$5.00". Every payout on the admin
      reconciliation ledger is understated 100×. AP#63 (unit divergence).
      FIX: drop the `/100` → `formatCurrency(p.amount_cents)`. One line.
      GUARD: the existing `financialMathAudit.test.js` greps for re-implemented
      accumulation but NOT for `formatCurrency(x / 100)` mis-unit calls — add a
      static-grep failing on that shape (closes the exact blind spot the bug
      slipped through).

P0-2  EVENTS — volunteer slots DUPLICATE on every edit-save.
      `useUpdateActivity.js:49-58` (single-event `update()`) re-INSERTs
      `event_duties` from `formData.duties` with NO prior DELETE. The edit
      wizard re-hydrates the full existing duty set into `form.duties`
      (CreateActivityWizard.jsx:51-64), so each save of an event that has
      volunteer slots doubles them (Scorekeeper ×2 → ×4 → ×6). Silent,
      compounding, data-level. (useCreateActivity inserts fresh — only the
      EDIT path is broken.) AP#13 / write-correctness.
      FIX: delete existing event_duties for the event, then insert (or
      reconcile by diff). Mirror in updateSeries' reconcileSeries if it has
      the same shape (verify). Add a test: edit-save twice → slot count stable.

DOCTRINE-1  MEMBERS — per-player GAME STATS are live (decision, not a code call).
      `PlayerProfilePage.jsx:65-90` (+ `usePlayerSeasonStats.js`) renders PTS,
      REB, AST, GP, FG, 3PT, FT, STL, BLK, TO, PF for a single player at
      `/teams/:teamId/player/:playerId`. §16.12 locks per-player GAME stats as
      forbidden in 2026 (team-level only; engagement stats OK). §8 build order
      states explicitly: "BLOCKED: 5-C player stats/box score — §16.12 forbids
      per-player game stats in 2026 (do not build)." This is the forbidden
      build, shipped + mounted. Likely orphaned (RosterSection.jsx:37 passes a
      DEAD `teamId` prop to PlayerRow, suggesting the deep-link in was
      half-removed — so the page may be URL-reachable but unlinked). This is a
      DOCTRINE RULING for Frank/architect per §11.8 r6, NOT a unilateral fix.
      OPTIONS: (a) strip the game-stat block, keep the page as identity +
      ENGAGEMENT only (attendance/streak, §16.12-allowed); (b) remove the
      page + route entirely (if orphaned); (c) ratify a §16.12 carve-out
      (unlikely). CC LEAN: (a) — keep an engagement-only profile, strip the box
      score; it's the smallest doctrine-compliant surface.

PATTERN ALPHA — AP#36 unchecked-error on the SECONDARY/enrichment query.
  The dominant cross-section defect. The PRIMARY query is always error-checked;
  the enrichment query (counts, names, tournaments, balances) swallows its
  error into an empty/zero state that reads as real data. ≥10 instances:
    Schedule: useDigestEvents.js:57 + :67, ImportSchedulePage.jsx:34,
              useTournamentAggregateRsvp.js:48, useScheduleData.js:54 (logged-
              only, softer).
    Events:   useConflictCheck.js:26 (SAFETY-CRITICAL — swallowed error → zero
              conflicts → admin double-books with no warning).
    Financials: CoachPayoutsSection.jsx:14-26 (error → empty "no payouts" reads
              as "all coaches paid"), FinancialDashboardPage.jsx:35-43
              (Promise.all, no .error on any leg → page hangs on "Loading…"),
              FinancialImportPage.jsx:29-38 (misleading "no active season").
    Members:  PlayerProfilePage.jsx:24-33 (Promise.all, no .error → false
              "Player not found").
  → SWEEP PR: destructure `error`, surface before using `data`. Extend the AP#36
    audit grep to "second query in a multi-fetch / Promise.all". Prioritize the
    safety/money callsites (useConflictCheck, CoachPayouts, FinancialDashboard).

PATTERN BETA — AP#63 same concept, divergent source/scope (the platform's #1
  bug class; present in ALL FOUR sections).
    Members:  academy badge — RosterSection/PlayerRow read players.member_type
              (global) vs PlayerProfilePage reads team_players.roster_type
              (per-team). Agent VERIFIED a live contradiction on a real player
              (member_type='futures_academy' but roster_type='rostered' →
              roster shows Academy pill, profile shows none). This is the

    Events:   going-count derived 4 independent times (EventDetailHero.jsx:23,

              :124); past-event predicate 3 ways (one off the SD-2 spine,
              EventBriefingHistory.jsx:48 uses start-based < now); check-in
              denominator uses full roster vs hero uses `eligible` (B8). Agree
              today (shared inputs) but no AP#43 invariant test locks them.
    Financials: P0-1 unit divergence; plus format split — CoachPayouts full
              cents vs CoachCompCard/AdminProgramHealth whole-dollar rounded
              for the SAME "coach pay" concept.
    Schedule: public venue line (PublicSchedulePage.jsx:36 raw `location`) vs
              in-app (useActivities.js:50 `location || locations.name`) — public
              rows show NO venue for imported/tournament events; in-app shows it.
