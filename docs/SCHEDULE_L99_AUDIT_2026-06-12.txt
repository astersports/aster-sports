SCH-1  Hardcoded LH-cobalt tint on game/tournament cards
  EventCard.jsx:46 — backgroundColor literal rgba(74,143,212,0.10)
  for game/tournament events. That is Legacy Hoopers' accent-soft
  baked into a platform component: every other tenant's schedule will
  render LH blue tints over their own brand. Should be
  var(--as-accent-soft) (identical value for LH at runtime via org
  override). AP #8 / §0 rule 4 violation; the §0 grep #3 only scans
  #FFFFFF so this class evades it.

SCH-2  Parent list fires N+1 queries (per-child, per-card)
  ChildRsvp.jsx:25-29 (player_activations, one query per academy
  child per card) + :31-44 (event_rsvps, one query per child per
  card). A parent with 2 kids and ~20 visible cards ≈ up to 60-80
  requests on first paint; rsvpCache only suppresses re-renders, not
  initial fan-out. Meanwhile useEventRsvpCounts already batch-fetches
  every rsvp row for the same events (2 queries total). Fix shape:
  batch hook keyed event×player (one .in() for rsvps, one for
  activations) or derive per-child response from the rows the counts
  hook already pulls. AP #4 class.

SCH-3  Sub-44px tap targets on primary controls
  ChildRsvp.jsx:87 (compact minHeight 32 — the parent's MAIN action
  on this page), shared/FilterSelect.jsx:10 (minHeight 36 — the
  team/type/child filters on both views),
  schedule/ShareScheduleButton.jsx:37 (copy button minHeight 36).
  §7 "44px minimum (non-negotiable)" / §16.4.

SCH-4  In-progress events vanish into the collapsed Past section
  ScheduleListSections.jsx:25-31 — lookbackMs=0 means upcoming =
  start_at >= now: an event leaves "Next 7 days" the second it
  starts and lands inside the closed "Past 7 days" expander while
  it is being played. GamesView.jsx:66-67 same split (in-progress
  game shows under Results with a "—" pending marker). Game day is
  the highest-traffic moment for this page; the event being played
  is the one thing not visible. → SD-2.

SCH-5  Fetch errors invisible on the page
  useActivities.js:53-59 deliberately preserves last-known-good and
  returns `error` for a "Couldn't refresh — retry" surface;
  SchedulePage.jsx:26 destructures only {activities, loading,
  refetch} — the error contract is built upstream and consumed
  nowhere. On a flaky connection parents silently see stale-or-empty.
  Same omission at ParentHome/AdminHome/CoachHome (:35/:32/:33) —
  cross-surface; fix here + AP #43 invariant test covering all four.
  AP #36 corollary (error surfaced-but-unconsumed) + AP #44 lineage.

SCH-6  Density: doctrine drift cluster (§16.2)
  (a) EventCard.jsx:142 — `density !== 'minimal'` gates the parent's
      inline ChildRsvp. §16.2: density "never gates whether a
      Needs-you item appears." Compounding: Schedule's density
      defaults to 'minimal' (useDensity.js:6 FALLBACK + no explicit
      default at SchedulePage.jsx:37), so a fresh parent account gets
      NO inline RSVP on Schedule until they discover the Detailed
      toggle. Fix regardless of SD-1: ChildRsvp renders in minimal
      too (it already has a compact row form).
  (b) Per-section density keys still live: 'schedule-list'
      (SchedulePage.jsx:37, ScheduleListSections.jsx:41),
      'locations-list' (AdminLocationsPage.jsx:26,44), 'rides-list'
      (EventRidesTab.jsx:67, ride/OfferCard.jsx:33). They SHADOW the
      Account-level card_density.default written by
      AppearanceForm.jsx:43 (resolution order in useDensity.js:24-30:
      sectionKey -> default -> fallback). Once a user touches the
      Schedule toggle, the Account "Density" preference permanently
      stops affecting Schedule. Two controls, two truths. And NO home
      surface consumes useDensity at all (grep: zero home callers) —
      §16.2's "home-level setting" currently governs no home cards.
      → SD-1 (genuinely open: doctrine text vs live shape).
  (c) Retired 3-level remnants: EventCard.jsx:15 default
      density='medium'; EventCard.jsx:113 `density !== 'maximum'`
      branch reachable ONLY via the undefined-density path (see
      SCH-15) — i.e. the third state survives as implicit behavior.

SCH-7  Two different ShareScheduleButton components (same name)
  schedule/ShareScheduleButton.jsx (mounted by SchedulePage.jsx:21,81)
  = copy-link only, builds URL from window.location.origin, custom

  (:12-14 — navigator.clipboard?.writeText no-ops, setCopied(true)

  SchedulePage.jsx:81 passes teamName="" so the share sheet / modal

SCH-8  game_results: duplicate fetch paths + an unhandled-throw
  GamesView.jsx:14-33 defines a private useGameResults even though
  SchedulePage already computed useGameResultsMap for the same events
  (GamesView just doesn't receive it). The private copy also refetches
  every 60s on the Games tab: pastIds (GamesView.jsx:66-68) is rebuilt
  each useNow tick, new array ref -> effect refires. And the shared
  hook useGameResultsMap.js:22 does `if (error) throw error` inside
  .then() with no catch — an unhandled promise rejection (Sentry noise,
  zero UI signal); GamesView's copy console.errors; the counts hooks
  console.warn. Three error styles for one concept. Consolidate to the
  one hook, {data,error} contract, pass down from the page.

  from device-local getDate()/getDay(): for a non-ET device near
  midnight the chip's day number and its underlying date diverge.
  :14 useMemo([]) freezes "today" at mount (stale strip after

  SchedulePage.jsx:43-45 — eventDates feeding the dots come from RAW
