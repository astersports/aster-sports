C-2 [BREAKS WHEN S2 LANDS — highest-risk pairing] the CLIENT rollover team-copy
    (useSeasonRollover.js:42-45) omits team_type_id. The moment S2 (teams.
    team_type_id NOT NULL) lands, every rollover team-copy 23502s mid-loop, and
    because the rollover is NON-ATOMIC (C-3) it strands a half-rolled season.
    The staged S1 rollover RPC already carries team_type_id forward — so the fix
    is to land S1 + switch the client to the RPC BEFORE S2's DDL. Sequencing is
    correctness-load-bearing (the architect already locked S1-before/with-S2; this
    is the concrete reason).

    new-season insert -> OLD-season archive -> team/roster loop -> audit row, no
    txn. A mid-loop throw (C-2, or any transient) leaves the new season created
    AND the old one already archived; re-running creates a SECOND season. S1's
    single-RPC rebuild closes this. (CLAUDE.md §8's "7-B atomic execution" claim
    is false — already in the retraction.)

C-4 [edge] S3 (registrations partial-unique, APPLIED) raises a raw 23505 on a

    player_id) WHERE status<>'cancelled' DO NOTHING closes it; until then the
    client maps no 23505 -> generic "didn't go through". (useSubmitRegistration.js)
