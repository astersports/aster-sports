------------------------------------------------------------------------------

175 while only 2 signed in. "Go big on the Hub" therefore puts account-claim on
the critical path. Claim is the on-ramp for the ENTIRE in-app parent experience
(Hub, RSVP, inbox), not just the Hub.

------------------------------------------------------------------------------
4. CLAIM FLOW MECHANICS
------------------------------------------------------------------------------
  a. Guardian record has an email on file.
  b. Claim entry = magic link (passwordless) to that email, delivered via the
     briefing CTA or a dedicated invite (Resend, existing channel).
  c. Claimant clicks -> authenticates (magic link / OAuth).
  d. VERIFY the authenticated email matches the guardian record's email, then
     write auth user_id onto guardians.user_id (the claim). Never link a user to
     a guardian whose email they did not prove.
  e. ON CLAIM, populate the household layer (FAMILY_HUB_BUILD_PLAN §2):
       - create family_children from the guardian's linked players
         (roster_members / players — map from the REAL schema the pre-flight read).
       - create child_links (link_type 'ember_player', this org) per player.
     Hub lights up immediately post-claim.
  f. EDGE CASES:
       - multiple guardians per family: each claims their own guardian record.
       - a kid rostered on multiple teams: multiple player rows -> match to ONE
         family_child (de-dupe by player identity), multiple child_links.
       - already-claimed / re-claim: idempotent; do not double-link.

------------------------------------------------------------------------------
5. PRIVACY / RLS
------------------------------------------------------------------------------
- Link a user ONLY to their own guardian record + their own players, on verified
  email ownership.
- Household tables stay user-scoped per FAMILY_HUB_BUILD_PLAN §5; a claim creates
  rows owned by the claiming user only.
- No path where claiming exposes another family's data.
