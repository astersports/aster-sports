0. WHY G8 EXISTS (the dependency the architect named)
------------------------------------------------------------------------------
SEAM-3: the recovery sweep (G5) re-invokes send-tournament-message to re-drive
stuck `queued`/`failed` sends. That is SAFE ONLY IF the send is idempotent — and
CC verified the idempotency was NOT test-covered. A sweep built on unproven
idempotency could double-send to every family — worse than the stranded send it
fixes. So G8 closes the test gap FIRST. G5 does not re-drive until G8 is green.

1. WHAT IDEMPOTENCY ACTUALLY MEANS HERE (two guards, both load-bearing)
------------------------------------------------------------------------------
A re-invoke is safe because of TWO guards in send-tournament-message/index.ts:
  (a) MESSAGE-LEVEL: `if (alreadySent(message)) return 409` — a finalized message
      (sent_at set) short-circuits; never re-sends. (index.ts:101)
  (b) RECIPIENT-LEVEL: the recipient query filters `delivery_status='queued'`
      (index.ts:116), so a re-invoke only processes still-queued rows and never
      re-sends to already-`sent` ones. THIS is the one that makes a partial-send
      re-drive safe.
G8 proves (a) by unit test and LOCKS (b) by a static invariant test (a DB-query
guarantee a pure unit test can't reach).

2. APPROACH — AP#30 mirror extraction (the established pattern)
------------------------------------------------------------------------------
The edge fn is Deno; vitest can't import index.ts. Per AP#30, pure logic shared
with vitest lives in a mirrored pair. So the two decision kernels were extracted:
  NEW  src/lib/briefings/sendDispatch.js          (vitest source of truth)
  NEW  send-tournament-message/_dispatch.ts       (Deno mirror, registered in
       edgeFunctionMirrorAudit PAIRS, baseline 4 — matches the other typed pairs)
  - alreadySent(message)            -> the sent_at 409 decision (guard a)
  - classifyBatchResult(group,res)  -> per-recipient writeback decision: batch
       error => whole group failed (one batch-level error string); batch success
       => per row, id => sent / else => failed with "<email>: <error>".
index.ts now DELEGATES to these (the IO — the .update() writeback — stays in
index.ts). Refactored two spots only: line 101 (alreadySent) + the send-loop
writeback (classifyBatchResult + apply).

3. BEHAVIOR-EQUIVALENCE (the thing to check on a live send path)
------------------------------------------------------------------------------
Same rows get the same delivery_status, same sent/failed counts, same error
strings as the prior inline loop. ONE IO shape change: the writeback now does a
bulk `.update().in('id', failedIds)` + `.update().in('id', sentIds)` per batch
instead of a per-row Promise.all of `.eq()` updates. Identical OUTCOME (each row
ends at the same status), fewer round-trips. NOT touched: the 409 guard's effect,
the `delivery_status='queued'` recipient filter, the unsubscribe suppression, the
pilot fail-closed gate, the finalize (status='sent'+sent_at), the push fan-out.

4. TESTS (the deliverable — G8 IS the tests)
