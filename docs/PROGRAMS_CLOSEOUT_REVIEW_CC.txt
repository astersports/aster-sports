3. FRANK'S QUESTION — program taxonomy: freeform vs curated/general (LeagueApps)
------------------------------------------------------------------------------
Frank, verbatim:
  "Does it make sense to allow for a program setup Freeform if another type of
   program wants to be setup? Or use the general system like LeagueApps?"

Context / reference (LeagueApps manager, screenshots 2026-06-07):
  - LeagueApps "Create program" opens a fixed "Create a new..." archetype menu:
    League · Event · Tournament · Camp · Club team · Class. A CURATED, fixed set
    of program archetypes (not freeform) — each with its own setup flow.
  - Its Program Summary index groups by Live and Upcoming / Unpublished /
    Completed, with columns Activity Start · Invoiced · Paid · Outstanding ·
    Players · Staff. (Useful reference for our own index column set later.)

Our current model:
  - program_type is a DB ENUM: season, tryout, camp, clinic, interest_list,
    evaluation — a curated, DB-constrained set. ProgramTypeChooser + type-aware
    create (divisions season-only, status default per type, etc.).

The fork:
  (A) FREEFORM — let an admin spin up a program of an arbitrary/custom type (a
      generic container they configure) when a new offering doesn't fit.
  (B) CURATED/GENERAL — keep a fixed archetype set (like LeagueApps' 6), each
      with a tailored setup flow; extend the set deliberately when a genuinely
      new archetype appears.

Why it matters / CC framing (architect decides):
  - This is Fork 2 at the product level. Freeform program types become free-text
    -> the exact float-free-enum risk that produced briefings BUG B, AND it
    breaks per-type behavior (you can't "exclude camps from standings / brackets"
    if the type is an arbitrary string). Every per-type rule in the platform
    (statusForProgramType, team-type defaults, records exclusion, divisions
    gating, PROGRAM_NOUN) keys off a known type.
  - Data point worth weighing: LeagueApps — a mature, general youth-sports
    platform — chose CURATED archetypes, not freeform. That's evidence for (B):
    a curated set with good coverage + a deliberate "add an archetype" path beats
    freeform. If our 6 enum values don't cover a real offering, the move is to
    add an archetype (DB enum value + its setup flow + per-type rules), not open
    freeform.
  - CC lean: (B) curated, aligned with Fork 2's table+parity. But it's Frank's +
    architect's call; if there's a real need for one-off ad-hoc programs, a
    single generic "other/custom" archetype (still a known type, treated
    non-competitive by default) is the safer middle than true freeform.

  Sub-questions for the architect if (B):
   - Do our 6 enum values map cleanly onto the archetypes Frank actually runs
     (season=league, camp, clinic=class?, tryout, evaluation, interest_list)? Is
     anything LeagueApps offers (Event, Tournament, Club team) missing as a
     PROGRAM archetype vs already handled elsewhere (tournaments are their own
     surface; events are per-team)?
   - If a new archetype is needed, what's the "add an archetype" checklist
     (enum value + ProgramTypeChooser entry + statusForProgramType + PROGRAM_NOUN
     + team-type default + any records/index rules)? Codify it so adding one is
     mechanical and parity-tested (ties to Fork 2).
