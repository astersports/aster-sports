2.1 TeamDetailPage.jsx — page composition
────────────────────────────────────────────────────────

Lines 31–43: declares `role`, `myTeamIds`, `myChildren` from `useAuth()` directly. **Anti-pattern #42 violation** — should use `useHomeRole().activeRole` for page-level rendering (so admin "view as parent/coach" preview reaches this page). Prior Q1 in the L99 Teams doc flagged this.

Lines 44–53: `enrichedPlayers` `useMemo` joins `players` (from useRoster) with `grid` (from useAttendanceData) — i.e., the page fires BOTH hooks against the same teamId and reconciles them client-side. The two hooks return different shapes (useRoster gives jersey/grade/guardians from `roster_members` + `players` join; useAttendanceData gives goingCount/streak from `event_rsvps` + `event_arrivals` + `check_ins` aggregates). The enrichment is correct but means the page does the join the database could do.

Line 55: `team = programs.find((p) => p.id === teamId)` — O(N) lookup against the full programs list every render. For 5 programs this is a no-op; for orgs with 30+ teams it becomes warming-pressure on every re-render. Should be `useMemo`'d.

Lines 56–60: `nextEvent` derives from full `activities` list, filters by team_id + status + start_at, sorts ascending, finds first ≥ now. Correct, memoized. Note: depends on `useNow()` ticking, which re-runs the memo every minute and re-sorts the list.

Lines 61–62: `myChild` uses `myChildren.find` with two predicates (`teamIds.includes(teamId) || teamId === teamId`) — the second clause looks like a typo/legacy single-team shape; modern AuthContext should expose `teamIds` array uniformly. Hardly affects function but smells like a backwards-compatibility shim per §11 anti-pattern #16. Also returns ONE myChild — Q8 in prior doc flagged the sibling case (two kids on same team).

Lines 67–110: render tree.
- Line 68: page background uses `team.team_color + '08'` (8% alpha hex appended) — non-standard hex syntax (`#xxxxxxAA` works in modern browsers but is opaque to some old Safari versions; per CLAUDE.md anti-pattern #11 the project uses `rgba()` not hex+alpha). Subtle compat risk.
- Lines 73–87: TeamHeaderCard + standalone Send-briefing chip stacked. The chip is a `<Link>` styled as a button — semantically a link, visually a button. Right-aligned with negative top margin (`marginTop: -8`) — a layout hack signaling "this would be nicer inside the card."
- Line 89: MyChildSpotlight renders unconditionally if `myChildPlayer` exists. Sandwich between hero and CoachQuickActions for parents. For staff, this is null.
- Line 90: CoachQuickActions renders for staff. Two tiles: Message + Schedule.
- Line 91: UpcomingEvents renders for everyone (no role gate).
- Lines 93–103: Roster section, with loading skeleton + empty state branches.
- Line 105: TeamHeatmap. Renders for everyone (parent-filtered internally per `TeamHeatmap.jsx:27-28`).
- Line 106: TeamPlayerStats — staff only.
- Line 107: TeamAchievements — renders only if achievements exist (gated inside component).
- Line 108: MessageTeamFAB — staff only.

2.3 TeamHeatmap.jsx — the "2% Going" bug deconstructed
────────────────────────────────────────────────────────

Line 30: `totalGoing = visibleGrid.reduce((s, r) => s + (r.goingCount || 0), 0)`.
Line 31: `totalPast = visibleGrid.reduce((s, r) => s + (r.totalPast || 0), 0)`.
Line 32: `teamPct = totalPast > 0 ? Math.round((totalGoing / totalPast) * 100) : 0`.

`goingCount` is incremented in `useAttendanceData.js:111` only when:
```
arrival === 'arrived' || checkedIn || rsvp === 'going'
```

So for the 11U Girls team in Spring 2026 with check-ins not actively run AND parents not actively pressing Going buttons, `goingCount` floors at near-zero — but `totalPast` increments on every past event the player was eligible for (line 110). The denominator stays large; the numerator stays near-zero; the headline reads "2% Going" or similar.

The bug class is the same one Cluster 2 / PR #240 closed at the player-row scope. That fix added the "Going" suffix to the row pills so `goingCount=1` reads "1 Going" not "100%." This fix needs to extend up to the team-level pulse header.

Lines 35–41: `pulseTitle` constructs the header. Comment block at lines 35–38 acknowledges the ambiguity from the prior round of feedback. But the fix landed at the LABEL level, not the LOGIC level — "2% Going" reads honestly about WHAT the percent measures (per the comment) but doesn't acknowledge the metric is noise when check-ins haven't run.

Line 43: `defaultOpen={false}` — correct per §16.14.
Lines 71–90: grid render. Note line 137 of `useAttendanceData.js` sorts the grid by `pct desc` — explaining V4.

2.4 PlayerRow.jsx — sparse-RSVP gating, role-gating gaps
────────────────────────────────────────────────────────

Line 16: `const showRsvp = player.totalPast > 0 && (role !== 'parent' || isMyChild)`.

This SHOULD gate the entire pill block — but doesn't. Reading further:

Line 23–24:
```
const responsesReceived = (player.goingCount || 0) + (player.maybeCount || 0) + (player.declinedCount || 0);
const sparseRsvp = responsesReceived <= 1;
```

Lines 61–65 (the "No RSVPs yet" pill):
```
{showRsvp && sparseRsvp && (
  <div ...>
    <span ...>No RSVPs yet</span>
  </div>
)}
```

So the diagnostic shows when `showRsvp && sparseRsvp` — `showRsvp` for a parent is `role !== 'parent' || isMyChild`. Wait, that means parents SHOULD only see the pill on their own kid's row. Let me re-read.

`(role !== 'parent' || isMyChild)`:
- If role is admin/coach: `true || X` → true (shows for all).
- If role is parent + `isMyChild === true`: `false || true` → true (shows only for own kid).
- If role is parent + `isMyChild === false`: `false || false` → false (doesn't show for peers).

So the gating IS correct in code. But screenshots 2–3 show admin view where everyone sees the diagnostic — that's correct admin behavior. The prior B2 claim was "diagnostic bleeds to parent view from peer rows" — re-reading the code, that's NOT what's happening today. Either the bug was fixed silently after the prior audit (line 16 was changed) or the prior audit was incorrect.

**REVISION TO B2:** The diagnostic correctly hides from parents on peer rows. The actual issue is different — for admin/coach, EVERY ROW shows "No RSVPs yet" when the team hasn't engaged with RSVPs yet. That's not a privacy leak; it's a wall of negative-state pills that bury actionable signal under repetition. The fix is different: cluster the diagnostic into a single team-level "11 of 13 families haven't RSVP'd for this week" call-out at the Roster header level, not 13 per-row pills.

This is a meaningful audit revision. The bug exists, but its scope is admin-noise, not parent-privacy.

Other PlayerRow notes:
- Line 14: `age` derived from `dob` using `NOW = Date.now()` at module top (line 6). NOW is captured at import time, not per-render — birthday-crossing fragility (a kid turns 11 on June 1; the page still shows "10y" until the module re-imports). Minor.
- Line 47–54: payment_status dot — admin/coach only. Per §11.5 ground-truth table, `payment_status` lives in `financial_accounts` now, not roster_members. This is the legacy column being read. Schema migration backlog.

**D1. `useAttendanceData.js:54` queries `players` table by `.in('id', playerIds)` with NO `.eq('org_id', orgId)`.**

**D2. Verify `usePrograms` and `useTeamRecords` org_id scoping.**
Quick check needed in PR-pre-flight: both queries should start with `.eq('org_id', orgId)` even though `teams` and `game_results` are org-scoped. 3.3 Refetch behavior
────────────────────────────────────────────────────────

`TeamDetailPage.jsx:35`: `useRefetchOnVisible(rosterRefetch)` — refetches roster on tab visibility change. Good for staleness.

────────────────────────────────────────────────────────
3.4 N+1 / inefficient query risks
────────────────────────────────────────────────────────

`useAttendanceData` fires 4 sequential awaits (line 43, 53–54, 62–66, then state set). The Promise.all collapses 2 batches:
- Batch 1: events + roster_members (parallel)
- Batch 2: players (await sequential after batch 1)
- Batch 3: rsvps + arrivals + check_ins + activations (parallel after batch 2)

That's 3 round-trips, not 7. Acceptable. Could be improved to 2 if the players query merged into batch 3 (no dependency on player_ids for batch 3 — they only use event_ids).

`TeamHeaderCard` reads from `summary` prop — no query.

No N+1 risks observed.

────────────────────────────────────────────────────────
3.5 RLS holes / cross-org leak risk
────────────────────────────────────────────────────────

Quick mental audit:
- An admin navigates to `/teams/<another-org's-team-id>` directly via URL.
- `usePrograms()` returns their own org's teams.
- `programs.find(p => p.id === teamId)` returns undefined.
- Line 65: renders EmptyState "Team not found." ✅ Correct fallback.

But: `useAttendanceData(teamId)` STILL FIRES against the other org's teamId because the hook doesn't check programs first. The `events` query has RLS on team_id → teams → org_id, so it returns []. Same for roster_members. So the queries are safe-by-RLS but waste round-trips. Defense-in-depth: gate the hook fires on `team` truthiness, not just `teamId`.

PART 5 — ACCESSIBILITY AUDIT
═══════════════════════════════════════════════════════════════════════════

Per CLAUDE.md §16.4 — accessibility is table stakes.

**A1. Tap target sizes:** ✅ All tappable surfaces hit 44px min — back button (line 69), TeamSwitcher chip, PlayerRow (56px min), CoachQuickActions tiles (44px), MessageTeamFAB (56px), Send briefing chip (44px). Sort chips inside RosterControls likely 44px (not read but standard pattern).

**A2. aria-labels:** Mostly present. MessageTeamFAB has `aria-label="Message team"` ✅. Back button has aria via `<ChevronLeft aria-hidden="true">` + visible text. **MISSING: aria-label on the page itself or H1 landmark for "Team detail for X."** The H1 reads "11U Girls" — fine semantically but a screen reader user landing here doesn't know they're on a team detail page (vs the teams list). Suggest aria-label on the wrapper div or document.title sync.

**A3. Tab order:** Visual scan suggests OK — back → switcher → header → send-briefing chip → coach actions → upcoming → search → sort chips → roster rows → pulse → player stats → FAB. The FAB at position:fixed; in tab order it lands LAST per DOM order — but visually it floats above content the user already passed. Screen reader user tabbing in DOM order hits the FAB after Player Stats — which is fine functionally but the FAB has no "skip to FAB" affordance and a keyboard user has to tab through 13 player rows to reach it.

**A4. Color contrast:** Sample check on `var(--em-text-tertiary)` (#8896AB) against `var(--em-bg-page)` (#F7F8FA) — that's ~3.4:1 contrast. Per WCAG AA, body text needs 4.5:1. **Tertiary text fails AA on this page background.** Used for sort labels ("UPCOMING (NEXT 7 DAYS)" header), meta lines, "5th" grade pills. Borderline but cumulatively significant. The same tokens elsewhere may render against `var(--em-bg-card)` (#FFFFFF) where contrast is ~4.0:1 — still below AA.

**A5. prefers-reduced-motion:** `sf-press`, `sf-fade-in`, `sf-stagger-N` animations are defined in `index.css` per CLAUDE.md §7 with `prefers-reduced-motion` guard. ✅

**A6. Focus visible:** No explicit `:focus-visible` styling observed on any tap target. Touch-screen primary surface so low-priority, but keyboard users get default browser ring which is inconsistent with the design language.

**A7. Screen reader for Pulse grid:** The grid uses CSS grid + `display: contents` (line 72). Cells have no aria-label — a screen reader reads "blank" for empty cells, "✕" for no_show, etc. without context. Better: each cell could carry `aria-label="May 14, no response"` or wrap the grid in a `<table>` semantic.

PART 6 — PERFORMANCE AUDIT
═══════════════════════════════════════════════════════════════════════════

**P1. Re-render triggers.** TeamDetailPage `useNow()` ticks every minute. That re-evaluates `nextEvent` memo (correct) AND re-renders the entire page tree because `now` is in the closure. PlayerRow's `NOW = Date.now()` is module-level — doesn't re-render, but doesn't update age either.

**P2. enrichedPlayers memo.** Re-runs whenever `players` or `grid` change. `grid` re-computes whenever useAttendanceData's inputs change OR useNow ticks. So enrichedPlayers re-runs every minute. The reduce is cheap (13 players) — no concern.

**P3. Pulse grid render cost.** 13 players × ~6 events visible = 78 cells per pass. Each cell renders a div with 2 lookups. Cheap. The horizontal scroll for season range (180d / ~26 events) bumps to 13×26 = 338 cells — still cheap.

**P4. Bundle.** All 12 components on this page are tree-shaken individually — no lazy import. TeamPlayerStats imports PlayerStatsTable from records, which pulls in a stats fixture set. The Roster section depends on PlayerRow + GuardianRow which depends on InviteButton. Total subtree from this page is moderate but unmeasured. Per CLAUDE.md §16.10 (350KB total compressed budget), worth a `npm run build` baseline. None of this page should code-split — it's a primary route — but a route-split between /teams (list) and /teams/:id (detail) might shed 30–50KB of FAB / Achievements / pulse code from the list-view's first paint.

**P5. Empty-state Player Stats permanent dead space.** For Legacy Hoopers (no live-scoring), the Player Stats section renders for staff with permanent "no live-scored games yet" text. It costs DOM + render + the empty-state subtree. Should gate on org-level "live-scoring active" rather than just role.

13.3 useTeamRecords violates anti-pattern #48 (NEW finding C5)
──────────────────────────────────────────
`src/hooks/useTeamRecords.js:47`:
```

```

This is the exact bug class anti-pattern #48 documents — PostgREST `.order` with `foreignTable` does NOT sort top-level parent rows by the joined table's column. The query returns `game_results` rows; ordering them by `events.start_at` via the foreignTable hint is a no-op for the parent ordering.

**Mitigation already in place:** `computeSummary` in `src/lib/teamRecords.js:38-42` re-sorts the games array in JS:
```
const sorted = [...games].sort((a, b) => {
  const da = a.event?.start_at || a.start_at || '';
  const db = b.event?.start_at || b.start_at || '';
  return da < db ? -1 : da > db ? 1 : 0;
});
```

So the bug is masked at the consumer level — but the `.order` call on line 47 is dead/misleading code. A future reader (or a future hook copying this pattern without copying computeSummary's re-sort) inherits the same bug. PR D should remove the `.order` line OR replace with a comment explaining the JS-side re-sort.
