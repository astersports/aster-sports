2. THE GAP (current design vs. operator vision)
====================================================================
The vision is "paste a LINK → AUTO real-time updates across 3 sites." The shipped design
is "paste TEXT → operator-reviewed commit, schedule only, one surface (the app)."
Three concrete gaps:
  G1. No automated fetch. tourney_url is stored but never fetched. The paste-only
      decision (1.2) was made because TM 403's WebFetch.
  G2. No scraper feeds ingest-game-results — scores never flow automatically; the writer
      is built-but-unwired.
  G3. No shared consumption. AAU + Legacy run their own scrapers; they do not read the
      platform hub, so "one entry → three sites" is not in place.

KEY OPENING [VERIFY]: the two public scrapers fetch tourneymachine.com directly with a
browser User-Agent and appear to operate in production. The architect's 403 was against
"WebFetch" (Claude's tool / a bare fetch). A server-side fetch with a real UA + headers
from an edge function or a Railway cron MAY succeed where WebFetch failed. If it does,
the paste-only constraint is liftable for TM, and ingest-game-results can be wired to a
real scraper. This must be settled by a spike before any build — not assumed either way.

3. DECISIONS REQUESTED
====================================================================

--------------------------------------------------------------------
D1 — Confirm the hub: platform = single source of truth; AAU + Legacy = thin read
     consumers (retire their independent scrapers).
--------------------------------------------------------------------
EVIDENCE: §1.1 (platform already has events + game_results + tournament tables + public
  read gates) vs §1.3 (two sites duplicate-scrape, persist nothing).
OPTIONS:
  (a) Platform is the hub. AAU + Legacy read platform public RPCs; their scrapers are
      deleted. REQUIRES: a public read path for upcoming/live tournament games (see D5);
      re-pointing both sites (Legacy is being rebuilt anyway; AAU swaps tRPC scraper for
      a platform read).
  (b) Status quo. Each site keeps its own scraper. REQUIRES: nothing, but the operator's
      "one entry → three sites" never happens and three codebases drift.
CC LEAN: (a). It is the charter's own model (platform = source of truth; public sites are
  thin consumers) and the only option that satisfies the vision.

--------------------------------------------------------------------
D2 — Automation fork: keep paste-only, or wire an automated scraper feeding the existing
     ingest-game-results (+ schedule), enabling "link → auto." Reopens the paste-only DR.
--------------------------------------------------------------------
EVIDENCE: §1.2 paste-only decision; §2 G1/G2 + the KEY OPENING [VERIFY]; ingest-game-
  results already accepts exactly the games[] payload a scraper would produce.
OPTIONS:
  (a) Spike-then-wire. First a feasibility spike: can a server-side UA fetch read TM
      Division pages reliably (and can setourney.app be read at all)? If yes for a source,
      build a scheduled scraper (edge fn or Railway cron) that fetches → parses (port the
      proven astersports-web parser) → POSTs games[] to ingest-game-results, and writes
      schedule rows the same controlled way. Keep PASTE as the fallback for sources that
      403/SPA. REQUIRES: the spike; a scraper component; cron; ingest_secret already set?
      (verify app_secrets.ingest_secret populated).
  (b) Keep paste-only. Operator pastes text per tournament; no auto-fetch. REQUIRES:
      nothing new, but it is not "dynamic/real-time" and not "paste a link."
  (c) Hybrid by source. Auto-fetch where feasible (likely TM/Zero Gravity); paste for the
      rest (likely SE Tourney until its feasibility is known). REQUIRES: same as (a) but
      scoped to the proven source first.
CC LEAN: (c), starting with the spike from (a). Wire TM auto-ingest if the spike passes
  (it is the source both existing scrapers already target); keep paste as the universal
  fallback; treat SE Tourney as its own feasibility question. This honors "one writer per
  concern" (the scraper feeds the existing writers) and only revisits paste-only for
  sources we can actually fetch.

--------------------------------------------------------------------
D3 — Upload point ("AAU as the main hub to upload links"): where does the operator enter
     the link, and where is it stored?
--------------------------------------------------------------------
EVIDENCE: tournaments.tourney_url already editable in the app
  (TournamentFormSheet.jsx:65, "SE Tourney link"); AAU page lives in a DIFFERENT app
  (astersports-web) on a DIFFERENT Supabase.
OPTIONS:
  (a) App admin is the single write surface (tourney_url already there + /admin/import-
      schedule). AAU + Legacy are READ-ONLY surfaces. REQUIRES: nothing new for storage;
      optionally surface a "manage tournaments" deep-link from the AAU admin.
  (b) Add an upload control on the AAU page that writes into the platform. REQUIRES: a
      cross-app authenticated write from astersports-web → aster-sports (RPC or shared
      service); more surface area, second auth path.
  (c) Both.
CC LEAN: (a). The link field + import flow already live in the app where org auth + admin
  roles exist. "AAU as hub" is best read as the AAU surface being the showcase that
  displays the distributed data; the WRITE stays in the app to avoid a second cross-app
  auth path. Confirm this interpretation.

--------------------------------------------------------------------
D4 — Live standings: populate tournament_pool_teams (during-tournament W/L + pool place)?
--------------------------------------------------------------------
EVIDENCE: table exists, 0 rows (§1.1). The public sites today render live pool standings
  from their scrape; the platform has the table but no writer.
OPTIONS:
  (a) In scope for the scraper (D2): the same fetch that yields games[] also yields pool
      standings → write tournament_pool_teams. REQUIRES: scraper parses standings; a
      public read for it (D5).
  (b) Defer to a phase 2; ship game_results-based records first.
CC LEAN: (b) defer. Records (final W/L) are the higher-value, already-modeled surface;
  live pool standings ride the same scraper later once D2 is proven.

--------------------------------------------------------------------
D5 — Public read exposure for UPCOMING/LIVE tournament games (not just played results).
--------------------------------------------------------------------
EVIDENCE: get_public_team_records returns played+published games only; upcoming games
  live in events (schedule). The AAU/Legacy "this weekend / live" sections need upcoming.
OPTIONS:
  (a) Add get_public_team_schedule(p_team_id) (SECDEF, org_is_public_listed-gated, returns
      upcoming events + tournament context), mirroring the existing public RPC pattern.
      REQUIRES: one migration + grants; architect-scoped (schema).
  (b) Reuse existing RLS public table reads (events_select_public) directly from the sites.
      REQUIRES: confirm events_select_public exposes the needed columns to anon.
CC LEAN: (a) a dedicated RPC — same proven pattern as get_public_team_records, keeps the
  public contract explicit and the sites thin. (AP #40: prefer the dedicated contract.)
