FINDING 1 — Dead per-send DB IO: fetchSignatureCoaches result is
            computed then discarded in 4 resolvers.
Severity: P3 (correctness OK — no leak; cost is wasted IO + stale,
          misleading comments).
Files:
  src/lib/engine/resolvers/gameRecap.js:76,94
  src/lib/engine/resolvers/gamesRecap.js:68,89
  src/lib/engine/resolvers/weeklyDigest.js:130,134
  src/lib/engine/resolvers/tournamentRecap.js:114-124
  (consumed via src/lib/engine/buildOrgContext.js:37 → org.signature_coaches)
What's wrong: These 4 resolvers call fetchSignatureCoaches(...) and pass
  the result into buildOrgContext as `signature_coaches`, which lands on
  context.org.signature_coaches. But NO compose function reads
  org.signature_coaches — every signoff/signature now flows through
  buildSignoffSection({ overrides }) which reads ONLY the admin-picked
  overrides.signoff_coaches (the toggle). Grep confirms zero live readers
  of org.signature_coaches (only tournamentRecap.js builds a
  signature_coaches_by_team that is likewise never read by its compose).
  So each game_recap / games_recap / weekly_digest / tournament_recap
  send fires an extra staff_profiles+team_staff query whose result is
  thrown away. The resolver comments ("The voice signature is the UNION
  …", "org Program Director + team coaches") are now stale — they
  describe the pre-toggle auto-signature model that the 2026-06-09
  explicit-pick model replaced.
Proposed fix: Remove the fetchSignatureCoaches calls + the
  signature_coaches plumbing from buildOrgContext (and the dead
  signature_coaches_by_team in tournamentRecap.js), OR, if a future
  auto-signature default is intended, wire org.signature_coaches into
  buildSignoffSection behind the toggle. Until then it is dead IO. Also
  delete signatureCoaches.js + voiceSignature's team-aware comments if
  the plumbing goes. Ship with the AP#27-pure compose tests unchanged
  (they don't read the field, so removal is safe). NOTE: confirm no
  test asserts org.signature_coaches before removing.

FINDING 2 — coach_roundup / family_guide send-gate relies on body
            auto-select; the per-body validate() is not enforced at the
            send gate, and countByAudienceType returns 1 unconditionally.
Severity: P3 (well-mitigated; worst case is a caught friendly-error
          toast, never a silent bad send).
Files:
  src/lib/briefings/audience.js:38 (coach_self/family_specific → 1)
  src/components/briefings/StepSendConfirm.jsx:56 (only games_recap has a
    kind-specific empty-selection gate)
  src/components/briefings/bodies/CoachRoundupBody.jsx:23-29 (validate exported, unused by gate)
  src/components/briefings/bodies/FamilyGuideBody.jsx:24-30 (validate exported, unused by gate)
What's wrong: For coach_roundup and family_guide, countByAudienceType
  returns a hard 1 regardless of whether audience_filter.coach_user_id /
  parent_user_id is actually set. The send gate therefore shows "1
  family" and ENABLES Send even if the picker is empty. The bodies
  auto-select (current user or first staff / first guardian) on mount,
  so in practice the id is set; but if staff/guardian load fails or the
  admin clears the select, Send is enabled and the resolver throws
  ("Missing coachUserId" / "Missing parentUserId") at send time. That
  error is caught and shown via translateBriefingError, so it's a
  friendly failure — but it's a gate hole vs. the games_recap pattern,
  which DOES hard-gate its empty selection.
Proposed fix: Mirror the gamesRecapEmpty pattern in StepSendConfirm —
  add `coachRoundupEmpty = state.kind==='coach_roundup' &&
  !state.audience_filter?.coach_user_id` and the family_guide
  equivalent, with actionable notes ("Pick a coach/parent above before
  sending."), folded into `disabled`. (Optional, lower priority since
  resolver-throw + toast already prevents a bad send.)
