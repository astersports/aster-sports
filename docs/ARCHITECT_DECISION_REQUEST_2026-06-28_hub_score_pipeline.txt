0. WHAT FRANK DESCRIBED (operator intent, verbatim shape)
=====================================================================================
- The HUB is the central tournament loader. It loads tournaments from the
  tournament site (TourneyMachine / Zero Gravity / CYO), then "triggers the tenant
  in the app" to update the games and subsequent scores.
- It ALSO surfaces on the public Aster Sports AAU website.
- Trigger condition: a tournament is loaded/uploaded in the Hub AND a tenant org
  has a team in that tournament (Zero Gravity AAU, or "any tournament loaded on the
  tournament site and uploaded in the hub").
- MULTI-COMPETITION + MULTI-TENANT. Example given: Legacy Hoopers = AAU (Zero
  Gravity) today; St Patrick winter season = ALL League Play (CYO), ~25 teams.

So: one Hub load fans out to whichever tenant org(s) have teams in that tournament,
across both AAU and CYO/league-play.

2. DECISION FORKS (each: REQUEST · EVIDENCE · OPTIONS · CC LEAN per AP#39)
=====================================================================================

FORK 1 — Load ownership: who is the single "front door" for loading a tournament?
  EVIDENCE: Hub `aau-submit-tournament` (paste URL) and app `aau-scrape-tournament`
    both scrape TourneyMachine into the SAME DB; the cron also refreshes. Two human
    entry points, one DB.
  OPTIONS:
    A. Hub `aau-submit-tournament` becomes the SOLE human load front door; the app's
       ingest edge fns stay as the shared WRITE layer (no UI of their own).
    B. Keep both entry points (Hub paste + app admin scrape).
  CC LEAN: A. One front door (the Hub) for the human "load a tournament" action;
    everything writes through the existing ingest functions. Less drift, one mental
    model. The app keeps the cron for live refresh.

FORK 2 — Fan-out trigger: how does a load reach every tenant with a team in it?
  EVIDENCE: intra-DB (shared Supabase), so no cross-project auth needed. Existing
    ingest is per-(org_id) and idempotent on (org_id, external_game_id).
  OPTIONS:
    A. Extend the live-score-poll cron + the load edge fn to iterate
       feed_source_mapping rows for the loaded tournament and run ingest per mapped
       org. (Reuses everything; intra-DB.)
    B. A Postgres trigger on division_games insert/update enqueues per-mapped-org
       ingest jobs.
    C. The load edge fn directly calls ingest-game-results once per mapped org.
  CC LEAN: A. Drive the fan-out off feed_source_mapping in the cron + the load fn —
    minimal new surface, reuses the idempotent per-org ingest, no new auth. (B's DB
    trigger calling edge fns adds a fragile in-DB→HTTP hop per AP#32.)

FORK 3 — Tenant-team ↔ tournament-team mapping (the join that drives fan-out):
  EVIDENCE: feed_source_mapping exists but is manual; Hub tracked_teams has no org
    link; externalTeamKey is the TM team identity.
  OPTIONS:
    A. Admin explicitly links their org-team to a tournament division-team at load
       time (reliable, manual).
    B. Auto-match by team name / externalTeamKey (zero-touch, fragile — name drift).
    C. Hybrid: auto-SUGGEST by name, admin CONFIRMS, persist in feed_source_mapping.
  CC LEAN: C. Auto-suggest + admin-confirm, persisted. HONEST-CLAIMS GUARD: never
    auto-ingest scores against an UNCONFIRMED mapping (a wrong auto-match would write
    a wrong score to a family-facing record). Confirmed mappings then fan out
    automatically on every refresh.

FORK 4 — Competition generality: AAU (TourneyMachine) vs CYO / league-play.
  EVIDENCE: entire pipeline is TourneyMachine-shaped (tm_id_tournament,
    external_game_id `tm:…`, the scraper). CYO/league-play (St Patrick, ~25 teams) is
    NOT wired and its DATA SOURCE is unknown to CC.
  OPTIONS:
    A. CYO is also on TourneyMachine → reuse the existing scraper as-is.
    B. CYO is a different site/feed → a new source adapter behind one normalized
       "loaded tournament/game" contract that ingest consumes.
    C. CYO entered via the existing paste → Claude schedule-parse path (semi-manual),
       scores entered/scraped separately.
  CC LEAN: RESOLVED 2026-06-28 (Frank) → Option A (reuse). CYO IS on TourneyMachine,
    same load path as AAU; no new source adapter. [GROUNDED] the spring CYO tournament
    is already loaded: tournaments row "WPCYO Spring League 2026"
    (id 254afad0-23af-4979-ac4a-88614b76e341, circuit league_play, 2026-03-01..06-30,
    schedule_status=final). So AAU + CYO are ONE source (TourneyMachine); the pipeline
    need not be source-agnostic for v1 — it can stay TM-shaped.
    CONCRETE GAP [GROUNDED]: that CYO row AND all 8 loaded tournaments currently have
    tm_id_tournament = NULL (they were loaded via the manual paste/Claude schedule path,
    not the TM scrape-ingest that stamps tm_id_tournament). So the first concrete step to
    enable auto fan-out is BACKFILLING tm_id_tournament (the TM link) onto the existing
    tournament rows — then the live-score-poll cron + ingest can act on them.

FORK 5 — Unify schedule + scores into one "load → games AND scores" flow.
  EVIDENCE: today events (schedule) come from the manual Claude parse path; scores
    match to pre-existing events (±3h). If a tournament is loaded but the tenant's
    events don't exist yet, scores have nothing to match → review-flag limbo.
  OPTIONS:
    A. Fan-out FIRST creates/updates the tenant's events from the tournament schedule,
       THEN ingests scores (which now match). One flow.
    B. Keep schedule manual; only scores fan out (status quo gap remains).
  CC LEAN: A. The fan-out must create/update the tenant's events (schedule) before
    score ingest, or scores strand. This is the substantive new build.
