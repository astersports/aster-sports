3. SUBSTRATE YOU MUST DESIGN WITHIN (condensed; full depth in the audit)
--------------------------------------------------------------------------------
DATA SHAPE
  • Roster membership = team_players (canonical: roster_type rostered|futures,
    jersey_number, status active|inactive) mirrored to roster_members by a
    trigger. To REMOVE: set team_players.status='inactive' FIRST, then drop the
    mirror — order matters (trigger enforced). "Delete never-accepted" can be a
    hard delete since no money/registration is attached.
  • Money = family-keyed, NOT player-keyed. One financial_accounts row per
    (guardian, season). An append-only financial_transactions ledger (types:
    payment / refund / adjustment / fee). family_balances VIEW is the ONE
    canonical balance source (balance = billed − net_paid; negative = credit).
  • financial_transactions has an UNUSED applied_to_player_id column — so a
    per-player action can be ATTRIBUTED to a kid while the money still lives on
    the family account. This is the seam that makes per-player actions cheap.
  • Deposit/installment schema EXISTS and is dormant: payment_plans +
    payment_plan_installments (per-seq due_date + amount_cents) + invoices (with
    due_date). Zero rows anywhere. So "$100 now, $400 due June 22" is
    representable today with no migration — it just has no UI.
  • Credit today = a positive 'adjustment' transaction (no stored credit pool).

WHAT'S REUSABLE (design to reuse, not reinvent)
  • RecordPaymentForm (FullScreenForm) already does payment / refund / adjustment
    with a live "balance after" preview — it's family-scoped on the Financials
    page. The redesign wants this same form REACHABLE FROM A ROSTER ROW with a
    player in context (writing applied_to_player_id).
  • AddChargeSheet (BottomSheet) already adds a 'fee'.
  • Void = append a reversing row (already built on the family page).
  • The Team Detail page is already a hero + collapsibles surface (§16.14).

GROUND TRUTH (live, 2026-06-19)
  • Summer Select 2026: 24 players rostered across "6th Grade Girls" (12U F, 14)
    and "5th Grade Boys" (11U M, 10). ZERO financial rows / registrations for the
    season — a clean slate. Frank's payment states are states he must ENTER.
  • Org-wide, only 'fee' + 'payment' transactions have ever been written; refund
    and adjustment are first-use. Money method has always been 'other'.
  • STALE BUG to fix in-flight: TeamDetailPage hardcodes a "Spring 2026" scope
    tag — wrong now. Should derive from the active season.

4. DECISIONS TO LOCK (render against your ruling; CC lean labeled)
--------------------------------------------------------------------------------
FORK-A — PER-PLAYER MONEY ATTRIBUTION
  A1 (CC LEAN): keep family billing; stamp applied_to_player_id on each txn so a
     roster row shows/records "Charlie · refund $50". Reuses every money hook.
  A2: new player-scoped accounts. Cleaner per-kid reporting; breaks reuse; costly.
  → Render implication: if A1, the money sheet shows "Player (family account: X)"
     and a per-player line within a family ledger; if A2, a standalone player
     ledger. Please rule.

FORK-B — ROSTER-EDIT SURFACE + "DELETE" SEMANTICS  ★ highest design value
  Where do add / remove / jersey / move live? Options to render one of:
    B1 a dedicated "Manage roster" admin screen (own route),
    B2 an edit-mode toggle on the Team Detail roster section (inline row actions),
    B3 row actions in an overflow per PlayerRow.
  AND: is "remove" one verb or two? CC LEAN: TWO — "Remove" = inactivate (keeps
     history, for a real player leaving) vs "Delete" = hard-remove a
     never-accepted port (no history needed). Render both as distinct,
     differently-weighted actions (delete = destructive/confirm).
  → CC LEAN on surface: B2 (edit-mode on the existing roster collapsible) keeps
     it in context and avoids a new route, but B1 is cleaner for bulk cleanup —
     YOUR call; this is the core UX decision.

FORK-C — DEPOSIT / PLAN DEPTH ($100 now, $400 due June 22)
  C1 (thin): model as 'fee' $500 + 'payment' $100; "$400 due" = residual balance;
     the date lives in a note. No dated reminder. Cheapest.
  C2 (full): use the dormant payment_plans + installments (seq1 $100 paid, seq2
     $400 due 2026-06-22) or an invoice.due_date; gives a real dated schedule +
     a parent-facing timeline + (optional) reminders. No migration needed.
  → CC LEAN: C1 now / C2 fast-follow UNLESS Frank wants the dated "due June 22"
     reminder to actually fire this build. Either way the deposit-vs-full CHOICE
     needs a capture control on the money sheet — please render that control.

FORK-D — CREDIT REPRESENTATION
  D1 (CC LEAN): "issue credit" = positive 'adjustment'; negative balance = credit
     available; reuses the form. D2: a real credit pool (new table) + apply-credit
     flow. → Render implication: D1 = one "Issue credit" mode in the money sheet;
     D2 = a separate credit-balance card + "apply" affordance.

FORK-E — ACTIONS AS RPCs
  CC will build the 4 roster-facing actions as audited SECDEF RPCs
  (drop_roster_member, record_payment, issue_refund/adjust, void) regardless of
  your render — noting it so you know each action is atomic + can show an
  optimistic state then confirm. No render impact; just confirm you're aligned.

6. LOCKED DESIGN SYSTEM (renders MUST conform — CC builds to var(--as-*))
--------------------------------------------------------------------------------
TOKENS (use these names; no hardcoded hex except team_color inline from DB):
  surfaces/text: --as-bg-page #F7F8FA · --as-bg-card #FFF · --as-bg-card-hover ·
    --as-bg-secondary · --as-bg-tertiary · --as-text-primary #1A1D23 ·
    --as-text-secondary #4A5568 · --as-text-tertiary #8896AB · --as-text-inverse ·
    --as-border-default #E2E8F0 · --as-border-subtle.
  status: --as-success #16A34A (+soft) · --as-warning #D97706 (+soft) ·
    --as-danger #DC2626 (+soft) · --as-info #3B82F6 (+soft) · --as-academy #7C3AED.
  money/achievement: gold = ACHIEVEMENT ONLY (never money). Money is plain text.
  brand (Legacy Hoopers runtime override): accent = cobalt #4a8fd4 (--as-accent),
    --as-accent-hover #5BA0E0, --as-header #4a8fd4. Use var(--as-accent), not hex.
  shadows: only --as-shadow-sm / -md / -lg. radius: 6 badge / 10 card+input /
    16 modal / 9999 avatar.
COMPONENTS:
  • Tap target ≥44px. Inputs 44px, bg --as-bg-tertiary, 1.5px border, 10px radius,
    accent focus ring.
  • Cards: --as-bg-card, --as-border-default, 10px, --as-shadow-sm.
  • Buttons: primary (accent bg, inverse text) / secondary (accent border) /
    destructive (danger bg) / ghost. Press = scale(0.97)+vibrate(50).
  • Modal backdrop rgba(0,0,0,0.3) — never black/50.
  • Icons: Lucide only, 16/20/24, stroke 1.75. Avatars 24/32/40/56.
  • Badges: 6px radius, 11px, weight 500.
FORM ROUTING (hard rule, AP#15): 3+ fields → FullScreenForm; 1–2 actions →
  BottomSheet. (So: the money sheet + add-player = FullScreenForm; a single
  remove-confirm = ConfirmDialog/BottomSheet.)
DETAIL PAGE (§16.14): hero card + collapsible sections; section headers show a
  summary so state reads without expanding; destructive/secondary actions live in
  an overflow, NOT as floating chrome.
TYPOGRAPHY: Inter 24/20/17/15/13/11; event time 17 bold; labels 11 uppercase
  tracking-wide.
ELITE (§16): optimistic UI on per-row writes (pessimistic on cascading/bulk);
  kindness microcopy (e.g. rollback = "Looks like that didn't go through. Try
  again?"); accessibility (aria-label on every control, live region on state
  change, ≥4.5:1 text contrast, keyboard nav); density-aware where lists are long.

--------------------------------------------------------------------------------

    design a surface that implies a second, separately-computed balance.
  • Per-role hero contract: parent (View calendar + coach contact) / coach-not-
    self (Message+Briefing) / coach-self (contact hidden) / admin
    (Message+Briefing+contact). Keep it.
  • Payment status one source; if you show an amount AND a dot, both derive from
    the same balance + LABEL the scope (all-seasons family vs this-season).
  • Coaches don't see money actions/amounts (privacy lean) — confirm or override.
  • 150-LOC per component (drives the PlayerRow split in S7).
