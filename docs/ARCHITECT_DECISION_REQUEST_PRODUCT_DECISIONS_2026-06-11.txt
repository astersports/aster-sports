D-a — RECORD-MONEY ENTRY POINT FOR PAID-IN-FULL FAMILIES
------------------------------------------------------------------------------
REQUEST: Should the admin be able to open the record-money form (payment /
refund / adjustment) for a family whose balance is $0 / paid-in-full?

EVIDENCE:
- src/components/admin/FamilyBalanceList.jsx:76 — the row's onClick gates on
  owing state: `onClick={() => f.balance > 0 && onRecordPayment(f)}`; :81 cursor
  is pointer only when owing; :100 the "Tap to record" hint renders only when
  owing. Paid rows are inert.
- src/pages/FinancialDashboardPage.jsx:105 — the ONLY mount of RecordPaymentForm
  in the app (grep: 2 files total, the form + this mount). No other entry point.
- The form itself (src/components/admin/RecordPaymentForm.jsx, F-4 / PR #978)
  already supports payment / refund / adjustment(credit|charge) with
  mode-specific guards — the capability exists; only the entry point gates it.

OPTIONS + REQUIREMENTS:
1. Open for ALL families — drop the balance>0 gate; row affordance becomes
   "Tap to record" for everyone; a $0-balance row opens the form with refund/
   adjustment as the sensible default. REQUIREMENTS: one-line gate change +
   affordance copy + default-mode tweak + test update. Tiny PR.
2. Keep the gate; add a secondary entry point (per-family overflow action or a
   future FamilyDetail surface). REQUIREMENTS: a new UI surface/mount — larger
   lift; no existing page naturally hosts it today.
3. Keep as-is. REQUIREMENTS: none; refunds/adjustments for paid families stay
   unrecordable in-app (operator falls back to SQL when one happens).

CC LEAN: Option 1. Refunds happen precisely to paid-in-full families — the
current gate blocks the form exactly when the refund workflow needs it. The
form's mode guards already prevent nonsense entries; no new surface needed.

D-b — "ADD ANOTHER CHILD" DROPS TO MANUAL ENTRY, NOT THE SELECT SCREEN
------------------------------------------------------------------------------
REQUEST: After an authed parent registers child #1 via the select screen, "Add
another child" (children-roster hub) appends a blank MANUAL child (re-type).
Wire it back to the select screen now, or carry as a trigger?

EVIDENCE:
- src/components/register/StepChildrenRoster.jsx:28 — "+ Add another child"
  calls onAdd, which appends an empty child to the client children[] array
  (manual entry; per the file's header comment, lines 5-7).
- The select step exists only at funnel entry: src/pages/RegisterFlowPage.jsx:72-83
  (B3 funnel; SELECT_CHILDREN dispatch at :83).
- Safety today: the (program, player_id) dedupe guard in submit_registration
  rejects a re-typed duplicate ("already registered") — the double-reg hole
  stays closed either way. The cost is parent re-typing + a late-surfacing
  rejection if they re-add an already-registered kid.

OPTIONS + REQUIREMENTS:
1. Wire "Add another child" to re-enter StepSelectChildren when authed and
   unregistered eligible kids remain (already-picked kids excluded).
   REQUIREMENTS: a reducer edge + RegisterFlowPage routing + tests. Small-medium;
   touches the registration state machine.
2. Keep manual; register as a TRIGGER for the next funnel-surface PR (the
   audit's own framing: "worth wiring back later (trigger)").

CC LEAN: Option 2 — trigger now, build on next funnel touch. The safety
property is intact; the UX cost is small today (one open single-division
program; multi-child families mostly register in one pass via multi-select).

D-e — TEAMMATE AGE (DOB-DERIVED) VISIBLE ON PLAYER PROFILE
------------------------------------------------------------------------------
REQUEST: Any parent on a team can open a teammate's PlayerProfilePage and see
that child's age derived from dob. Confirm as intended, or restrict to
own-child viewers?

EVIDENCE:
- src/pages/PlayerProfilePage.jsx:24 — selects `dob` for any player the viewer
  can read; :37 derives age from dob; :59 renders the "{age}y" chip.
- Reachability: parents read teammate rows BY DESIGN via
  current_user_teammate_player_ids() (§11.5 ground-truth helpers) — row-level
  access is intended; the open question is the UI surfacing of dob-derived age
  (and the dob column read itself) to non-own parents.
- §16.7 privacy-locks precedent: derived personal data (streaks, attendance
  trending) is own-child-only for the parent role.

OPTIONS + REQUIREMENTS:
1. Gate the age chip (and the dob select) to own-child viewers + coach/admin;
   teammate view shows name/grade/team only. REQUIREMENTS: ownership check via
   parentContext player ids + conditional select/render + AP#43 cross-role
   invariant test. Small PR. NOTE: this is presentation-level — the dob COLUMN
   stays row-readable under RLS; true column hardening (column grants / a
   restricted view) is a separate, heavier change that can stage later if
   defense-in-depth is wanted.
2. Confirm as intended (roster-transparency posture: ages are conventional on
   youth rosters) + document the intent in §16.7 so the next audit doesn't
   re-flag it. REQUIREMENTS: doc-only.

CC LEAN: Option 1 (UI gate now; column hardening only if the architect wants
defense-in-depth). Grade already communicates cohort; another family's exact
age isn't load-bearing for any parent workflow, and own-child-only is the
platform's stated posture for derived personal data.
