(b) DATA MODEL + RLS + THE TWO SCHEMA ADDITIONS v2 REQUIRES
------------------------------------------------------------------------------
  EXISTING:
   financial_accounts (guardian×season): season_fee_cents, discount_cents,
     discount_reason, notes. RLS ALL → the INVOICE, directly editable.
   financial_transactions (ledger): account_id, transaction_type
     (payment|refund|adjustment|fee), amount_cents, payment_method, reference,
     receipt_url, occurred_at, recorded_by, applied_to_player_id, notes,
     processing_fee_cents, registration_id. RLS = INSERT+SELECT ONLY (append-only).
   family_balances (view): canonical billed/net_paid/balance per account (#63).
   coach_payouts: amount_cents, status, payment_method, paid_at, … RLS ALL.
   event_coach_assignments: event_id, coach_user_id, status, assigned_by. RLS ALL.
   coaching_assignments: pay_per_session_cents, scope, per coach×team. RLS ALL.

  SCHEMA ADDITION 1 (Fork 1 — traceable ledger corrections):
   financial_transactions.reverses_transaction_id uuid NULL
     REFERENCES financial_transactions(id).
   A VOID inserts a reversing entry (opposite sign, same type-family) that
   POINTS at the original. The original is never mutated. family_balances nets
   the pair to zero. The ledger UI renders the original struck-through with its
   reversal beneath. → immutability preserved + corrections auditable.
   (RLS stays INSERT+SELECT — voids are inserts, not updates.)

  SCHEMA ADDITION 2 (Fork 2 — per-session settlement):
   event_coach_assignments gains:
     pay_cents int NULL        -- amount owed for THIS session; seeded from the
                               -- coaching_assignment rate at assignment time,
                               -- editable per session ("edit each session").
     pay_status text           -- 'owed' | 'paid' | 'excluded' (default 'owed')
     settled_by_payout_id uuid NULL REFERENCES coach_payouts(id) -- which payout
                               -- paid this session (null = unpaid).
   OWED = Σ pay_cents WHERE pay_status='owed'. PAID = Σ pay_cents WHERE
   pay_status='paid'. A coach_payout settles a chosen set of sessions (flips
   them to 'paid' + stamps settled_by_payout_id). Backfill: existing assignments
   get pay_cents from rate×scope; pay_status from whether prior payouts cover them
   (best-effort, or all 'owed' and let Frank settle forward).
   This replaces the rate×count derivation in useCoachPayouts with a true
   per-session ledger — the source of "edit each session" + "which sessions are
   paid".

(c) DESIGN — the hub + two ROUTED detail pages
------------------------------------------------------------------------------
HUB /admin/financials (unchanged shell): summary + season tabs + Families/Coaches
  segments. Cards become tap-throughs to detail ROUTES (Fork 3B). Forms inside
  a detail (record payment, void confirm, settle sessions) are overlays/sheets.

FAMILY DETAIL  /admin/financials/family/:accountId
  Header: family · billed / collected / balance (family_balances).
  INVOICE block (editable, financial_accounts): season fee, discount (+reason),
    notes → "Edit invoice". family_balances recomputes billed (one source #63).
  LEDGER block (full financial_transactions history):
    Each txn: date · type · method · amount · ref/notes · running balance.
    Voided rows struck-through with their linked reversal beneath.
    Actions: Record payment/refund/adjustment (existing form); VOID (posts a
    linked reversing entry — Schema-1); attach receipt_url (column exists).
  → "edit the invoices" = the invoice block; "fix a payment" = traceable void.

COACH DETAIL  /admin/financials/coach/:userId
  Header: owed / paid / balance · rate · default method (Pay settings).
  SESSIONS block (event_coach_assignments as PAYABLE LINE ITEMS — the detail
    behind owed): date · event · team · pay_cents (editable) · status
    (owed/paid/excluded) · settling payout. Edit each session's pay; exclude a
    session; multi-select unpaid sessions → "Mark paid" creates/links a payout.
  PAYOUTS block (coach_payouts, mutable): each payout editable amount/status/
    method/date + delete; shows which sessions it settled. Record payout
    (method pre-filled).
  → "edit each of the sessions" = the sessions block, literally.

ARCHITECTURAL FORKS — v2 LEANS (committed; operator ratifies)
------------------------------------------------------------------------------
FORK 1 — Family ledger corrections.
  LEAN (v2): APPEND-ONLY + LINKED REVERSALS (Schema-1). Voids are inserts that
  reference the original; nothing is ever edited/deleted; family_balances nets
  the pair. The invoice (account fee/discount) is the one directly-editable
  surface. Rationale: a financial ledger MUST be immutable for audit integrity;
  the linkage makes every correction traceable to what it fixed. (Rejected:
  in-place edit/delete of transactions — simpler UI, but silently rewrites
  financial history and risks balance drift. Wrong for money.)

FORK 2 — Coach session management.
  LEAN (v2): PER-SESSION SETTLEMENT (Schema-2). Each session is an editable,
  settleable line item; owed/paid derive from the session ledger, not rate×count.
  Rationale: it's the literal ask ("edit each of the sessions") and the only
  model that answers "which sessions has this coach been paid for". The rate
  becomes the DEFAULT seed for a session's pay_cents, still per-session editable.
  (Rejected: v1's add/remove-assignment-only — can't edit a single session's pay
  or track per-session settlement.)

FORK 3 — Surface shape.
  LEAN (v2): DEDICATED ROUTES (/admin/financials/family/:id, /coach/:id), with
  in-page forms as overlays/sheets. Rationale: a money-management hub benefits
  from deep-linkable, refresh-safe detail pages (link a payment_overdue alert
  straight to that family's ledger; bookmark a coach). Back-button is already
  hardened (useGoBack). (Rejected: overlay-only — fine for a quick action,
  thin for a surface you "manage in one spot".)
