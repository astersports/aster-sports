3. THE MONETIZATION MODEL (resolved + one open sub-fork)
=====================================================================
RECONCILIATION [architect, accepted]: do NOT 7-day-paywall the WHOLE Hub. The anon public
read (search any team, standings/schedule/scores) is the acquisition wedge + SEO surface ‚Äî
**free forever**. Apply the **7-day trial ‚Üí $20/mo to the ASSISTANT layer only** ‚Äî exactly
what `PlusGate` already gates. Gateway AND revenue; matches the price already in code.
This refines Frank's "Tier 0 free 7 days then monthly": the *public read* is free always;
the *assistant* is the trial‚Üísubscription.

MONEY-PATH WRITER [Frank-decided 2026-06-29]: a **NEW Supabase edge function** in
aster-sports is the SOLE writer of `family_entitlements` (Stripe webhook ‚Üí edge fn ‚Üí
status). Keeps billing on the same backbone as `is_entitled` (which already reads it),
host-independent, the architect's "one writer, Stripe-only" honored. Reuses the studio's
proven Stripe code (`astersports-web/server/{stripe,billing,webhook,shadowBilling}.ts`,
`trial_end`, `trialReminders`) as a TEMPLATE ‚Äî but stays independent of the Railway studio
server. Flip `entitlement` to read `is_entitled` (the "only change needed" per the comment).

WHAT'S NET-NEW for the money path: the checkout + webhook edge functions (aster-sports has
NO Stripe today ‚Äî confirmed: no dep, no fns), a `plan` column on `family_entitlements`
(free/plus), trial-reminder cron, customer portal, SCA/3DS handling, dunning UX.

OPEN SUB-FORK [DR-T, lean]: trial SHAPE ‚Äî (a) card-up-front reverse-trial (matches
`shadowBilling` rails, far higher conversion) vs (b) Stripe-native `trial_period_days`
no-card "free 7 days then charge" (lower friction, what Frank's phrasing implies). CC LEAN:
(a) card-up-front for the ASSISTANT trial ‚Äî the public read is already the zero-friction
free wedge, so asking for a card to start the premium trial is fair and converts ~2-3√ó.

4. THE ONE REAL STRUCTURAL BLOCKER (and why it's bounded)
=====================================================================
[G] aster-sports REQUIRES an org. `user_roles.organization_id` is NOT NULL (mig 016);
`current_user_org_id()` returns NULL for a no-org user; every org-scoped RLS then returns
zero rows; `HomePage` routes role=null ‚Üí a "account is ready" PLACEHOLDER
(`HomePage.jsx:57-64`). So a signed-in parent NOT yet in a program has no home today.

WHY BOUNDED: the public gateway (anon `get_public_*` RPCs) and the family layer
(`family_*`, user_id-scoped, NOT org-scoped) do NOT depend on `current_user_org_id()` ‚Äî
they already work without an org. [G re-verified 2026-06-29] `organization_id` IS NOT
NULL (`016_user_preferences.sql:39-40`) ‚Äî so a no-org parent has NO `user_roles` row at
all (not a null-org row); `current_user_org_id()` returns NULL ‚Üí empty org scope. The fix
is therefore PURELY ADDITIVE ‚Äî no constraint relaxation, no migration risk. (This corrects
the architect's "org_id nullable" phrasing; the additive conclusion is unchanged.) The fix
is NOT "rip out org-scoping"; it is:
  (a) a NO-ORG FAMILY HOME route that renders the Hub from the public RPCs + family_*
      tables (not the org-scoped contexts), and
  (b) make the global providers anon/no-org-tolerant (SeasonProvider hard-gates on orgId
      `SeasonContext.jsx:23,47`; WeatherProvider only fetches when user truthy) ‚Äî the Hub
      should consume public hooks directly, not `useSeason()`.
Tier 2 (onboarded) keeps the existing org model untouched.

TRACKING PERSISTENCE GAP [DR-P]: astersports-web's Hub tracks anon via localStorage
(`trackingStore.ts`); aster-sports' `tracked_teams` is authenticated-only with NO
localStorage‚Üíaccount sync. Decide: port the anon localStorage store + a sync-on-login
migration (keeps the no-login track-then-convert funnel) vs account-only tracking (forces
signup to track). CC LEAN: port localStorage + sync-on-login ‚Äî preserves the wedge.
