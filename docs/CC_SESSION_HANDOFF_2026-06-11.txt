3. CC ↔ ARCHITECT PROTOCOL (the rules — operate exactly like this)
==============================================================================
  LANES & DIVISION OF LABOR (§11.8 r2)
    • CC implements + verifies against the CLOSEST ground truth available (file:line,
      execute_sql results, deployed state — the §11.7 source hierarchy). Produces the
      artifact WITH evidence + a labeled LEAN (AP#39 — commit to the lean). CC does
      NOT settle an architectural fork unilaterally; it surfaces evidence + lean and
      the architect/Frank decide.
    • ARCHITECT frames decisions/forks, rules them, produces mobile-first HTML
      renders, ratifies. Does NOT touch DB/repo/edge (that's all CC).
    • FRANK ratifies + relays. His real-time signals (go / stop / "1" / routing)
      drive continuation — there is no pre-locked session contract (AP#56/#59 retired).

  ROUND-TRIP IS DOC-TO-DOC (§11.8 r1, r3, r6; §49)
    • EVERY architect-facing artifact = a committed plain-text .txt in docs/ PLUS a
      full chat paste PLUS delivered as a file (SendUserFile — Frank prefers files).
      Plain text; NO triple-backtick wrap around the whole doc; NEVER a
      summary-instead-of-paste; NEVER deferred. The doc IS the deliverable; the paste
      is how the architect receives it.
    • A DECISION the architect must make → a committed ARCHITECT_DECISION_REQUEST_
      <date>.txt: per decision = REQUEST (1 line) · CONTEXT/EVIDENCE (file:line or
      query) · OPTIONS each with REQUIREMENTS · CC's labeled LEAN.
    • The RENDER is the build target (§16.15) — build to the architect's mockup; flag
      divergences, don't silently reinterpret.

  PHASE CLOSE IS EVIDENCE-GATED (§11.8 r4 — the architect's hard rule)
    • A phase is NOT "closed" on a summary line. CC writes the close-out review
      (audit result + open forks + triage); the architect RATIFIES the build + forks;
      then a REAL-DATA device-smoke (not a rollback-txn fixture) is the close GATE.
      Fixtures are correct pre-flight, not a real run. Carry follow-ups with named
      triggers.

  VERIFY-BEFORE-BUILD (AP#60 / the gate that caught #958)
    • When the architect's spec carries a data/behavior assumption, CC verifies it
      LIVE before building. If verification fails, SURFACE it as a finding+decision —
      do NOT build over it. (#958: the architect's collected formula was wrong; CC
      caught it via a rollback-txn replay, fixed the view, and only then built H-1.)

  SESSION HANDOFF (§11.8 r5)
    • Every session ends with a committed CC_SESSION_HANDOFF_<date>.txt (this file's
      kind) Frank can hand the architect.
