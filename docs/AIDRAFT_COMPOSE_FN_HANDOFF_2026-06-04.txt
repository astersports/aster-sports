2. THE BINDING CONTRACT (verbatim from src/hooks/useAiDraft.js — DO NOT DRIFT)
------------------------------------------------------------------------------
The hook and its test (src/hooks/__tests__/useAiDraft.test.js) lock this shape.
The edge fn MUST conform; if the engine lane needs a different shape, that is a
contract change and the hook + test change with it (flag it, don't silently
diverge).

CALL SITE:  supabase.functions.invoke(AI_DRAFT_FN_SLUG, { body })
SLUG:       AI_DRAFT_FN_SLUG = 'briefing-ai-draft'  (PENDING confirm — see §4a)

REQUEST body (JSON):
  Always present:
    org_id    : uuid   (the caller's org, from AuthContext)
    kind      : string (briefing kind — see §4d for the v1 supported set)
    mode      : string (seen in tests: 'draft', 'redraft' — see §4d)
    audience  : object (seen in tests: { team_id } or {} — see §4d)
  Exactly one of these two shapes, depending on lane:
    AUTO-PROPOSED:  proposal_id : string         (no gist)
    FREE-FORM:      facts : object  +  gist : string
  (The hook only adds proposal_id / facts / gist when non-null. So a free-form
   call sends facts+gist and NO proposal_id; an auto-proposed call sends
   proposal_id and NO facts/gist.)

RESPONSE body (JSON) on success:
    body         : string            -> the drafted PROSE (rendered into editor)
    card_summary : string            -> short "10-second summary" line
    facts_used   : array             -> the facts the draft actually used
                                        (hook passes through as-is; shape is
                                        the engine lane's call — see §4d)
    warnings     : array of string   -> e.g. missing-fact blanks ("venue missing")
  Any missing field defaults to '' / [] client-side (test case d). Keep the
  four keys exactly: body, card_summary, facts_used, warnings.

RESPONSE on failure:
    { error: "<human message>" }     -> hook throws Error(error), surfaces to UI.
  This MUST be returned in the JSON BODY (the hook reads data.error first, then
  the transport error), because supabase.functions.invoke collapses non-2xx to
  a generic FunctionsHttpError with no detail. Mirror the pattern already used
  by useImportSchedule.

SERVER-SIDE responsibilities the hook explicitly delegates (per its header):
  - JSON parse, markdown-fence stripping, and RETRY all happen server-side.
    The hook consumes the clean JSON shape as-is. Do not push parse/retry to
    the client.

3. WHAT THE EDGE FN MUST DO (engine lane build)
------------------------------------------------------------------------------
(Reference implementation to model on: supabase/functions/suggest-briefing-closer
 is the closest existing LLM-calling fn; briefing-cron-dispatch /
 briefing-auto-draft-tick show the secret + service-role patterns.)

3.1 Auth
  - Called via supabase.functions.invoke with the caller's user JWT, so the fn
    runs verify_jwt:true (the default). It is an authenticated admin action, NOT
    a shared-secret/cron fn. => No CRON_SECRET / token-param auth here.
  - If you add config.toml at all, do NOT set verify_jwt=false (that would
    require shared-secret auth and trip the AP #31 audit test). Leaving it
    default (true) is correct and needs no config entry.

3.2 System prompt source (the moat)
  - Read organizations.voice_config.ai_draft_profile for org_id and drop it in
    verbatim as the system prompt. It is per-org (tenant #2 gets its own). It is
    already seeded for LH. The fn must NOT hardcode the voice; it reads it.

3.3 Facts discipline (non-negotiable, voice profile §7)
  - Facts (score, record, opponent, date, time, venue, parking, arrival) are
    GIVEN to the model from the DB and used VERBATIM. The model writes prose
    AROUND facts; it never invents a fact. Missing fact => emit a `warnings`
    entry + leave a clear blank; do not guess.
  - This maps directly to the `facts_used` + `warnings` response keys.

3.4 The two modes
  - FREE-FORM (announcement / custom_message lane = AI-1): the request carries
    `facts` (whatever the admin supplied) + `gist` (the admin's intent in a
    sentence). The fn composes prose from gist + facts in the org voice.
  - AUTO-PROPOSED (calendar-anchored kinds = AI-2): the request carries
    `proposal_id`. The fn loads that proposal's assembled facts (see §4b) and
    composes prose for the anchored kind (game_recap, weekly_digest, etc.).

3.5 LLM
  - Use the latest Claude model (per CLAUDE.md env guidance). Key lives in
    app_secrets, read at request time via the getAppSecret helper pattern — NOT
    Deno.env (AP #33). (suggest-briefing-closer already establishes how an LLM
    key is handled here — match it.)

3.6 Output hygiene
  - Return STRICT JSON { body, card_summary, facts_used, warnings }. Strip any
    markdown code fences the model adds before returning. Retry on parse failure
    server-side. Body prose must obey the voice profile HARD DON'TS (no em
    dashes, no corporate jargon, signoff by name).
