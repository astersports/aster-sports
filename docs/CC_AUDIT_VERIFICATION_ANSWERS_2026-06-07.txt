V2. DOES THE DIGEST SEND PATH ALWAYS SET pilot_only=true?  (architect's OWED)
------------------------------------------------------------------------------
Answer: the question's premise doesn't match the code — and the real answer
reframes F-WEEKLY-DIGEST-FAIL-OPEN's live severity DOWN (defense-in-depth gap,
not a live "blast 175" path).
  - digestSend.sendWeeklyDigest (src/lib/digestSend.js:25-128) takes a pre-fetched
    `recipients` array and has NO pilot_only param and NO get_digest_recipients
    call. It builds slices straight from the passed recipients
    (buildSlicesFromRecipients, :45). So the resolver default Agent 1 flagged
    (resolveWeeklyDigest pilotOnly=false, weeklyDigest.js:60) governs the PREVIEW
    path (useResolverPreview), NOT the production send recipient set.
  - Production digest pilot scoping happens UPSTREAM: sendWeeklyDigestFromWizard ->
    mapWizardStateToDigestArgs -> filterSendable over `recipients`, and the
    recipients themselves come from useDigestRecipients(pilotOnly=pilotModeEnabled)
    [Agent 7]. So the wizard's recipient set IS pilot-filtered before digestSend
    sees it.
  - THE REAL BACKSTOP (verified [file:line]): even if digestSend were ever handed
    unfiltered recipients, the edge fn re-checks at send time —
    send-tournament-message/index.ts:163 `pilotMode = ...pilot_mode_enabled ?? true`
    (fail-closed), and :180-186 if pilotMode AND any guardian_id-bearing row is a
    non-pilot guardian, it 403s and ABORTS THE WHOLE SEND (message-level, not a
    silent drop). [LIVE pilot_mode_enabled=true] So a weekly_digest with a real
    non-pilot family in the queue is blocked at the gateway today.
  - NET RECONCILE: F-WEEKLY-DIGEST-FAIL-OPEN's "broadcast to all 175" requires
    pilot_mode_enabled to read false/null AT THE EDGE FN — which is not the case
    live. The resolver fail-open default is a PREVIEW + defense-in-depth concern,
    still worth fixing (FORK D fail-closed), but it is NOT a live mass-send path.
    THE INTERLOCK: this backstop IS the edge-fn gate that F-FAILCLOSED-UNTESTED
    (G7) says is untested. So weekly_digest's safety leans on an unproven gate —
    which is exactly why G7 (test the gate) and FORK D (fail-close the resolvers)
    should ship together. "Is the property enough" -> the property is the edge gate,
    and it's untested.
  - Caveat retained (Agent 7 sub-hole, confirmed at :167): the gate runs the
    guardian check ONLY when guardianIds.length>0; an all-NULL-guardian_id queue
    (synthetic REDIRECT rows) skips it by design. Fine for REDIRECT test sends;
    means the gate protects guardian_id-bearing (real-family) rows only.
